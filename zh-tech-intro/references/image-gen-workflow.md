# 图片生成工作流

> 本文件从 SKILL.md 阶段5独立出来，**仅在用户触发图片生成时加载**。
> 触发词：「生成图片」「把图画出来」「渲染一下」「帮我生图」。

---

## 基本原则

逐张生成、逐张确认，不批量、不跳过用户。每张图经过自动质检后再给用户看，用户确认后再写入文章，再进入下一张。

---

## 第一步：检测并配置环境

```bash
bash scripts/gen-image.sh --check
```

脚本自动检测三样东西，缺失时给出安装指引：

| 工具 | 作用 | 缺失时 |
|------|------|--------|
| `codex` CLI | 驱动 GPT Image 2 | 打印安装命令，需用户手动操作 |
| `gpt-image-2` skill | 图片生成脚本 | `npx skills add https://github.com/agentspace-so/agent-skills --skill gpt-image-2` |
| `picgo` | 上传图床获取 URL | **自动尝试安装**；失败则降级本地路径 |

输出 `STATUS: ready` 后继续；否则按提示完成配置后重新运行 `--check`。

> **picgo 图床配置（一次性）**：`picgo set uploader`。推荐新手用 SM.MS（免费）：注册 https://smms.app 获取 Token，填入即可。配置后 `picgo upload 任意图片.png` 验证，看到 `https://` 开头的 URL 说明成功。

---

## 第二步：列出所有画图提示

扫描文章中所有 `# 画图提示` 代码块，按顺序列出，告知用户共 N 张、每张约 1～3 分钟，询问是否现在开始或跳过某几张。

---

## 第三步：逐张生成、质检、确认

### 3.1 生成图片

提取该画图提示代码块文本（去掉 `# ` 前缀）：

```bash
bash scripts/gen-image.sh --prompt "提示文本" --out /tmp/zh-tech-img-N.png --upload
```

### 3.2 自动质检

用视觉能力读取图片，对照画图提示检查：

| 检查项 | 具体内容 |
|--------|---------|
| **完整性** | 所有节点、标签、箭头是否都出现？有无漏掉的元素？ |
| **文字准确性** | 中文是否正确？有无乱码、错字、截断、英文混入？ |
| **连接正确性** | 箭头方向是否符合描述？有无悬空节点、断线、连错方向？ |
| **反模式检测** | 泳道图结构、旁路未连汇合点、列间箭头混乱、层间箭头扇形展开、节点有流水编号、金字塔层序画反、数据流箭头穿插层内 |
| **视觉整洁度** | 有无严重留白失衡、节点过于拥挤、线条覆盖？ |

质检结果分三档：通过 → 3.3；有瑕疵但可接受 → 附说明展示给用户；有明显问题 → 3.3b 自动修复。

### 3.3 先上传，再展示给用户确认

**先上传拿到 URL，再让用户确认**——不要等用户确认后才上传，上传有时间成本。

展示格式：

```
第 N 张 / 共 M 张：[图片标题]

[CDN URL]

质检结论：✅ 通过 / ⚠️ 有以下瑕疵：[简述]

请确认：
A) 确认，写入文章，继续下一张
B) 重新生成（可告诉我哪里不对）
C) 跳过这张，暂时留空
```

用户选 A → 写入文章（见第四步），继续下一张
用户选 B → 根据反馈调整提示词，回到 3.1（最多 2 次额外重试）
用户选 C → 代码块保留，不插入图片，继续下一张

### 3.3b 自动修复重试（质检不通过时）

| 检测到的问题 | 修复策略 |
|-------------|---------|
| 泳道图结构 | 改用「N 张卡片从左到右水平排列，卡片之间用粗箭头连接，禁止使用泳道/等高列结构」 |
| 旁路节点悬空 | 对每条旁路末端加「→ 向下连入「[汇合节点名]」」 |
| 列间箭头混乱 | 添加独立「列间连接」段落，逐一声明每列出口连到哪列入口 |
| 层间箭头扇形展开 | 加「每层之间只画一条居中垂直向下箭头，不展开连到各节点」 |
| 文字截断或乱码 | 缩短单节点文字，拆分过长标签 |
| 节点有流水编号 | 加「节点不加流水序号，除非画图提示明确要求」 |
| 金字塔层序画反 | 加「底部最宽为第一层，顶部尖顶为第 N 层，从下到上编号递增」 |
| 留白失衡 | 减少某区域节点数量，或拆分密集节点为简洁描述 |

最多自动重试 **3 次**（含第一次）。3 次后仍有问题 → 3.3c。

### 3.3c 三次均未通过：询问用户

```
自动修复 3 次后仍有问题：[问题描述]
最近一次结果：[URL]

A) 接受这张（有瑕疵但够用）
B) 告诉我具体哪里不对，我来重新描述一次
C) 跳过这张，暂时留空
D) 查看 3 次生成结果，我来选一张
```

---

## 第四步：写入文章

写入时，**同步更新画图提示代码块**：把最终生成成功的提示词回写到文章代码块，替换原始版本——提示词是图片的"源码"，内容变更重新生成时用的是最新版。

写入格式（代码块保留不删）：

```markdown
​```
# 画图提示：[图片标题]（此处为最终生成成功的版本）
# ...
​```

![图片标题](https://cdn.example.com/img-N.png)
```

写入后告知用户，继续下一张。

---

## 联动规则：文章内容变更时主动提醒

文章内容实质性变化（机制描述改了、节点增删了、流程顺序调整了），且该章节有已生成图片时，**必须主动告知用户**：

> 「[章节名] 的内容有变化，对应图片可能已过时，建议删除旧图重新生成，还是先保留？」

用户选重新生成：走完整流程（质检+确认+上传+写入），旧图 `![...]()` 在新图写入后删除。

---

## 兼容性一览

| 场景 | 结果 |
|------|------|
| codex 未安装 | 打印安装步骤，跳过，文章不变 |
| gpt-image-2 skill 未安装 | 打印安装指引，跳过，文章不变 |
| picgo 未安装 | 自动安装；失败则存本地，插入本地路径（后续手动替换） |
| picgo 未配置图床 | 存本地，打印配置指引，插入本地路径 |
| 全部就绪 | 生成 → 质检 → 上传 → 用户确认 → CDN URL 写入文章 |
