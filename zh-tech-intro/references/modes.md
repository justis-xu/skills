# 三种文章模式与写作提示词

> 从 SKILL.md 独立出来，阶段1选模时按需读取。

---

## Mode A：工具 / 框架快速入门

**适用**：让读者快速了解工具是什么、怎么用。

```
# 一、前言（SCQA 内化结构，有吸引人数字则加数据表）
# 二、它是什么（定义 + 核心价值 + 类比）
# 三、快速上手（安装 + 最小可跑示例 + 关键 API）
# 四、核心能力（分节讲 3~5 个主要功能，每个配「是什么/为什么/怎么做」三层）
# 五、和同类工具对比（对比表格 + 适用场景结论）
# 六、总结
```

---

## Mode B：架构源码深挖

**适用**：工程师想真正理解"为什么这样设计"。

```
# 一、前言（SCQA 内化结构，有吸引人数字则加数据表）
# 二、概要设计（代码结构目录树 + 模块依赖图 + 主干流程鸟瞰，不深入细节）
# 三、[主干流程 A] 详细设计（是什么 → 为什么这样设计 → 怎么实现，逐层拆解）
# 四、[主干流程 B] 详细设计（同上）
# 五、[最独特机制]（公式/数据结构 + 直觉解释 + 代码 + 权衡取舍）
# 六、[可选] 特殊子系统深讲
# 七、评测体系（如有论文或 benchmark）
# 八、总结
```

**注意**：
- 主干流程 A/B 按系统特点命名（数据库：写入/查询；Agent：规划/执行）
- 概要设计是关键承接层——前言说"是什么"，概要设计给读者一张地图，再进细节

**读者模式对 Mode B 的影响**：

| 要素 | R1（纯小白）| R2（有技术背景）| R3（技术专家）|
|------|------------|----------------|--------------|
| 代码块 | 禁止，改用图示+公式+文字 | 可用简化伪代码（标 `# 简化示意`）| 可引用真实源码片段 |
| 专业术语 | 首次出现必须类比解释 | 一句话说清在本系统中的含义 | 直接使用，不解释 |
| 子标题编号 | 必须（2.1, 3.1.2）| 推荐 | 可选 |
| 是什么 vs 为什么 | 必须分段 | 分段或同段明显区分 | 可合并 |
| `>` 引用块频率 | 每 2 节至少 1 个 | 每 3 节至少 1 个 | 按需 |

## Mode C：版本 / 方案对比

**适用**：做技术选型，或对比新旧方案。

```
# 一、前言（SCQA 内化结构，对比背景）
# 二、问题根源（旧方案为什么不够好）
# 三、[方案/版本 A]（核心设计 + 优势 + 局限）
# 四、[方案/版本 B]（核心设计 + 优势 + 局限）
# 五、关键差异深挖（选 2~3 个维度横向对比，用对比图）
# 六、适用场景与选型建议
# 七、总结
```

## 可复制的写作提示词（填变量即用）

```
请用中文为以下技术项目写一篇深度介绍文章，写作模式：[Mode A/B/C]

写作风格要求：
- 工程师视角，先说痛点问题，再说解决方案
- 每个核心机制必须覆盖「是什么/为什么/怎么做」三层，不能只介绍功能
- 前言按 SCQA 逻辑自然流动（不贴标签），有吸引人的基准数字才加数据表
- 关键参数（容量、阈值、超参）必须对照源码核实，不能直接引用 README 数字
- 代码块、表格、散文三者交替，不连续出现超过 3 个同类型块
- 画图提示放在代码块内，背景白色，节点标签用中文，不写 px/尺寸/颜色值，不用泳道图

项目信息：
- 项目名称：
- 一句话定位（它解决什么问题）：
- 核心数字（评测结果/容量/阈值等，已源码核实）：
- 源码目录/文档位置：
- 最值得深讲的 2~3 个核心机制：
- 目标读者（想快速上手/想深度理解/想技术选型）：
```
