# 中文技术介绍文章 Skill

## 开始前必须确认：读者类型与写作目的

**如果用户没有说明目标读者或写作目的，必须在开写前询问以下问题（一次性问，不要分开问）：**

> 开始前想确认几个问题：
> 1. 这篇文章面向什么样的读者？（完全不了解这个领域的普通人 / 有技术背景但不熟悉这个系统的工程师 / 读过类似系统想深入理解设计决策的专家）
> 2. 文章发在哪里？（微信公众号 / 技术博客 / 内部文档 / 其他）
> 3. 写作目的是什么？（让读者了解这个系统是什么 / 帮工程师快速上手 / 做技术选型参考 / 展示核心设计的精妙之处）

根据回答选择对应的读者模式：

### 读者模式 R1：纯小白

**特征**：对这个领域完全没有背景，甚至不了解基础概念。

**写作要求**：
- **禁止源代码块**：不得出现真实源码或「简化示意」伪代码块；实现细节改用图示、公式、文字描述
- **基础概念必须解释**：涉及认知科学、向量检索等专业概念时，先给直觉类比，再给定义
- **多用 `>` 引用块**：作为阶段性结论的高亮，或关键概念的直觉解释
- **英文术语首次出现必须给中文**：用「中文名（英文）」格式，后续只用中文
- **是什么和为什么必须分段**：两者各成独立段落，不得混写进同一句话
- **多原因必须用列表**：一个设计决策有 2 个以上原因时，用无序列表，不用分号长句
- **编号子标题**：较长章节必须使用 `2.1`、`3.1.2` 格式的数字编号子标题，不出现超过 800 字无子标题的段落

**禁止**：源代码块（包括「简化示意」）、不加解释的行业缩写、一句话里塞多个概念

---

### 读者模式 R2：有技术基础

**特征**：会写代码，了解基本 CS 概念，但不熟悉这个具体系统。

**写作要求**：
- 可以包含简化的伪代码，但**必须标注 `# 简化示意`**
- 专业术语不需要从头解释，但仍需一句话说清它在这个系统里的具体含义
- 是什么和为什么可以在一段内写，但要明显区分（通常是：第一句说定义，后续说原因）
- 多原因用列表
- 配图可以比 R1 更简洁

**禁止**：直接贴大段源码、假设读者完全熟悉系统内部设计

---

### 读者模式 R3：技术专家深度解析

**特征**：深入了解同类系统，来看设计决策、权衡取舍、关键数字。

**写作要求**：
- 可以直接引用源码片段（但需要加中文注释说明「为什么」）
- 重点放在设计取舍、非显而易见的机制、关键参数的来源
- **前言策略**：读者已熟悉问题背景，跳过痛点铺垫，直接抛出"这个系统做了什么反直觉的设计"——用 3~5 条亮点无序列表开头，让读者扫一眼就能判断是否值得精读

**禁止**：把显而易见的基础概念解释半页

## Hard Gates（必须满足，开写前逐条确认）

**开写前先做两件事**：确认文章属于哪种模式（见下方"三种文章模式"），并读 `references/style-guide.md`。

---

### 一、研究与事实核查

1. **先研究后写作**：至少读过源代码核心文件或论文摘要，记录实际数字（容量、阈值、评测结果）

2. **代码真实性验证**：文中所有数字（容量、阈值、超参、评测结果）必须对照源码或论文原文核实。发现文档与源码不一致时，以源码为准并在文章中注明。禁止直接引用 README 中未经核实的数字。

   > **反例**：某文档说权重是 `0.20/0.25/0.15/0.15/0.10/0.15`，实际源码是 `0.24/0.30/0.15/0.15/0.10/0.06`。必须核实后再写。

3. **不虚构伪代码**：只给出源码中真实存在的代码片段。如果源码中没有某段逻辑的清晰实现，用文字描述即可；不要自己发明 `# 简化示意` 代码块冒充实现细节。

4. **推测性结论必须标注**：源码中找不到明确注释或代码依据、只能通过行为推断的机制，不能写成确定事实。用 `>` 引用块单独呈现，在结论前明确说明"这是推测"。

   > **反例**（把推测写成确定事实）：  
   > 「@提及 Agent 时，系统会把子 Agent 的记忆注入主 Agent 上下文，让主 Agent 知道子 Agent 的历史知识，方便组织调用参数。」（这段逻辑在源码里没有注释，是推断）  
   >
   > **正例**（`>` 引用块标注推测）：  
   >
   > > 为什么要在主 Agent 的上下文里也注入子 Agent 的记忆？代码里没有注释说明，这里是推测：可能是为了让主 Agent 了解子 Agent 的历史知识，组织调用时传入更准确的上下文；也可能只是工程实现上的顺手设计。

   判断标准：删掉源码，这段话还能靠逻辑推断出来吗？**能 → 标为推测**；不能（有明确代码/注释支撑）→ 可以写成事实。

5. **技术术语以行为为准，不用推测的实现词汇**：描述系统内部机制时，用能从外部观察到的行为（"后台异步任务"、"后台运行"），不用未经源码确认的底层实现词汇（"分叉进程"、"子进程"、"fork()"）。

   - **可以用**：后台异步任务、后台运行、异步写入、不阻塞主流程、发射后不管
   - **避免用**（除非源码明确）：分叉进程、fork、子进程、线程、守护进程

### 二、内容架构

6. **先问题后方案**：每个核心机制先说"为什么需要"，再说"怎么实现"

7. **由浅入深，先易后难**：每个机制的介绍从最简单的情况开始，确立基础认知后再引入复杂情况和边界条件。不要在读者还不知道"普通情况"是什么时，就先抛出"只在特殊情况下才有意义"的概念。

   - **先写 Happy Path**：讲一个机制时，先说"最简单的情况下它是怎么工作的"；等读者建立了基本模型，再说"当 X 发生时，还有另一层"
   - **次要概念后置**：某个概念只在特定条件下才会出现时，先解释不需要这个概念的基本情况，再用引用块或"当…时"的句式引入该概念
   - **检查信号**：如果一个章节的第一句话里出现了只在下文才解释的概念，这是由浅入深违反的信号

   > **反例**（先抛专有概念，读者还不知道基础情况是什么）：  
   > "取历史时，循环先扫描压缩边界标记，从标记之后开始取。"（读者还不知道"没有压缩时怎么取"）  
   > **正例**（先简单，再引入复杂情况）：  
   > "最简单的情况：会话刚开始，所有消息都原样送过去。但随着对话变长，上一轮可能已经做过压缩——这时候就需要只取还没被摘要代替的那部分。找到这个起点，靠的是**压缩边界**标记。"

8. **内容详细度（三层结构）**：每个核心机制必须覆盖三层。**三层如何写，取决于读者模式**：

   | 层次 | 定义 | R1（纯小白）写法 | R2/R3 写法 |
   |---|---|---|---|
   | **是什么** | 这个机制/组件的定义 | 先给生活类比，再给系统定义，独立一段 | 一句话定义 |
   | **为什么** | 不这样做会有什么问题（先讲痛点） | 独立一段，多个原因用列表 | 可合并进上段 |
   | **怎么做** | 实现细节、关键数字 | 用图示 + 公式 + 文字，**无代码** | 关键代码片段 + 关键数字 |

   > **R1 正例**（三层分段）：  
   > （是什么）"短期记忆相当于对话的工作台——把最近几轮消息放在随手可取的地方，而不是每次都去翻历史档案。系统用一个容量有限的双端队列来实现，最旧的消息在头部，最新的在尾部，默认容量上限是 64000 个字符。"  
   >
   > （为什么）"之所以需要单独的短期记忆，有两个原因：一是长期记忆的检索需要先做向量化计算，再去数据库里做近邻搜索，有一定延迟；二是当前对话里最近说的话，高概率和当前问题相关，单独存起来能做更快的顺序检索。"

   **按读者模式校准解释深度，禁止过度降维**：R2/R3 读者文章中，禁止用生活化比喻过度解释读者已知的专业概念。向量、嵌入、余弦相似度等从业者日常词汇，直接用即可，不要用"一串数字""某种计算方法"这类措辞代替。判断标准：目标读者在日常工作中会不会直接说这个词？**会 → 直接用；不会 → 才需要解释**。

9. **前言结构遵循 SCQA**：前言按 Situation → Complication → Answer 组织，但**绝对不能把这些标签写成章节标题**：
   - **S – Situation**：一两句交代系统是什么、定位是什么
   - **C – Complication**：说清楚核心矛盾，用类比把矛盾变成读者能立刻想象的场景，不超过 4 句
   - **A – Answer**：简短概括解法，2~3 条要点，直接写，不加英文标签名
   - **数据表**：只在有吸引人的基准数字时才加，无好数字跳过
   - **深度预告**：如果需要，只用一句话，不要单独设标题或编号列表
   - 禁用套话开头（"今天我们来聊聊 X"、"随着 AI 的发展……"）

10. **不写文章大标题**：文章第一个标题就是 `# 一、前言`（或 `# 一、简介`），不在最开头写 `# PowerMem：AI 智能记忆系统深度解析` 这类大标题。

11. **一标题一主题**：若一个 `###` 子标题下实际包含两个独立机制，必须拆成两个子标题。判断标准：两个机制能否各自单独理解？能 → 必须拆分。

12. **禁止跨节重复**：某机制已在其他章节详细讲解时，当前节删除重复内容，必要时一句话带过。判断标准：删掉这段，读者对当前位置的理解会减少吗？不会 → 删。

13. **章节标题不加元描述前缀**：不写 `## 关键设计：三层语义索引`，直接写 `## 三层语义索引`。"关键设计"、"核心机制"、"重要组件"是作者判断，不是内容本身，一律删去。

14. **禁止章节间交叉引用**：文章正文中不得出现「见第 X 章」「详见 X.X 节」「配套文章 XX」等跳转提示。每个章节应自给自足。如果一个机制确实在两处都需要提及，用一句话概括要点。**定稿检查**：搜索全文中的「见第」「详见」「配套文章」，逐一删除或替换为内联说明。

15. **删除与主题无关的细节**：写作时遇到某个实现细节，先问"这对读者理解核心设计有帮助吗？"

    > **判断标准**：删掉这段，读者对系统的核心理解会减少吗？不会减少 → 删。

16. **常量分散到上下文，不单独成节**：系统的关键参数（阈值、容量、超参）禁止集中到独立的"关键常量"章节。每个参数在第一次有意义时才引入（紧跟讲述它的机制），并解释这个值的来由。

17. **保持整体连贯，禁止跳跃和断层**：文章必须能从头读到尾，前后衔接自然，读者不会突然感到"这里怎么突然跳到这个话题了"。

    **章节之间**：相邻章节必须有逻辑关联，结束一节时给读者一个"下一节讲什么、为什么要讲"的过渡，不能戛然而止然后切换话题。

    **段落之间**：段与段之间不能无缘由地突然切换视角或引入新概念。如果要引入新概念，前一段必须为它铺垫（可以是一句"但还有一个问题没解决"或"这只是其中一条路径"）。

    **常见断层信号**（写完一节后自查）：
    - 读者读完这节，还不知道"接下来讲什么"——需要加过渡
    - 某个词/概念突然出现，前面没有任何铺垫——需要往前加一句引入
    - 前面说了"有三种情况"，后面只讲了两种，第三种消失了——需要补齐或删掉那句"三种"
    - 某节的开头和上一节的结尾完全没有关联——需要加一句衔接

    **拿不准衔接是否自然时**：把这两段读给用户听，或者直接问用户"这里从 A 跳到 B 你觉得顺吗"，不要自己猜。

### 三、语言与表达

17. **用自然中文表达，避免技术腔；拿不准时给选项问用户**：行文中不能出现直接从英文技术词汇翻译而来、读起来别扭的用语。判断标准：把这句话读给一个没看过这篇文章的同事听，他能不能自然理解？

    常见问题类型：

    | 类型 | 反例 | 正例 |
    |------|------|------|
    | 直译英文技术词 | 记忆预取（prefetch 直译，硬件术语感） | 后台记忆检索 |
    | 直译英文结构 | 两条异步旁支（async side branch 直译） | 两个后台任务 |
    | 文学化词汇错位 | 后面才不会迷失（技术场景里太文学） | 后面才不会绕 |
    | 英文动词名词化 | 高效清桌、清桌后 AI 恢复 | 快速完成上下文压缩、压缩完成后 AI 接着工作 |
    | 直译结构感 | 活在单次对话里 | 只在本次对话期间有效 |

    特别注意：英文名词化动词（如"清桌"借用英文 "clear the desk" 的动词用法）在技术博客里非常容易出现，读起来像机器翻译。优先用中文日常表达：完成、替换、丢弃、保留、读取。

    **拿不准时，必须给选项问用户，不要自己猜**：如果一个概念的中文说法不确定哪个更自然，列出 2~3 个候选说法，加一句描述各自的语感或适用场合，让用户选。不要因为"差不多"就随便用一个——读起来别扭的词会破坏整篇文章的可读性。

    > **示例**：「这里有个概念我有几种说法，想听你的感觉：
    > - "触发检查"——偏正式，像操作手册
    > - "触发一次核验"——口语一点，读起来更顺
    > - "跑一遍检查"——更口语，有点随意
    >
    > 你觉得哪个更符合这篇文章的调子？」

18. **动机说明整合进段落，不用加粗疑问句**：行文中遇到需要解释设计动机的地方，把「是什么」和「为什么」写成两个独立段落，动机直接作为第二段的第一句引出，不要用 `**为什么不直接存原始消息？**` 这类加粗疑问句——它会让文章读起来像 FAQ，破坏叙事流畅感。

    > **正例**（分段自然引出动机）：
    >
    > "STM 底层是一个容量有限的双端队列，最旧的消息在头部，最新的在尾部。容量单位是字符数，默认上限 64000 个字符。
    >
    > 使用 STM 而不是直接查 LTM，有两个原因：
    > - 长期记忆的检索需要向量化查询，有一定延迟
    > - 当前会话的最近几轮对话高概率相关，顺序检索更快、更准"

19. **源代码注释不得直接引用英文原文**：行文中提到"源码注释说明了某个设计意图"时，必须将注释翻译成中文后再写进正文，不要直接贴英文原句。

    > **反例**：「源代码注释也直接写道：`Forward recall is better than backward recall`。」  
    > **正例**：「源代码里也有注释专门说明了这个设计意图：向前方向（时间更晚的回应）的相关性高于向后方向（时间更早的背景）。」

20. **所有英文必须有中文，裸露英文强制禁止**：正文中任何英文单词或短语，无论是组件名、人名、操作命令、学术术语，都必须满足以下规则之一：

    - **首次出现**：给出「中文名（英文）」格式，后续只使用中文名
    - **通用术语可直接替换**：能找到约定俗成中文译名的，直接用中文，不需要保留英文

    **豁免：约定俗成保留英文的术语**，直接使用英文，不翻译，**也不加括号中文解释**：
    - **Token**（不写"词元"）
    - **LLM**（不写"大语言模型"，可直接用缩写）
    - **MCP**（不写"模型上下文协议"）
    - **API / SDK**（不展开翻译）
    - **Agent**（可用，也可写"智能体"，视语境）
    - **ReAct**（不展开翻译，直接使用）
    - **hook**（不写"钩子"——中文技术社区直接说 hook，翻译成"钩子"反而陌生）
    - **JWT / OAuth / SSO** 等认证协议名
    - **Redis、gRPC、Kafka、PostgreSQL** 等广泛使用的技术产品名

    判断标准：中文技术社区直接用英文说出来的词，不翻译，也不加括号解释。

    需要处理的常见类型（不在豁免列表内的）：

    | 类型 | 反例（不允许） | 正例 |
    |---|---|---|
    | 组件名 | Derivative、IngestionService | 派生节点（Derivative）、摄取服务（IngestionService） |
    | 人名 | Tulving、Atkinson & Shiffrin | 图尔文（Endel Tulving）、阿特金森和谢夫林 |
    | 操作命令 | ADD、DELETE | 新增（ADD）、删除（DELETE） |
    | 技术术语 | Cross-Encoder、citation | 交叉编码器（Cross-Encoder）、溯源记录 |
    | 集合/对象名 | Episode | 原始消息，或首次出现时写「原始消息（Episode）」 |

    图示中的节点标签原则上全用中文；Token、MCP、API 等豁免词汇在图中可直接使用英文。

### 四、结构化格式

21. **多原因必须用列表，不用分号长句**：当一个设计决策有 2 个或以上**原因/好处/代价**时，必须用无序列表，不要用分号把它们堆成一个长句。

    > **反例**：「这个设计有三个好处：写入零延迟；Ingestion 失败可以重跑；可以对历史消息重新提炼。」  
    > **正例**：
    > 「这个设计有三个好处：
    > - 写入零延迟，接口不阻塞
    > - Ingestion 失败可以重跑，对历史消息没有破坏性
    > - 更换更好的 LLM 后，可以对历史消息重新提炼」

    **注意边界**：这条规则只针对"列举同一事物的多个原因/好处/代价"。用分号描述多个并列实体各自的职责，是正常散文节奏，不受此限制——但前提是每个实体的描述要足够简短（约 20 字以内）。判断方法：把分号换成"其次"或"另外"读一读——语义通顺 → 是原因列举，改用列表；语义断裂 → 是并列实体描述。

22. **有序 vs 无序列表，按语义选择**：列表的标记类型不是随意的，语义决定形式。

    - **有序列表（`1. 2. 3.`）**：用于有明确先后顺序的内容——执行步骤、触发顺序、阶段序列。判断标准：**调换顺序会改变意思 → 用有序列表**
    - **无序列表（`-`）**：用于平级并列的内容——多个原因、多个特征、多个同等选项。判断标准：**调换顺序不改变意思 → 用无序列表**

    > **反例**（顺序触发用了无序列表）：  
    > "- 90% 时提交折叠规则...  
    > - 95% 时阻塞新请求..."（有先后顺序，应改有序）
    >
    > **正例**：  
    > "1. 90% 时提交折叠规则...  
    > 2. 95% 时阻塞新请求..."

    常见需要有序列表的情形：步骤 1/2/3、阶段 A 然后阶段 B、先判断 X 再判断 Y、"依次尝试三步"。

23. **结构化元素选择指南**：文章中散文、列表、表格、引用块要交替出现：

    | 内容特征 | 推荐形式 |
    |---|---|
    | 一个事物/机制的描述 | 散文段落 |
    | 同一设计的 2+ 个原因/好处/代价 | 无序列表 |
    | 有明确顺序的步骤 | 有序列表 |
    | 多个方案/维度的横向对比 | 表格 |
    | 多个同类别选项的特征汇总 | 表格 |
    | 核心结论/设计原则/权衡提炼 | `>` 引用块 |
    | 连续 3 段以上纯散文 | 插入代码块/图/表格/引用块打断节奏 |

    **表格的四类高价值场景**，遇到时必须优先用表格而不是散文或列表：

    1. **带括号注释的列表** → 改为两列或三列表格（每项已有两个属性，用表格更清晰）
    2. **多条件触发逻辑** → 用表格展示条件分层（必须满足 / 任一满足 / OR 条件），列可根据结构增加
    3. **字段/属性说明** → 用表格，列为「字段名 · 作用 · 注意事项」；字段名加粗，不用行内代码
    4. **两套机制的并排比较** → 用对比表格，行为对比维度，列为两套机制

24. **表达形式优先级**（非常重要）：选择表达形式时按优先级从高到低依次决策，能用更高优先级的形式说清楚，禁止降级用散文替代。

    | 优先级 | 形式 | 适用场景 |
    |--------|------|---------|
    | 1（最高） | 画图提示 | 执行流程、系统架构、多路径对比 |
    | 2 | 表格 | 多维度对比、触发条件分层、字段说明 |
    | 3 | `>` 引用块 / 列表 | 核心结论、设计原则；同一事物的多个原因/好处/代价 |
    | 4 | 加粗 | 段落中值得停顿的关键数字或概念 |
    | 5（最低） | 纯正文散文 | 以上均不适用时 |

25. **数字编号子标题**（R1 必须，R2 推荐，R3 可选）：内容较长的章节（正文超过 600 字或包含 3 个以上子主题）必须拆成 `2.1`、`3.1.2` 格式的编号子标题，不允许大段文字无结构堆叠。

    > 子标题是给读者的"路牌"——他们可以先扫一遍标题，决定哪里跳读，哪里精读。没有子标题的长章节等于强迫读者连续阅读。

26. **Markdown 行内格式主动运用**：正文段落不能是纯文字堆叠。三类常见用法：

    - **加粗（`**text**`）**：段落中首次出现的核心概念、关键数字（阈值/容量/延迟）、表格第一列的项目名称。不要对每个名词加粗——只加真正值得"停一下"的词，平均每段 1~2 处。
    - **行内代码（`` `code` ``）**：文件路径、shell 命令、字段名、具体取值。凡是需要**原样输入或精确引用的代码内容**，用行内代码。**禁止**用行内代码包裹中文词语、章节名、概念名等非代码内容。
    - **数字高亮**：关键阈值、容量上限、延迟数字，优先用加粗（「软上限约 **2,000 Token**」），让读者扫一眼就能捕捉。

    **适度原则**：格式是为了帮助读者，不是装饰。加粗密度过高（每句都有），效果和不加一样——读者视觉疲劳，反而什么都抓不住。

### 五、图示

27. **画图提示规范**：
   - 必须放在代码块内，第一行写 `# 画图提示：`
   - 背景统一白色
   - 不出现函数名和英文技术术语作为图中可见标签
   - **禁止使用纵向泳道图**
   - **禁止在 prompt 中写 px、尺寸、颜色值（色号）等渲染参数**（会限制模型发挥）
   - **禁止在 prompt 中使用英文设计术语**（如 `pill`、`border-radius`、`icon badge`、`chip` 等），一律改为中文描述（如"小圆角标签"、"圆角边框"、"圆形徽标"）
   - **禁止用 ASCII 箭头文字代替真正的画图提示**：不要写 `新消息 → 写入队列 → 超限？→ 驱逐` 这类文字流程图，一律改成正式的 `画图提示` 代码块
   - **【超级重要】禁止使用任何代码化画图工具生成图表**：严禁在文章中输出 Mermaid、draw.io、Chart.js、Graphviz、PlantUML 或任何其他画图 DSL / 绘图库的代码块。所有图示必须且只能用 `画图提示` 代码块表达（第一行为 `# 画图提示：`）。即使用户问"能不能画个图"，也只写 `画图提示`，绝不生成 Mermaid 等工具代码。
   - **多阶段管道（4 个以上阶段）禁止等宽列+贯通顶部深色标题栏**——这是泳道感的根本原因，改用卡片+箭头布局（见下方画图提示规范摘要中的模板）
   - **用色调描述，不用色号**：写"蓝色系"而不是 `#3B82F6`，写"冷灰色调"而不是 `#2C2C2C`

   如果用户已经提供了预先画好的图片，处理方式是：在对应的画图提示代码块之后插入 `<img>` 标签，**保留画图提示不删除**，由用户决定最终用哪张。

28. **图先行，解释后置**：凡是文章中包含配图（`<img>` 标签或画图提示代码块），图的说明文字和解释段落必须放在图的**下方**，不得放在上方。图上方只允许有：引出图的一句话，或直接过渡到图。

    - **`<img>` 图片**：图片标签前只放一句引出语；图片标签后紧跟描述段落或要点注释
    - **画图提示代码块**：代码块本身代表图的位置；代码块之后放解释性小结、`>` 关键洞察或补充说明
    - **结构化内容（表格/列表）与图并存时**：先放图，再放表格或列表作为详细展开——读者先看整体视觉，再看文字细节

29. **插入真实图片后，文字必须描述图的实际内容**：当文章中插入了 `<img>` 真实图片时，紧跟图片的文字段落必须根据图片**实际展示的内容**来写——描述图中可以看到什么、关键的观察结论是什么。不能只写模糊的背景介绍。

### 六、代码与公式

30. **可复现（R2/R3 专属）**：R2/R3 读者文章至少包含 2 个可复制的代码块，配中文注释。R1 读者文章**禁止代码块**，用图示 + 公式 + 文字描述代替所有实现细节。

31. **公式/图优先于代码**：概念性内容（热度权重、算法评分、数学推导）必须用 LaTeX 公式（行内 `$...$`，块级 `$$...$$`）或画图提示来表达，**禁止**用代码块来"解释概念"；代码块只用于展示真实源码片段。

32. **不在正文里写函数名或类名**：介绍某个机制时，说清楚它做什么、什么时候触发、有什么效果，不要把源码里的函数名或类名写进文章正文（如 `extractSessionMemory`、`buildPrompt`）。函数名是实现细节，版本一变就失效，且对读者理解机制没有帮助。

    > **规则**：如果你在描述一个机制时写下了一个驼峰命名或下划线命名的函数/方法名，问自己：删掉它，读者对这个机制的理解会减少吗？不会 → 删；会 → 换成行为描述。

### 七、示例与节奏

33. **> 引用块的两种用途**：
    - **前置高亮**：在一个机制或设计原则的段落**之前**，用 `>` 先抛出核心结论，让读者带着预期去读后续细节。适合"结论反直觉"或"结论重要到值得先建立印象"的地方。
    - **后置总结**：一段讨论结束后，把提炼性的核心结论或有哲理感的句子用 `>` 单独呈现，起到视觉停顿 + 记忆锚点的作用。

    两种用途都有效，不要只用后置。引用块至少每 2～3 个章节出现一次，不要让全文连续 5 个段落都是纯文字。

34. **解释机制时必须给一个具体例子**：解释后台进程、钩子函数、异步任务等"看不见"的机制时，仅说"它在后台做 X"还不够——读者需要一个可以想象的具体场景：触发时用户在做什么、系统在运行什么、最终产出了什么文件/状态/结果。

    > **反例**：「会话记忆的写入由独立后台进程负责，在对话进行中持续更新笔记文件。」  
    > **正例**：「会话记忆的写入由独立后台进程负责。举个例子：你和 Claude Code 讨论了 20 条消息，当 Token 总量超过 10,000 时，后台进程悄悄启动，把"当前任务是重构认证模块、已决定用 JWT、下一步是写单测"这类信息整理进笔记文件。你完全感知不到这个过程，但下次上下文压缩时，这份笔记已经备好了。」

35. **输入→输出对齐展示**：解释"用户触发了什么 / 系统响应是什么"类的示例时，用以下固定结构：

    - **触发侧**（用户说的话、输入消息、触发语句）：用 `>` 引用块
    - **输出侧**（实际写入的内容、API 响应、存储结构）：用代码块
    - **解释性文字**（为什么、边界条件、设计意图）：用 `>` 引用块，放在该组例子**之后**

36. **先解释，再给样例**：介绍任何结构、机制、字段时，先用文字或表格说清楚"是什么、有哪些部分、各自做什么"，再给代码块或具体示例作为验证。不要一上来就贴样例让读者自己猜结构。

37. **密集散文的两步改造法**：遇到一段把多个要点堆在一起的密集散文（"A 做了 X，B 则是 Y，Z 的情况下……"），按以下两步改造：

    **第一步**：给每条要点加加粗标签，让读者能快速扫到关键词而不用通读整段。

    **第二步**：如果有一条反直觉结论或总结性原则，用 `>` 引用块单独呈现，放在段落末尾或最前面（前置高亮）。

    **触发信号**：一段 3 行以上纯文字，且 3 个以上要点混在一起，读者无法扫读定位。

## 工作流程（四阶段）

> **互动原则**：每个阶段的关键决策点，都必须先给用户看选项，等用户确认后再往下走。不要假设用户的意图，不要一口气出结果——哪怕感觉方向已经很明确，也要把选项摆出来让用户选。

### 阶段 1：定位与选模

**目标**：在动笔之前，搞清楚用户真正想要什么。

**第一步：问清背景（一次性问，不要分开问）**

如果用户没有说明，必须主动提问以下三点：

1. **读者是谁**？（完全不了解这个领域的普通人 / 有技术背景但不熟悉这个系统的工程师 / 读过类似系统想深入理解设计决策的专家）
2. **发在哪里**？（微信公众号 / 技术博客 / 内部文档 / 其他）
3. **写作目的**？（让读者了解这个系统是什么 / 帮工程师快速上手 / 做技术选型参考 / 展示核心设计的精妙之处）

**第二步：提供文章角度选项（不要自己拍板）**

根据用户的回答，给出 **2~3 个不同的文章角度**，每个角度说清楚：

- 这个角度聚焦什么、跳过什么
- 适合什么读者和场合
- 大致的章节结构是什么样的

让用户选择或说"综合一下"，不要直接输出骨架。示例格式：

> **角度 A（入门导览）**：从"这个系统解决什么问题"切入，介绍核心概念和使用方法，跳过实现细节。适合想快速了解是否值得深入的读者。章节：简介 → 核心概念 → 快速上手 → 与同类工具对比
>
> **角度 B（架构深挖）**：从设计决策切入，重点拆解 2~3 个最关键的机制，配源码片段。适合想真正理解"为什么这样设计"的工程师。章节：简介 → 整体架构 → 核心机制 A → 核心机制 B → 设计取舍总结
>
> **角度 C（方案对比）**：把这个系统和 1~2 个同类方案并排比较，找出关键差异和适用场景。适合做技术选型的读者。章节：简介 → 问题背景 → 各方案对比 → 我的判断

**第三步：确认素材**

用户选定角度后，确认有哪些可读的素材（源码目录、论文、README、分析笔记），以及用户是否有特别想强调或者不想涉及的点。

### 阶段 2：研究与骨架

**目标**：先把骨架给用户看，确认方向后再大量研究。

**第一步：快速预研（不超过 10 分钟）**

浏览源码目录和 README，找到：

- 这个系统最核心的 3~5 个机制（不一定要全懂）
- 有没有关键数字（评测结果、容量上限、延迟数字）
- 哪些地方最容易被误解或忽视

**第二步：出骨架草稿，等用户确认**

把骨架展示给用户，包括：

- 每个章节的标题和一句话说明（这节讲什么、从什么角度讲）
- 预计重点放在哪 2~3 个核心机制上
- 哪些地方计划配图、哪些用代码块

**明确告诉用户**：这是动笔前的最后确认，如果章节结构、侧重点、或某个机制的取舍有问题，现在改比写完再改省力得多。

**第三步：用户确认后，深度研究**

骨架确认后，才开始认真读源码，记录：

- 关键数字（容量上限、阈值、超参、评测结果）
- 核心数据结构的字段
- 主要调用关系

**与文档/README 交叉核实**：发现不一致，以源码为准，在文章中注明。如需配图，整理"图清单"（每张图展示什么、用哪种图类型），必要时再次和用户对一遍。

### 阶段 3：逐节写作

**原则**：

- **一次只写一个章节**，写完后暂停，等用户反馈后再写下一节。不要连续输出多节，即使感觉很顺。
- **遇到不确定的地方，先问再写**：如果某个机制有两种讲法（例如是否要配图、是否要展开实现细节），把两种思路各用一句话描述，让用户选，不要自己猜。
- **拿不准中文说法时，给候选词让用户选**：写作中遇到某个概念不知道怎么说更自然，不要凑合用一个，列出 2~3 个候选说法（加一句描述各自语感），问用户哪个更顺。这比写完后再逐句改省力得多。
- 每节完成后自检：是否覆盖了「是什么/为什么/怎么做」三层？是否有具体数字和代码？
- **每节写完后，退后一步看整章乃至整篇文章**：这节和前后节的衔接是否流畅？有没有跨节重复或前后矛盾？新内容有没有让文章整体结构变得不均衡？需要调整时，主动告知用户并提议修改，不要等用户反问。

**前言写法（开头重点关注）**：

- 前言按 SCQA 逻辑流动（Situation → Complication → Answer），但**不要把这些词写成章节标题**，框架是内化的
- 数据表只在有吸引人的基准数字时才加；无好数据时跳过，不要强加表格
- 深度预告如果需要，只用一句话，不要单独设标题、不要写成编号列表
- 禁用套话开头（"今天我们来聊聊 X"、"随着 AI 的发展……"）
- **Complication 段要有画面感**：用类比把核心矛盾变成读者能立刻想象的场景，不要只说"存在问题"。好的类比示例——用白板比喻上下文窗口：
  > 把上下文窗口想象成一块有限大小的白板。Agent 的所有工作记忆——对话历史、工具结果、中间推理——都必须写在这块白板上。白板空间用完时，你必须擦掉旧内容才能写新内容。关键问题是：擦掉什么、保留什么、用什么方式擦？
  
  类比之后紧跟一句系统的解法定位，形成完整的 Complication → Answer 转折。

### 技能自我进化

这个技能是活的文档，每次写作和生图都是改进它的机会。写作或生图过程中遇到以下情况时，主动询问用户是否要更新技能。

#### 触发条件

| 情况 | 说明 | 示例 |
|------|------|------|
| **规则空白** | 做了一个判断，SKILL.md 里没有对应规则，且以后多半还会遇到 | 发现某类图适合用新的布局，但 diagram-style.md 没有这个类型 |
| **规则需修正** | 现有规则在实践中有例外、前提缺失，或直接就是反效果 | 某条"禁止"规则在特定场景下反而是最优选择 |
| **新踩坑（图片）** | 生图时遇到了 diagram-style.md 没有记录的渲染问题及修法 | 新发现某种箭头描述方式会导致连线混乱 |
| **更好的提示词写法** | 找到了比 SKILL.md 示例更清晰的图描述结构或表达方式 | 用"横向排列"替代"顺时针闭环"后图的质量显著提升 |
| **写作规律**  | 某类内容反复需要做同样的调整，说明 SKILL.md 应该直接规定 | 每次写完前言都要手动加 SCQA 过渡，不如写进规范 |

#### 触发时机

- **立即触发**：遇到新踩坑、规则明显错误时——这类问题不处理会在下次重蹈覆辙
- **节奏间隙触发**：遇到规则空白或可优化点时，等当前动作完成后再提（写完一节、确认完一张图之后），不打断主流程

**不要触发**：明显的一次性边角场景、用户已经说过"不用管"的点、琐碎的措辞差异。阈值是：**如果下次遇到同样场景，还会需要做同样判断，就值得记录；如果只是这篇文章的特殊情况，就不记录。**

#### 提议格式

在合适的间隙向用户提议，格式如下：

```
💡 发现可以优化技能的地方：

[一句话说明发现了什么，现在技能里是怎么说的（或没说）]

建议加入 / 修改的内容：
---
[具体文字，格式和 SKILL.md 保持一致，可以直接用]
---

要更新技能吗？
A) 直接加（用上面的文字）
B) 我来改一下再加（用户给出修改版）
C) 不加这次
```

用户选 A：直接编辑 SKILL.md 对应位置，新规则加在语义最近的章节，写完告知加在了哪里。
用户选 B：等用户给出修改版后写入，再次确认位置和内容。
用户选 C：记录这次不加，同一条建议不重复提。

同一次会话里如果累积了多条可优化的点，在自然断点（如阶段结束）一次性列出，不要每发现一条就打断一次。

#### 图片踩坑的特殊处理

生图遇到新踩坑时，修法要**同时更新两个地方**：
1. `references/diagram-style.md` 的「七、常见踩坑教训」章节——加入新踩坑描述和修复方法
2. SKILL.md 的「3.3b 自动修复重试」表格——加入对应的检测模式和修复策略

这样下次生图时自动质检能认出这个问题，修复也有依据。

## 三种文章模式

> 完整骨架、读者模式对比表、可复制写作提示词见 `references/modes.md`，阶段1选模时读取。

三种模式简述：
- **Mode A**：工具/框架快速入门——从"是什么/怎么用"切入，配对比表格
- **Mode B**：架构源码深挖——概要设计 → 主干流程详解 → 关键设计，含源码
- **Mode C**：版本/方案对比——问题根源 → 两方案各自分析 → 关键差异深挖

## 画图提示规范（摘要）

> 完整规范、12 种图类型模板、12 个踩坑教训见 `references/diagram-style.md`

**五条强制规则**：

1. 节点标签全部用中文（`ShortTermMemory` → 「短期记忆」）
2. 不出现函数名、类名、字段名、英文缩写
3. **prompt 中不写 px、尺寸、颜色值（色号）、字号**——这些渲染细节写了反而限制模型，越少越好
4. **prompt 中不使用英文设计术语**（pill、border-radius、icon badge、chip 等），改为中文描述（"小圆角标签"、"圆角边框"、"圆形徽标"）
5. **用色调描述代替色号**：写"冷灰色调"而不是 `#2C2C2C`，写"蓝色系"而不是 `#3B82F6`——给模型发挥空间，渲染效果更好

**图类型快速对照**（完整版含参考图见 `diagram-style.md`）：

| 场景                   | 图类型                                     |
| ---------------------- | ------------------------------------------ |
| 多层系统架构总览       | 横向多层架构图                             |
| 有阶段划分的处理流程   | 竖向三阶段流程图                           |
| 多维度权重/评分        | 横向条形图 + 说明卡片                      |
| 公式 + 分级条件        | 左右分栏（公式+卡片）                      |
| 两方案对比             | 并排对比图                                 |
| 多阶段内各有子流程     | 横向阶段列流程图                           |
| **4个以上阶段的管道流程** | **卡片+箭头布局（见下方说明）**          |
| 树状结构搜索/遍历      | 树形层次检索路径图                         |
| 跨模块流程（原泳道图） | **竖向三阶段 or 横向阶段列（禁用泳道图）** |

**卡片+箭头布局（多阶段管道的正确画法）**

泳道感的根本原因：**等宽列 + 深色贯通顶部标题栏**，不管怎么描述，这个结构几乎必然渲染成泳道图。正确做法：把每个阶段改成独立的圆角矩形卡片，卡片之间用箭头连接，高度由内容决定。

Prompt 关键描述模板（直接套用）：
```
# N 张卡片从左到右水平排列，卡片之间用粗箭头（→）连接。
# 禁止使用泳道/等高列结构。每张卡片是独立的圆角矩形，高度由内容决定。
# 整体背景：极浅灰色，无外边框。
# 卡片通用样式：白色底，圆角边框，浅灰描边。
# 顶部：彩色实心圆形徽标（含编号数字）+ 徽标右侧阶段名（加粗）
# 徽标下方：内容区，每条信息用小圆角标签呈现，标签背景色与徽标同色系但更浅
# 卡片 ①「阶段名」  徽标：蓝色系
#   小圆角标签（浅蓝底）：...
# 卡片 ②「阶段名」  徽标：橙色系
#   小圆角标签（浅橙底）：...
```

## 禁止事项

> 完整清单（35 条）见 `references/prohibitions.md`，以下为最高频触犯、代价最高的条目：

- 禁止：开写前没有询问/确认目标读者类型（R1/R2/R3）
- 禁止：跳过大纲阶段直接写正文；一口气写完全文再给用户看
- 禁止：文章数字未与源码核实；把推测性结论写成确定事实
- 禁止：核心机制只说"是什么"，不解释"为什么"和"怎么做"
- **【超级重要】禁止：使用 Mermaid、draw.io、Chart.js、Graphviz、PlantUML 等任何代码化画图工具**——所有图示必须且只能用 `画图提示` 代码块表达；用户说"画个图"也只写 `画图提示`，绝不输出画图 DSL 代码
- 禁止：画图 prompt 中写 px/尺寸/色号/英文设计术语；使用纵向泳道图；图中出现函数名或英文字段名
- 禁止：文章最开头加 `# [系统名]：[定位描述]` 大标题，第一章就是 `# 一、前言`
