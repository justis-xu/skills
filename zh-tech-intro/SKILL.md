# 中文技术介绍文章 Skill

## 开始前必须确认：读者类型与写作目的

**如果用户没有说明目标读者或写作目的，必须在开写前询问以下问题（一次性问，不要分开问）：**

> 开始前想确认几个问题：
> 1. 这篇文章面向什么样的读者？（完全不了解这个领域的普通人 / 有技术背景但不熟悉这个系统的工程师 / 读过类似系统想深入理解设计决策的专家）
> 2. 文章发在哪里？（微信公众号 / 技术博客 / 内部文档 / 其他）
> 3. 写作目的是什么？（让读者了解这个系统是什么 / 帮工程师快速上手 / 做技术选型参考 / 展示核心设计的精妙之处）

根据回答选择对应的读者模式：

### 读者模式 R1：纯小白

**特征**：对这个领域完全没有背景，甚至不了解基础概念。

**写作要求**：
- **禁止源代码块**：不得出现真实源码或「简化示意」伪代码块；实现细节改用图示、公式、文字描述
- **基础概念必须解释**：涉及认知科学、向量检索等专业概念时，先给直觉类比，再给定义
- **多用 `>` 引用块**：作为阶段性结论的高亮，或关键概念的直觉解释
- **英文术语首次出现必须给中文**：用「中文名（英文）」格式，后续只用中文
- **是什么和为什么必须分段**：两者各成独立段落，不得混写进同一句话
- **多原因必须用列表**：一个设计决策有 2 个以上原因时，用无序列表，不用分号长句
- **编号子标题**：较长章节必须使用 `2.1`、`3.1.2` 格式的数字编号子标题，不出现超过 800 字无子标题的段落

**禁止**：源代码块（包括「简化示意」）、不加解释的行业缩写、一句话里塞多个概念

---

### 读者模式 R2：有技术基础

**特征**：会写代码，了解基本 CS 概念，但不熟悉这个具体系统。

**写作要求**：
- 可以包含简化的伪代码，但**必须标注 `# 简化示意`**
- 专业术语不需要从头解释，但仍需一句话说清它在这个系统里的具体含义
- 是什么和为什么可以在一段内写，但要明显区分（通常是：第一句说定义，后续说原因）
- 多原因用列表
- 配图可以比 R1 更简洁

**禁止**：直接贴大段源码、假设读者完全熟悉系统内部设计

---

### 读者模式 R3：技术专家深度解析

**特征**：深入了解同类系统，来看设计决策、权衡取舍、关键数字。

**写作要求**：
- 可以直接引用源码片段（但需要加中文注释说明「为什么」）
- 重点放在设计取舍、非显而易见的机制、关键参数的来源
- **前言策略**：读者已熟悉问题背景，跳过痛点铺垫，直接抛出"这个系统做了什么反直觉的设计"——用 3~5 条亮点无序列表开头，让读者扫一眼就能判断是否值得精读

**禁止**：把显而易见的基础概念解释半页

## Hard Gates（必须满足，开写前逐条确认）

1. **选模式**：开写前必须确认文章属于哪种模式（见下方"三种文章模式"）

2. **读风格**：写作前必须读 `references/style-guide.md`

3. **先研究后写作**：至少读过源代码核心文件或论文摘要，记录实际数字（容量、阈值、评测结果）

4. **先问题后方案**：每个核心机制先说"为什么需要"，再说"怎么实现"

4b. **由浅入深，先易后难**：每个机制的介绍从最简单的情况开始，确立基础认知后再引入复杂情况和边界条件。不要在读者还不知道"普通情况"是什么时，就先抛出"只在特殊情况下才有意义"的概念。

   - **先写 Happy Path**：讲一个机制时，先说"最简单的情况下它是怎么工作的"；等读者建立了基本模型，再说"当 X 发生时，还有另一层"
   - **次要概念后置**：某个概念只在特定条件下才会出现时，先解释不需要这个概念的基本情况，再用引用块或"当…时"的句式引入该概念
   - **检查信号**：如果一个章节的第一句话里出现了只在下文才解释的概念，这是由浅入深违反的信号

   > **反例**（先抛专有概念，读者还不知道基础情况是什么）：  
   > "取历史时，循环先扫描压缩边界标记，从标记之后开始取。"（读者还不知道"没有压缩时怎么取"）  
   > **正例**（先简单，再引入复杂情况）：  
   > "最简单的情况：会话刚开始，所有消息都原样送过去。但随着对话变长，上一轮可能已经做过压缩——这时候就需要只取还没被摘要代替的那部分。找到这个起点，靠的是**压缩边界**标记。"

5. **画图提示规范**：
   - 必须放在代码块内，第一行写 `# 画图提示：`
   - 背景统一白色
   - 不出现函数名和英文技术术语作为图中可见标签
   - **禁止使用纵向泳道图**
   - **禁止在 prompt 中写 px、尺寸、颜色值（色号）等渲染参数**（会限制模型发挥）
   - **禁止在 prompt 中使用英文设计术语**（如 `pill`、`border-radius`、`icon badge`、`chip` 等），一律改为中文描述（如"小圆角标签"、"圆角边框"、"圆形徽标"）
   - **禁止用 ASCII 箭头文字代替真正的画图提示**：不要写 `新消息 → 写入队列 → 超限？→ 驱逐` 这类文字流程图，一律改成正式的 `画图提示` 代码块
   - **【超级重要】禁止使用任何代码化画图工具生成图表**：严禁在文章中输出 Mermaid、draw.io、Chart.js、Graphviz、PlantUML 或任何其他画图 DSL / 绘图库的代码块。所有图示必须且只能用 `画图提示` 代码块表达（第一行为 `# 画图提示：`）。即使用户问"能不能画个图"，也只写 `画图提示`，绝不生成 Mermaid 等工具代码。
   - **多阶段管道（4 个以上阶段）禁止等宽列+贯通顶部深色标题栏**——这是泳道感的根本原因，改用卡片+箭头布局（见下方画图提示规范摘要中的模板）
   - **用色调描述，不用色号**：写"蓝色系"而不是 `#3B82F6`，写"冷灰色调"而不是 `#2C2C2C`

   如果用户已经提供了预先画好的图片，处理方式是：在对应的画图提示代码块之后插入 `<img>` 标签，**保留画图提示不删除**，由用户决定最终用哪张。

6. **可复现（R2/R3 专属）**：R2/R3 读者文章至少包含 2 个可复制的代码块，配中文注释。R1 读者文章**禁止代码块**，用图示 + 公式 + 文字描述代替所有实现细节。

7. **代码真实性验证**：文中所有数字（容量、阈值、超参、评测结果）必须对照源码或论文原文核实。发现文档与源码不一致时，以源码为准并在文章中注明。禁止直接引用 README 中未经核实的数字。

   > **反例**：某文档说权重是 `0.20/0.25/0.15/0.15/0.10/0.15`，实际源码是 `0.24/0.30/0.15/0.15/0.10/0.06`。必须核实后再写。

8. **内容详细度**：每个核心机制必须覆盖三层。**三层如何写，取决于读者模式**：

   | 层次 | 定义 | R1（纯小白）写法 | R2/R3 写法 |
   |---|---|---|---|
   | **是什么** | 这个机制/组件的定义 | 先给生活类比，再给系统定义，独立一段 | 一句话定义 |
   | **为什么** | 不这样做会有什么问题（先讲痛点） | 独立一段，多个原因用列表 | 可合并进上段 |
   | **怎么做** | 实现细节、关键数字 | 用图示 + 公式 + 文字，**无代码** | 关键代码片段 + 关键数字 |

   > **R1 反例**（三层全混一段）：  
   > "系统使用 STM 存当前会话消息，这是一个字符容量为 64000 的双端队列，因为 LTM 检索延迟高所以用 STM 提供快速访问。"
   
   > **R1 正例**（三层分段）：  
   > （是什么）"短期记忆相当于对话的工作台——把最近几轮消息放在随手可取的地方，而不是每次都去翻历史档案。系统用一个容量有限的双端队列来实现，最旧的消息在头部，最新的在尾部，默认容量上限是 64000 个字符。"  
   >
   > （为什么）"之所以需要单独的短期记忆，有两个原因：一是长期记忆的检索需要先做向量化计算，再去数据库里做近邻搜索，有一定延迟；二是当前对话里最近说的话，高概率和当前问题相关，单独存起来能做更快的顺序检索。"  
   >
   > （怎么做）"写入时，新消息追加到队列尾部。总字符数超过上限后，最早的消息从头部被移出，放进待压缩缓冲区，由后台异步任务负责把旧消息压缩成摘要，主流程不等待——写入延迟只取决于队列操作本身。"

   > **R2/R3 反例**："系统使用混合搜索提高召回率。" ← 缺少为什么和怎么做  
   > **R2/R3 正例**："单路向量搜索的问题是专有名词在向量空间中容易被稀释，召回不全。混合搜索的解法是：`score = vectorWeight * vectorScore + textWeight * textScore`，两路并行执行后合并，专有名词命中率显著提升。"

8b. **按读者模式校准解释深度，禁止过度降维**：R2/R3 读者文章中，禁止用生活化比喻过度解释读者已知的专业概念。向量、嵌入、余弦相似度、梯度、损失函数、哈希、索引等从业者日常词汇，直接用即可，**不要用"一串数字""某种计算方法"这类措辞代替**——读者会觉得被低估。

    判断标准：目标读者在日常工作中会不会直接说这个词？**会 → 直接用；不会 → 才需要解释**。

    > **反例**（过度降维，R2/R3 读者会反感）：「把每条记忆转化成一串数字（向量），找出数字最接近的记忆」  
    > **正例**（直接用专业表述）：「把每条记忆和查询问题分别嵌入向量空间，通过计算余弦相似度找出最相关的记忆」

9. **前言结构遵循 SCQA**：前言是吸引读者继续阅读的钩子，按 Situation → Complication → Question（隐含）→ Answer 组织，但**绝对不能把这些标签写成章节标题**，框架是内化的，不是贴标签的：
   - **S – Situation**：一两句交代系统是什么、定位是什么（"PowerMem 是一个为 AI Agent 设计的持久化智能记忆层"）
   - **C – Complication**：说清楚核心矛盾——朴素方案在哪里会出问题，为什么不好解决。通常写成"写入侧…检索侧…"的并列句式，不超过 4 句
   - **A – Answer**：简短概括解法，用 2~3 条要点说核心机制，直接写就好，不要给这段起任何英文标签名
   - **数据表（可选）**：如果有令人印象深刻的基准测试数字（准确率对比、延迟对比），放在 S 或 C 附近做锚定；如果项目没有好看的测评数字，**不要强求加表格**，一句话带过或不提
   - **深度预告（可选，且要自然）**：如果需要，用一句"本文将拆解 X、Y、Z" 收尾，不要写成编号列表，不要单独起一个标题叫"深度预告"

   > **反例**（机械套框架标签）：
   >
   > ```
   > ### SCQA 框架：数据表现
   > ### 核心矛盾
   > ### 解法
   > ### 深度预告
   > ```
   >
   > 这些标签是写作框架的内部术语，写进文章里读者会一头雾水，也会让整篇文章看起来生硬。
   >
   > **正例**（框架内化，自然流动）：
   > "PowerMem 在 LOCOMO 长对话基准上的准确率达 78.7%，延迟仅 1.44s——但这套数字背后的设计是怎么来的？
   >
   > | 维度     | PowerMem | 全量上下文 | 提升       |
   > | -------- | -------- | ---------- | ---------- |
   > | 准确率   | 78.7%    | 52.9%      | +48.8%     |
   > | 检索延迟 | 1.44s    | 17.12s     | 11.9× 更快 |
   >
   > PowerMem 围绕两条主干流程构建：**写入侧**的核心问题是原始对话充满噪声，直接存进向量库只会越堆越乱；**检索侧**的核心问题是单路向量搜索对命名实体天然不友好。本文将深入拆解事实提取、三路混合检索、Ebbinghaus 遗忘曲线三个核心机制。"

10. **不写文章大标题**：文章第一个标题就是 `# 一、前言`（或 `# 一、简介`），不在最开头写 `# PowerMem：AI 智能记忆系统深度解析` 这类大标题。标题过长、定语过多，读者一眼就知道是 AI 写的。

11. **不虚构伪代码**：只给出源码中真实存在的代码片段。如果源码中没有某段逻辑的清晰实现，用文字描述即可；不要自己发明 `# 简化示意` 代码块冒充实现细节。

12. **动机说明整合进段落，不用加粗疑问句**：行文中遇到需要解释设计动机的地方，把「是什么」和「为什么」写成两个独立段落，动机直接作为第二段的第一句引出，不要用 `**为什么不直接存原始消息？**` 这类加粗疑问句——它会让文章读起来像 FAQ，破坏叙事流畅感。也不要开独立 `###` 小节，除非「为什么」需要 4 段以上才能回答清楚。

    > **正例**（分段自然引出动机）：
    >
    > "STM 底层是一个容量有限的双端队列，最旧的消息在头部，最新的在尾部。容量单位是字符数，默认上限 64000 个字符。
    >
    > 使用 STM 而不是直接查 LTM，有两个原因：
    > - 长期记忆的检索需要向量化查询，有一定延迟
    > - 当前会话的最近几轮对话高概率相关，顺序检索更快、更准"

13. **多原因必须用列表，不用分号长句**：当一个设计决策有 2 个或以上**原因/好处/代价**时，必须用无序列表，不要用分号把它们堆成一个长句。

    > **反例**：「这个设计有三个好处：写入零延迟；Ingestion 失败可以重跑；可以对历史消息重新提炼。」  
    > **正例**：
    > 「这个设计有三个好处：
    > - 写入零延迟，接口不阻塞
    > - Ingestion 失败可以重跑，对历史消息没有破坏性
    > - 更换更好的 LLM 后，可以对历史消息重新提炼」

    **注意边界**：这条规则只针对"列举同一事物的多个原因/好处/代价"。用分号描述多个并列实体各自的职责，是正常散文节奏，不受此限制——**但前提是每个实体的描述要足够简短**：
    > **允许（描述简短）**：「写入引擎把原始对话提炼成结构化记忆；检索引擎在问答时把最相关的记忆取回来；遗忘引擎定期淘汰低热度的陈旧记忆；巩固引擎负责去重、合并和层间迁移。」（每段约10字，读者一眼扫过）

    > **不允许（描述太长）**：「会话记忆由独立后台进程在对话中持续更新笔记文件；自动记忆有三条路径，分别由主 Agent 直接写、提取 Agent 后台写、autoDream 定期整合写；Agent 记忆由插件 Agent 自主写入，无后台进程。」（每段已超过20字，且第二项内部还有子列举，堆在一句里极难阅读）
    > ↓ 改为列表：
    > 「三套体系的写入机制各不相同：
    > 1. **会话记忆**：独立后台进程在对话进行中持续更新笔记文件
    > 2. **自动记忆**：三条路径并行——主 Agent 实时直写、提取 Agent 轮次结束后异步补写、autoDream 定期跨会话整合
    > 3. **Agent 记忆**：插件 Agent 自主写入，没有后台提取进程」

    判断方法：把分号换成"其次"或"另外"读一读——语义通顺 → 是原因列举，改用列表；语义断裂 → 是并列实体描述。但即便是并列实体描述，**单项描述超过约 20 字，或者描述中出现子枚举，就必须改成列表**。

13b. **有序 vs 无序列表，按语义选择**：列表的标记类型不是随意的，语义决定形式。

   - **有序列表（`1. 2. 3.`）**：用于有明确先后顺序的内容——执行步骤、触发顺序、阶段序列。判断标准：**调换顺序会改变意思 → 用有序列表**
   - **无序列表（`-`）**：用于平级并列的内容——多个原因、多个特征、多个同等选项。判断标准：**调换顺序不改变意思 → 用无序列表**

   > **反例**（顺序触发用了无序列表）：  
   > "- 90% 时提交折叠规则...  
   > - 95% 时阻塞新请求..."（有先后顺序，应改有序）
   >
   > **正例**：  
   > "1. 90% 时提交折叠规则...  
   > 2. 95% 时阻塞新请求..."

   常见需要有序列表的情形：步骤 1/2/3、阶段 A 然后阶段 B、先判断 X 再判断 Y、"依次尝试三步"。写完列表后检查一下：如果把项目打乱，读者会不会困惑？会 → 改有序。

14. **所有英文必须有中文，裸露英文强制禁止**：正文中任何英文单词或短语，无论是组件名、人名、操作命令、学术术语，都必须满足以下规则之一：

    - **首次出现**：给出「中文名（英文）」格式，后续只使用中文名
    - **通用术语可直接替换**：能找到约定俗成中文译名的，直接用中文，不需要保留英文

    **豁免：约定俗成保留英文的术语**，直接使用英文，不翻译，**也不加括号中文解释**：
    - **Token**（不写"词元"）
    - **LLM**（不写"大语言模型"，可直接用缩写）
    - **MCP**（不写"模型上下文协议"）
    - **API / SDK**（不展开翻译）
    - **Agent**（可用，也可写"智能体"，视语境）
    - **ReAct**（不展开翻译，直接使用）
    - **hook**（不写"钩子"——中文技术社区直接说 hook，翻译成"钩子"反而陌生。写 post-sampling hook、stop hook 等组合形式时也保留英文，不做翻译）
    - **JWT / OAuth / SSO** 等认证协议名（开发者日常直接说这些词，加括号解释反而显得低估读者）
    - **Redis、gRPC、Kafka、PostgreSQL** 等广泛使用的技术产品名（直接用，不写"一种缓存方案"之类的括号注释）

    判断标准：中文技术社区直接用英文说出来的词，不翻译，也不加括号解释。

    > **反例**（括号画蛇添足）：「这个项目用 JWT（一种登录认证方案）做认证，后来换成了 OAuth（另一套认证标准）。」  
    > **正例**（直接用）：「这个项目用 JWT 做认证，后来换成了 OAuth。」

    需要处理的常见类型（不在豁免列表内的）：

    | 类型 | 反例（不允许） | 正例 |
    |---|---|---|
    | 组件名 | Derivative、IngestionService | 派生节点（Derivative）、摄取服务（IngestionService） |
    | 人名 | Tulving、Atkinson & Shiffrin | 图尔文（Endel Tulving）、阿特金森和谢夫林 |
    | 操作命令 | ADD、DELETE | 新增（ADD）、删除（DELETE） |
    | 技术术语 | Cross-Encoder、citation | 交叉编码器（Cross-Encoder）、溯源记录 |
    | 集合/对象名 | Episode | 原始消息，或首次出现时写「原始消息（Episode）」 |

    图示中的节点标签原则上全用中文；Token、MCP、API 等豁免词汇在图中可直接使用英文。

19. **源代码注释不得直接引用英文原文**：行文中提到"源码注释说明了某个设计意图"时，必须将注释翻译成中文后再写进正文，不要直接贴英文原句（如 `` `Forward recall is better than backward recall` ``）。可以说明"源码注释专门说明了这个设计意图"，然后给出中文解释。

    > **反例**：「源代码注释也直接写道：`Forward recall is better than backward recall`。」  
    > **正例**：「源代码里也有注释专门说明了这个设计意图：向前方向（时间更晚的回应）的相关性高于向后方向（时间更早的背景）。」

21. **公式/图优先于代码**：概念性内容（热度权重、算法评分、数学推导）必须用 LaTeX 公式（行内 `$...$`，块级 `$$...$$`）或画图提示来表达，**禁止**用代码块来"解释概念"；代码块只用于展示真实源码片段。

22. **常量分散到上下文，不单独成节**：系统的关键参数（阈值、容量、超参）禁止集中到独立的"关键常量"章节。每个参数在第一次有意义时才引入（紧跟讲述它的机制），并解释这个值的来由。

23. **一标题一主题**：若一个 `###` 子标题下实际包含两个独立机制，必须拆成两个子标题。判断标准：两个机制能否各自单独理解？能 → 必须拆分。

24. **禁止跨节重复**：某机制已在其他章节详细讲解时，当前节删除重复内容，必要时一句"详见第 X 节"带过。判断标准：删掉这段，读者对当前位置的理解会减少吗？不会 → 删。

25. **章节标题不加元描述前缀**：不写 `## 关键设计：三层语义索引`，直接写 `## 三层语义索引`。"关键设计"、"核心机制"、"重要组件"是作者判断，不是内容本身，一律删去。

20. **插入真实图片后，文字必须描述图的实际内容**：当文章中插入了 `<img>` 真实图片（而非画图提示占位）时，紧跟图片的文字段落必须根据图片**实际展示的内容**来写——描述图中可以看到什么、关键的观察结论是什么。不能只写模糊的背景介绍（如"整个系统有两条主干流程"），而是要说"图中左侧是 X 路径，右侧是 Y，底部是 Z"。

    > **反例**：「MemMachine 的核心由三大子系统组成，整个系统有两条完全独立的主干流程：[图片]」  
    > **正例**：「[图片] 图中可以看到系统的三大模块：左侧是情节记忆路径，负责……；右侧是语义记忆路径，负责……；底部是两条路径共用的资源层……」

15. **`>` 引用块的两种用途**：
    - **前置高亮**：在一个机制或设计原则的段落**之前**，用 `>` 先抛出核心结论，让读者带着预期去读后续细节。适合"结论反直觉"或"结论重要到值得先建立印象"的地方。
    - **后置总结**：一段讨论结束后，把提炼性的核心结论或有哲理感的句子用 `>` 单独呈现，起到视觉停顿 + 记忆锚点的作用。

    两种用途都有效，不要只用后置。引用块至少每 2～3 个章节出现一次，不要让全文连续 5 个段落都是纯文字。

26. **用自然中文表达，避免技术腔**：行文中不能出现直接从英文技术词汇翻译而来、读起来别扭的用语。判断标准：把这句话读给一个没看过这篇文章的同事听，他能不能自然理解？

    > **反例**（技术腔）：「活在单次对话里」「高效清桌」「清桌后 AI 恢复上下文」  
    > **正例**（自然中文）：「只在本次对话期间有效」「快速完成上下文压缩」「压缩完成后 AI 接着工作」

    特别注意：从英文动词直接借用而来的"名词化"词语（如"清桌"作动词用）在技术博客里非常容易出现，读起来像机器翻译。优先用中文日常表达：完成、替换、丢弃、保留、读取。

27. **解释机制时必须给一个具体例子**：解释后台进程、钩子函数、异步任务等"看不见"的机制时，仅说"它在后台做 X"还不够——读者需要一个可以想象的具体场景：触发时用户在做什么、系统在运行什么、最终产出了什么文件/状态/结果。

    > **反例**：「会话记忆的写入由独立后台进程负责，在对话进行中持续更新笔记文件。」  
    > **正例**：「会话记忆的写入由独立后台进程负责。举个例子：你和 Claude Code 讨论了 20 条消息，当 Token 总量超过 10,000 时，后台进程悄悄启动，把"当前任务是重构认证模块、已决定用 JWT、下一步是写单测"这类信息整理进笔记文件。你完全感知不到这个过程，但下次上下文压缩时，这份笔记已经备好了。」

16. **删除与主题无关的细节**：写作时遇到某个实现细节，先问"这对读者理解核心设计有帮助吗？"——如果答案是"这只是边角的工程实现，不是这个系统区别于其他系统的地方"，直接删除，不要为了"完整"而填充。

    > **判断标准**：删掉这段，读者对系统的核心理解会减少吗？不会减少 → 删。

29. **输入→输出对齐展示**：解释"用户触发了什么 / 系统响应是什么"类的示例时（如用户说了什么 → AI 记下什么、请求格式 → 存储内容、触发语句 → 执行结果），用以下固定结构：

    - **触发侧**（用户说的话、输入消息、触发语句）：用 `>` 引用块
    - **输出侧**（实际写入的内容、API 响应、存储结构）：用代码块
    - **解释性文字**（为什么、边界条件、设计意图）：用 `>` 引用块，放在该组例子**之后**

    > **反例**（全散文，输入输出混在一起）：  
    > 「用户说不要用模拟数据库，AI 记下来集成测试必须连真实数据库，不能用模拟数据库，原因是曾经发生过事故，这个约定影响整个项目，保存为团队共享记忆。」
    >
    > **正例**（引用块 + 代码块对齐，解释后置）：
    >
    > > "这些测试里不要用模拟数据库——上季度我们就吃过亏。"
    >
    > ```yaml
    > name: TestingPolicy
    > description: 集成测试必须连真实数据库，禁止模拟；源于上季度生产事故
    > type: feedback
    > # 影响整个项目 → 存项目目录共享
    > ```
    >
    > > 为什么存项目目录而不是私有目录？因为这条规则对整个团队都适用，不是个人偏好。

    **同组多例**：同一类型下的多个例子，保持相同的引用块 + 代码块节奏，共同的解释性文字（如"为什么这类信息值得保存"）用 `>` 汇总在所有例子之后，不要每个例子后面都跟一段散文。

    **文件内容示例**：解释某个机制最终写入了什么文件（如会话笔记、记忆文件、配置文件）时，用 `>` 引用块交代触发场景（"此时用户正在做什么、处于什么状态"），用代码块展示文件的实际内容——不要只在表格里描述"每章记什么"，而不给读者看文件长什么样。

    > **反例**（只有表格，看不见文件）：用表格列出十个章节各自记什么，但读者无法想象 `summary.md` 实际是什么格式。  
    > **正例**（引用块交代场景，代码块展示文件）：
    >
    > > 任务：给 auth 模块补集成测试。单元测试已写完，卡在数据库 mock 上。
    >
    > ```markdown
    > # 当前状态
    > 正写集成测试；下一步测 /api/login；db_session 事务隔离未完成
    >
    > # 错误与纠正
    > - 错误：async 函数里用 pytest.raises() 捕获不到
    > - 正确：改成 async with pytest.raises()
    > ```

28. **Markdown 行内格式主动运用**：正文段落不能是纯文字堆叠。关键信息应通过行内格式提升可见度，让读者快速定位重点。三类常见用法：

    - **加粗（`**text**`）**：段落中首次出现的核心概念、关键数字（阈值/容量/延迟）、表格第一列的项目名称。不要对每个名词加粗——只加真正值得"停一下"的词，平均每段 1~2 处。
    - **行内代码（`` `code` ``）**：文件路径（`` `~/.claude/MEMORY.md` ``）、shell 命令（`` `pytest --cov` ``）、字段名（`` `type` ``、`` `description` ``）、具体取值（`` `feedback` ``）。凡是需要**原样输入或精确引用的代码内容**，用行内代码。**禁止**用行内代码包裹中文词语、章节名、概念名等非代码内容（如 `` `当前状态` ``、`` `错误与纠正` `` 这类写法是错的，直接写「当前状态」即可）。
    - **数字高亮**：关键阈值、容量上限、延迟数字，优先用加粗（「软上限约 **2,000 Token**」），让读者扫一眼就能捕捉。

    > **反例**（关键信息淹没在纯文字里）：  
    > 「笔记存储在 ~/.claude/projects/路径/sessionId/session-memory/summary.md，每节上限约 2000 Token，整个文件总上限约 12000 Token。」  
    > **正例**（格式提升可见度）：  
    > 「笔记存储在 `~/.claude/projects/<路径>/<sessionId>/session-memory/summary.md`，每节软上限约 **2,000 Token**，整个文件总上限约 **12,000 Token**。」

    **适度原则**：格式是为了帮助读者，不是装饰。全篇加粗密度过高（每句都有加粗），效果和不加一样——读者视觉疲劳，反而什么都抓不住。判断标准：这个词如果被跳过，读者会不会漏掉核心信息？会 → 加粗；不会 → 留原样。

17. **结构化元素选择指南**：文章中散文、列表、表格、引用块要交替出现，选择规则如下：

    | 内容特征 | 推荐形式 |
    |---|---|
    | 一个事物/机制的描述 | 散文段落 |
    | 同一设计的 2+ 个原因/好处/代价 | 无序列表 |
    | 有明确顺序的步骤 | 有序列表 |
    | 多个方案/维度的横向对比 | 表格 |
    | 多个同类别选项的特征汇总 | 表格 |
    | 核心结论/设计原则/权衡提炼 | `>` 引用块 |
    | 连续 3 段以上纯散文 | 插入代码块/图/表格/引用块打断节奏 |

    **表格的两类高价值场景**，遇到时必须优先用表格而不是散文或列表：

    1. **带括号注释的列表** → 改为两列或三列表格。判断标准：每个列表项后面跟着括号说明（"代码规范（看代码就知道）、git 历史（工具能查）"），说明每项已经有两个属性，用表格更清晰。

       > **反例**（括号堆积散文）：  
       > 「**什么不该保存**：代码规范（看代码就知道）、版本历史（`git log` 能查）、已在规则文件里的内容（重复存储）」  
       > **正例**（两列表格）：
       >
       > | 内容 | 为什么不存 |
       > |------|-----------|
       > | 代码规范 | 看代码就知道 |
       > | 版本历史 | `git log` 更准 |

    2. **多条件触发逻辑** → 用表格展示条件分层（必须满足 / 任一满足 / OR 条件）。列可以根据条件结构增加：条件类型、阈值、附加条件 A、附加条件 B、设计意图。

       > **反例**（条件嵌套散文难读）：  
       > 「必须满足：增长 ≥ 5000 Token，加上以下任一：工具调用 ≥ 3，或本轮无工具调用」  
       > **正例**（多列表格，逻辑清晰）：
       >
       > | 阶段 | 前提（必须） | 附加条件 A（OR） | 附加条件 B（OR） | 意图 |
       > |------|------------|----------------|----------------|------|
       > | 初始化 | Token 首次 > 10000 | — | — | 过滤短对话 |
       > | 持续更新 | 增长 ≥ 5000 Token | 工具调用 ≥ 3 | 本轮无工具调用 | 前者捕实质操作，后者捕自然停顿 |

    3. **字段 / 属性说明** → 用表格，列为「字段名 · 作用 · 注意事项」。判断标准：只要你在用列表逐一说明一个数据结构的多个字段，就该换表格。可选列：示例值、类型、默认值。

       表格里字段名用**加粗**，不用行内代码（`` `name` ``）——代码格式适合正文里引用字段，表格首列已经是字段本身，加粗足够突出，代码块反而显得冗余。

       > **反例**（列表堆字段说明）：  
       > 「- **name**：短标题 - **description**：检索依据，越具体越好 - **type**：四选一类型标签」  
       > **正例**（表格，字段名加粗不加代码块）：
       >
       > | 字段 | 作用 |
       > |------|------|
       > | **name** | 记忆的短标题 |
       > | **description** | 检索的核心依据，越具体越好 |
       > | **type** | 四选一分类标签，决定落盘位置 |
       > | **正文** | 实际记忆内容，自由格式 Markdown |

    4. **两套机制的并排比较** → 用对比表格，行为对比维度，列为两套机制。触发信号：正文出现"A 的做法是 X；B 的做法不同，是 Y"或"原因是两者生命周期不同"这类句式，说明两套机制已经有多个维度可以对比，散文描述会把"是什么"和"为什么"搅在一起，表格能把两者拆开对齐。

       > **反例**（散文把两套机制的是什么和为什么混在一起）：  
       > 「自动记忆在每次用户发消息时，后台异步跑一次检索……Agent 记忆的做法不同：在插件 Agent 被启动的瞬间，系统把索引同步嵌入系统提示词，一次完成。原因是生命周期不同。主 Agent 的会话往往持续很长……插件 Agent 通常被调用来完成一个具体任务，生命周期短……」  
       > **正例**（对比表格，维度清晰）：
       >
       > | 对比维度 | 自动记忆（主 Agent） | Agent 记忆（插件 Agent） |
       > |---------|-------------------|----------------------|
       > | 索引注入 | 会话启动时无条件注入 | Agent 启动瞬间同步嵌入，一次完成 |
       > | 主题文件读取 | 每次发消息，后台异步检索最多 5 个 | Agent 自主判断，按需自取 |
       > | 生命周期 | 长，对话可能持续几十轮 | 短，通常完成一个任务后结束 |
       > | 设计逻辑 | 生命周期长 → 异步按需补充 | 生命周期短 → 一次注入更简洁 |

17b. **表达形式优先级**（非常重要）：选择表达形式时按优先级从高到低依次决策，能用更高优先级的形式说清楚，禁止降级用散文替代。判断方法：把这段文字改成图或表，信息量有损失吗？没有 → 必须改。

    | 优先级 | 形式 | 适用场景 |
    |--------|------|---------|
    | 1（最高） | 画图提示 | 执行流程、系统架构、多路径对比 |
    | 2 | 表格 | 多维度对比、触发条件分层、字段说明 |
    | 3 | `>` 引用块 / 列表 | 核心结论、设计原则；同一事物的多个原因/好处/代价 |
    | 4 | 加粗 | 段落中值得停顿的关键数字或概念 |
    | 5（最低） | 纯正文散文 | 以上均不适用时 |

18. **数字编号子标题**（R1 必须，R2 推荐，R3 可选）：内容较长的章节（正文超过 600 字或包含 3 个以上子主题）必须拆成 `2.1`、`3.1.2` 格式的编号子标题，不允许大段文字无结构堆叠。

    > 子标题是给读者的"路牌"——他们可以先扫一遍标题，决定哪里跳读，哪里精读。没有子标题的长章节等于强迫读者连续阅读。

30. **图先行，解释后置**：凡是文章中包含配图（`<img>` 标签或画图提示代码块），图的说明文字和解释段落必须放在图的**下方**，不得放在上方。图上方只允许有：引出图的一句话（"如下图所示"/"三条路径的关系如图"），或直接过渡到图。

    - **`<img>` 图片**：图片标签前只放一句引出语；图片标签后紧跟描述段落或要点注释
    - **画图提示代码块**（待渲染为图的 prompt）：代码块本身代表图的位置；代码块之后放解释性小结、`>` 关键洞察或补充说明
    - **结构化内容（表格/列表）与图并存时**：先放图，再放表格或列表作为详细展开——读者先看整体视觉，再看文字细节

    > **反例**（解释在图上方）：先写两段解释"这三条路径的关系是……互斥保护是……"，再贴 `<img>` 或画图 prompt。  
    > **正例**（图先，解释在下）：贴图，图后一行 `>` 引用块：「路径一写过则路径二自动跳过——互斥保护避免同一信息被重复提取。」

34. **不在正文里写函数名或类名**：介绍某个机制时，说清楚它做什么、什么时候触发、有什么效果，不要把源码里的函数名或类名写进文章正文（如 `extractSessionMemory`、`buildPrompt`）。函数名是实现细节，版本一变就失效，且对读者理解机制没有帮助。

    > **规则**：如果你在描述一个机制时写下了一个驼峰命名或下划线命名的函数/方法名，问自己：删掉它，读者对这个机制的理解会减少吗？不会 → 删；会 → 换成行为描述。

31. **技术术语以行为为准，不用推测的实现词汇**：描述系统内部机制时，用能从外部观察到的行为（"后台异步任务"、"发射后不管"、"后台运行"），不用未经源码确认的底层实现词汇（"分叉进程"、"子进程"、"fork()"）。实现细节容易随版本变动，行为描述更稳定、更不容易误导读者。

    - **可以用**：后台异步任务、后台运行、异步写入、不阻塞主流程、发射后不管
    - **避免用**（除非源码明确）：分叉进程、fork、子进程、线程、守护进程

    > **规则**：如果你在描述一个机制时写下了"分叉"或"子进程"，先问自己：源码里有没有明确的 `fork()` 或 `subprocess` 调用？没有 → 改成描述行为的词；有 → 可以保留，并标注源码出处。

33. **先解释，再给样例**：介绍任何结构、机制、字段时，先用文字或表格说清楚"是什么、有哪些部分、各自做什么"，再给代码块或具体示例作为验证。不要一上来就贴样例让读者自己猜结构。

    > **反例**（样例先，读者自己推）：直接给出一段代码块，之后才解释"这三个字段的作用各不相同……"  
    > **正例**（解释先，样例后）：先用表格列出字段名和作用，再写「以测试规范这条记忆为例：」，然后给代码块。

35. **推测性结论必须标注**：源码中找不到明确注释或代码依据、只能通过行为推断的机制，不能写成确定事实。处理方式：用 `>` 引用块单独呈现，在结论前明确说明"这是推测"。

    > **反例**（把推测写成确定事实）：  
    > 「@提及 Agent 时，系统会把子 Agent 的记忆注入主 Agent 上下文，让主 Agent 知道子 Agent 的历史知识，方便组织调用参数。」（这段逻辑在源码里没有注释，是推断）  
    >
    > **正例**（`>` 引用块标注推测）：  
    >
    > > 为什么要在主 Agent 的上下文里也注入子 Agent 的记忆？代码里没有注释说明，这里是推测：可能是为了让主 Agent 了解子 Agent 的历史知识，组织调用时传入更准确的上下文；也可能只是工程实现上的顺手设计。

    判断标准：删掉源码，这段话还能靠逻辑推断出来吗？**能 → 标为推测**；不能（有明确代码/注释支撑）→ 可以写成事实。

36. **密集散文的两步改造法**：遇到一段把多个要点堆在一起的密集散文（"A 做了 X，B 则是 Y，Z 的情况下……"），按以下两步改造：

    **第一步**：给每条要点加加粗标签，让读者能快速扫到关键词而不用通读整段。

    **第二步**：如果有一条反直觉结论或总结性原则，用 `>` 引用块单独呈现，放在段落末尾或最前面（前置高亮）。

    > **反例**（密集散文）：  
    > 「自动记忆检索支持两种机制，向量搜索是许多 AI 记忆系统的常见方案，但 Claude Code 只有 200 个文件上限，这个规模让向量方案的三个短板全部暴露：需要引入专门的嵌入模型和向量数据库，200 个文件根本用不上这套设施；向量相似度是统计近似，条数不多时 AI 直接理解语义反而更准；向量匹配识别不出需要类比推理的相关性，只有 LLM 才能判断。」
    >
    > **正例**（加粗标签 + 引用块提炼结论）：
    >
    > 「直觉上向量检索是"更专业"的方案，但 200 个文件的规模恰好让向量方案的三个短板全部暴露：
    > - **基础设施代价**：需要引入专门的嵌入模型和向量数据库，200 个文件的规模根本用不上这套设施
    > - **近似匹配的精度上限**：向量相似度是统计近似，条数不多时，AI 直接理解语义反而更准
    > - **推理型相关性盲区**：向量匹配识别不出"这条记忆和当前场景高度类似"的类比推理，只有 LLM 能判断
    >
    > > 200 个文件，正好落在 AI 直判的甜区：不需要额外基础设施，精度更高，还能处理向量检索天然无法覆盖的推理型关联。」

    **触发信号**：一段 3 行以上纯文字，且 3 个以上要点混在一起，读者无法扫读定位。

32. **禁止章节间交叉引用**：文章正文中不得出现「见第 X 章」「详见 X.X 节」「配套文章 XX」等跳转提示。每个章节应自给自足——读者需要的上下文要么在本节内已有，要么在总览章已建立，不要用引用指针代替内容。同理，不得引用尚未写成的配套文章。

    > **为什么**：交叉引用是作者的偷懒——它把"你去别的地方找"的负担转移给读者，破坏阅读流。如果一个机制确实在两处都需要提及，用一句话概括要点，不要让读者跳章。

    **定稿检查**：搜索全文中的「见第」「详见」「见.*章」「配套文章」，逐一删除或替换为一句内联说明。

## 工作流程（四阶段）

### 阶段 1：定位与选模

**执行**：

1. 确认文章主题和目标读者（想快速上手？想深度理解？想做技术选型？）
2. 选择文章模式：
   - **Mode A**：工具/框架快速入门 → 侧重"是什么 + 怎么用 + 和同类比有什么不同"
   - **Mode B**：架构源码深挖 → 侧重"为什么这样设计 + 关键机制 + 源码实现"
   - **Mode C**：版本/方案对比 → 侧重"设计哲学差异 + 权衡取舍 + 适用场景"
3. 确认有哪些可读的素材（源码目录、论文、README、分析笔记）
4. 生成文章骨架草稿，用户确认后进入下一阶段

### 阶段 2：研究与骨架

**执行**：

1. 阅读源代码/文档，记录：
   - 关键数字（容量上限、阈值、超参、评测结果）
   - 核心数据结构的字段
   - 主要函数的调用关系
2. **与文档/README 交叉核实**：发现不一致，以源码为准，在文章中注明
3. 确定 2~3 个最值得深讲的核心设计（最独特或最容易被误解的部分）
4. 细化大纲（每章节 3~5 个要点），标注哪些地方需要画图、哪些需要代码块
5. 如需配图，先列出"图清单"（每张图展示什么、用哪种图类型）

### 阶段 3：逐节写作

**原则**：

- 一次只写一个章节，写完后暂停等待用户反馈
- 每节完成后自检：是否覆盖了「是什么/为什么/怎么做」三层？是否有具体数字和代码？
- **每节写完后，退后一步看整章乃至整篇文章**：这节和前后节的衔接是否流畅？有没有跨节重复或前后矛盾？新内容有没有让文章整体结构变得不均衡？需要调整时，主动告知用户并提议修改，不要等用户反问。

**前言写法（开头重点关注）**：

- 前言按 SCQA 逻辑流动（Situation → Complication → Answer），但**不要把这些词写成章节标题**，框架是内化的
- 数据表只在有吸引人的基准数字时才加；无好数据时跳过，不要强加表格
- 深度预告如果需要，只用一句话，不要单独设标题、不要写成编号列表
- 禁用套话开头（"今天我们来聊聊 X"、"随着 AI 的发展……"）
- **Complication 段要有画面感**：用类比把核心矛盾变成读者能立刻想象的场景，不要只说"存在问题"。好的类比示例——用白板比喻上下文窗口：
  > 把上下文窗口想象成一块有限大小的白板。Agent 的所有工作记忆——对话历史、工具结果、中间推理——都必须写在这块白板上。白板空间用完时，你必须擦掉旧内容才能写新内容。关键问题是：擦掉什么、保留什么、用什么方式擦？
  
  类比之后紧跟一句系统的解法定位，形成完整的 Complication → Answer 转折。

### 阶段 4：定稿

自检清单：

**结构与内容**
- [ ] 前言自然流动，没有 SCQA/CQRD 框架标签出现在文章里（无"核心矛盾"标题、无"解法 Bullet"标题、无"深度预告"标题）
- [ ] 如果有吸引人的基准数字则有数据表，没有就不加
- [ ] 每个重要机制都覆盖了「是什么/为什么/怎么做」三层，且按读者模式使用对应写法
- [ ] 每个关键参数（容量、阈值）都已与源码核实，数字正确
- [ ] 删除了与核心设计无关的边角实现细节
- [ ] 关键常量（阈值、容量、超参）已分散到各自机制章节，没有独立的"关键常量"汇总节
- [ ] 没有跨节重复——某机制已在其他节讲过的，当前节已删或只一句带过
- [ ] 每个 `###` 子标题下只有一个主题，独立机制已拆分
- [ ] 章节/子标题中没有"关键设计："、"核心机制："等元描述前缀
- [ ] 概念性内容（公式、权重）用 LaTeX 或画图提示，没有用代码块来解释工作原理

**格式与节奏**
- [ ] 表格、代码块（R2/R3）/图示（R1）、散文、引用块四者交替，不连续超过 3 个同类型块
- [ ] 2 个以上原因/好处/代价的地方已改成无序列表，没有分号长句
- [ ] 较长章节已用数字编号子标题（R1 必须，R2 推荐）
- [ ] `>` 引用块每 2～3 节至少出现一次
- [ ] 关键数字（阈值、容量、延迟）已加粗；文件路径、命令、字段名已用行内代码；表格第一列项目名已加粗
- [ ] 全篇加粗密度适度，平均每段不超过 2 处，没有"每句都加粗"导致的视觉疲劳

**语言与图示**
- [ ] 没有 `**为什么 XXX？**` 加粗疑问句，动机说明已整合进段落或子标题
- [ ] 正文中无裸露英文——所有英文首次出现已给出「中文（英文）」格式，后续只用中文；能直接替换为中文的已替换
- [ ] 没有直接引用英文源码注释原文，已翻译成中文并说明"源码注释说明了……"
- [ ] 插入了真实图片的位置，紧跟图片的文字描述了图的实际内容（而非模糊背景介绍）
- [ ] 没有 ASCII 箭头文字流程图，全部改为画图提示代码块
- [ ] 画图提示在代码块内，无 px/尺寸/颜色值，无泳道图，无函数名，图中标签全中文
- [ ] 如有预先画好的图，已插入对应画图提示之后，原画图提示保留未删除
- [ ] （R1）正文中无源代码块

## 三种文章模式

### Mode A：工具/框架快速入门

**骨架**：

```
# 一、前言（SCQA 内化结构，有吸引人数字则加数据表）
# 二、它是什么（定义 + 核心价值 + 类比）
# 三、快速上手（安装 + 最小可跑示例 + 关键 API）
# 四、核心能力（分节讲 3~5 个主要功能，每个配「是什么/为什么/怎么做」三层）
# 五、和同类工具对比（对比表格 + 适用场景结论）
# 六、总结
```

### Mode B：架构源码深挖

**骨架**：

```
# 一、前言（SCQA 内化结构，有吸引人数字则加数据表）
# 二、概要设计（代码结构目录树 + 模块依赖图 + 主干流程鸟瞰，不深入细节）
# 三、[主干流程 A] 详细设计（是什么 → 为什么这样设计 → 怎么实现，逐层拆解）
# 四、[主干流程 B] 详细设计（同上）
# 五、关键设计：[最独特机制]（公式/数据结构 + 直觉解释 + 代码 + 权衡取舍）
# 六、[可选] 特殊子系统深讲
# 七、评测体系（如有论文或 benchmark）
# 八、总结
```

**说明**：

- 主干流程 A/B 根据系统特点命名。数据库系统：写入/查询；编译器：解析/代码生成；Agent 系统：规划/执行。
- **概要设计**是 Mode B 的关键承接层：前言说"是什么和为什么"，概要设计给读者一张地图（代码在哪里、模块怎么连、主流程长什么样），再进入每条流程的详细拆解。没有概要设计，读者会在细节章节中迷失。

**读者模式对 Mode B 的影响**：

| 要素           | R1（纯小白）                              | R2（有技术背景）              | R3（技术专家）           |
| -------------- | ----------------------------------------- | ----------------------------- | ------------------------ |
| 代码块         | 禁止，改用图示 + 公式 + 文字              | 可用简化伪代码（标 # 简化示意） | 可引用真实源码片段        |
| 专业术语       | 首次出现必须类比解释                      | 一句话说清在本系统中的含义    | 直接使用，不解释          |
| 认知科学背景   | 必须介绍（Tulving、Atkinson 等引用模型）  | 一段简述即可                  | 可跳过或一句话带过        |
| 子标题编号     | 必须（2.1, 3.1.2 等）                     | 推荐                          | 可选                     |
| 是什么 vs 为什么 | 必须分段                                | 分段或第一段内明显区分        | 可合并                   |
| `>` 引用块频率 | 每 2 节至少 1 个                          | 每 3 节至少 1 个              | 按需                     |

### Mode C：版本/方案对比

**骨架**：

```
# 一、前言（SCQA 内化结构，对比背景）
# 二、问题根源（旧方案为什么不够好，是什么/为什么/怎么暴露问题）
# 三、[方案/版本 A]（核心设计 + 优势 + 局限）
# 四、[方案/版本 B]（核心设计 + 优势 + 局限）
# 五、关键差异深挖（选 2~3 个维度做横向对比，用对比图）
# 六、适用场景与选型建议
# 七、总结
```

## 画图提示规范（摘要）

> 完整规范和 6 个实战示例见 `references/diagram-style.md`

**五条强制规则**：

1. 节点标签全部用中文（`ShortTermMemory` → 「短期记忆」）
2. 不出现函数名、类名、字段名、英文缩写
3. **prompt 中不写 px、尺寸、颜色值（色号）、字号**——这些渲染细节写了反而限制模型，越少越好
4. **prompt 中不使用英文设计术语**（pill、border-radius、icon badge、chip 等），改为中文描述（"小圆角标签"、"圆角边框"、"圆形徽标"）
5. **用色调描述代替色号**：写"冷灰色调"而不是 `#2C2C2C`，写"蓝色系"而不是 `#3B82F6`——给模型发挥空间，渲染效果更好

**图类型快速对照**（完整版含参考图见 `diagram-style.md`）：

| 场景                   | 图类型                                     |
| ---------------------- | ------------------------------------------ |
| 多层系统架构总览       | 横向多层架构图                             |
| 有阶段划分的处理流程   | 竖向三阶段流程图                           |
| 多维度权重/评分        | 横向条形图 + 说明卡片                      |
| 公式 + 分级条件        | 左右分栏（公式+卡片）                      |
| 两方案对比             | 并排对比图                                 |
| 多阶段内各有子流程     | 横向阶段列流程图                           |
| **4个以上阶段的管道流程** | **卡片+箭头布局（见下方说明）**          |
| 树状结构搜索/遍历      | 树形层次检索路径图                         |
| 跨模块流程（原泳道图） | **竖向三阶段 or 横向阶段列（禁用泳道图）** |

**卡片+箭头布局（多阶段管道的正确画法）**

泳道感的根本原因：**等宽列 + 深色贯通顶部标题栏**，不管怎么描述，这个结构几乎必然渲染成泳道图。

正确做法：把每个阶段改成独立的圆角矩形卡片，卡片之间用箭头连接，高度由内容决定。

Prompt 关键描述模板（直接套用）：
```
# N 张卡片从左到右水平排列，卡片之间用粗箭头（→）连接。
# 禁止使用泳道/等高列结构。每张卡片是独立的圆角矩形，高度由内容决定。
# 整体背景：极浅灰色，无外边框。
# 卡片通用样式：白色底，圆角边框，浅灰描边。
# 顶部：彩色实心圆形徽标（含编号数字）+ 徽标右侧阶段名（加粗）
# 徽标下方：内容区，每条信息用小圆角标签呈现，标签背景色与徽标同色系但更浅
# 卡片 ①「阶段名」  徽标：蓝色系
#   小圆角标签（浅蓝底）：...
# 卡片 ②「阶段名」  徽标：橙色系
#   小圆角标签（浅橙底）：...
```

## 禁止事项

- 禁止：开写前没有询问/确认目标读者类型（R1/R2/R3）
- 禁止：跳过大纲阶段直接写正文
- 禁止：一口气写完全文再给用户看
- 禁止：用"今天我们来聊聊 X"等套话开头
- 禁止：前言用"SCQA"/"核心矛盾"/"解法 Bullet"/"深度预告"等框架标签作为章节标题
- 禁止：项目没有好看的基准数字却强行加数据表
- 禁止：核心机制只说"是什么"，不解释"为什么"和"怎么做"
- 禁止：文章数字未与源码核实
- 禁止：画图 prompt 中写 px、尺寸、颜色值（色号）、字号等渲染细节
- 禁止：画图 prompt 中使用英文设计术语（`pill`、`border-radius`、`icon badge`、`chip` 等），改用中文描述（"小圆角标签"、"圆角边框"、"圆形徽标"）
- 禁止：使用纵向泳道图
- 禁止：图中出现函数名或英文字段名
- **【超级重要】禁止：使用 Mermaid、draw.io、Chart.js、Graphviz、PlantUML 等任何代码化画图工具**——所有图示必须且只能用 `画图提示` 代码块表达；用户说"画个图"也只写 `画图提示`，绝不输出画图 DSL 代码
- 禁止：文章最开头加 `# [系统名]：[定位描述]` 大标题，第一章就是 `# 一、前言`
- 禁止：写源码中不存在的伪代码（`# 简化示意`）冒充实现细节
- 禁止：表格维度名中英混写（写 `关联度` 而不是 `关联度 relatedness`）
- 禁止（R1）：在正文中出现源代码块，用图示 + 公式 + 文字描述代替
- 禁止（R1）：在同一句话或同一段内同时说「是什么」和「为什么」，必须分段
- 禁止（任何模式）：用分号长句列举同一事物的 2 个以上原因/好处/代价，改用无序列表（描述多个并列实体各自职责的分号散文不受此限）
- 禁止（任何模式）：正文中直接使用英文组件名而不给中文译名，尤其是首次出现时
- 禁止（任何模式）：用 `**为什么 XXX？**` 加粗疑问句作为行内标题引导动机说明，改为分段叙述或编号子标题
- 禁止：用代码块解释概念性内容（热度公式、算法权重等），改用 LaTeX 公式或画图提示
- 禁止：把关键常量/阈值/超参集中到单独的"关键常量"节，应分散到各自对应机制的上下文中
- 禁止：一个子标题下覆盖两个可独立理解的机制而不拆分
- 禁止：在某节重复讲解已在其他章节详细讲过的内容
- 禁止：章节或子标题使用"关键设计："、"核心机制："、"重要组件："等元描述前缀
- 禁止：解释"看不见"的机制（后台进程、钩子、异步任务）时只用抽象描述，不给具体例子（触发时用户在做什么、产出了什么）
- 禁止：使用技术腔中文（如"活在 X 里"、"高效清桌"、"清桌后"），改用日常中文表达
- 禁止：写完一节就继续下一节，不回头看整章乃至整篇文章的衔接与均衡性
- 禁止：多个并列实体的描述每项超过 20 字时还用分号散文，改用有序或无序列表
- 禁止：解释含有多个子流程或子路径的复杂机制时，直接跳入细节而不先给主线总览（主线总览应包含一张流程图或对比图，展示各路径的触发条件与执行顺序）
- 禁止：在文章中引用源代码片段时，将代码注释保留为英文原文——必须将注释翻译为中文，确保读者不需要翻译就能理解代码意图；翻译时意思到位即可，不必逐字对应
- 禁止：用行内代码（`` `text` ``）包裹中文关键词、章节名、概念名等非代码内容；行内代码只用于文件路径、命令、字段名、代码取值
- 禁止：能用表格或图讲清楚的内容用大段文字描述替代；凡信息可结构化，必须结构化
- 禁止：源码中找不到明确注释或代码依据的推测性结论，直接写成确定事实；推测必须用 `>` 引用块标注"这里是推测"（见规则 35）
- 禁止：多阶段管道流程图（4 个以上阶段）使用等宽列 + 贯通顶部深色标题栏——这是泳道感的根本原因，无论怎么描述都会渲染成泳道图；改用卡片+箭头布局（见画图提示规范）
