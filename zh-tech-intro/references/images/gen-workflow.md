# 图片生成工作流

> 本文件从 SKILL.md 阶段5独立出来，**仅在用户触发图片生成时加载**。
> 触发词：「生成图片」「把图画出来」「渲染一下」「帮我生图」。

---

## 基本原则

逐张生成、逐张确认，不批量、不跳过用户。每张图经过自动质检后再给用户看，用户确认后再写入文章，再进入下一张。

---

## 第一步：检测并配置环境

```bash
bash scripts/gen-image.sh --check
```

脚本自动检测四样东西，缺失时给出安装指引：

| 工具 | 作用 | 缺失时 |
|------|------|--------|
| `codex` CLI | 驱动 GPT Image 2 | 打印安装命令，需用户手动操作 |
| `gpt-image-2` skill | 图片生成脚本 | `npx skills add https://github.com/agentspace-so/agent-skills --skill gpt-image-2` |
| `picgo` | 上传图床获取 URL | **自动尝试安装**；失败则降级本地路径 |
| picgo 图床配置 | 指定上传目标 | 见下方说明 |

输出 `STATUS: ready` 后继续；否则按提示完成配置后重新运行 `--check`。

> **picgo 图床配置**：脚本检测到本机 PicGo GUI 应用（`~/Library/Application Support/picgo/data.json`）时，自动将其当前默认图床同步到 CLI 配置，无需手动操作。没有 GUI 应用时，运行 `picgo set uploader` 按提示填写图床信息；配置后 `picgo upload 任意图片.png` 验证，看到 `https://` 开头的 URL 说明成功。

---

## 第二步：列出所有画图提示

扫描文章中所有 `# 画图提示` 代码块，同时检查每个代码块之后是否已有 `<img>` 标签。按顺序列出时注明状态：

- ✅ 已有图片（显示现有 URL，询问是否需要因内容变更而重画）
- ⬜ 待生成

只对"待生成"的图执行后续步骤；"已有图片"的默认跳过，除非用户明确要更新。告知用户共 N 张待生成、每张约 1～3 分钟，询问是否现在开始或跳过某几张。

---

## 第三步：逐张生成、质检、确认

### 3.1 生成图片

提取该画图提示代码块文本（去掉 `# ` 前缀）：

```bash
bash scripts/gen-image.sh --prompt "提示文本" --out /tmp/zh-tech-img-N.png --upload
```

### 3.2 自动质检

质检分两个阶段，**第一阶段优先级最高**：

#### 第一阶段：用户迭代要求清单（每次重新生成前必做）

每次用户指出问题，立刻把要求写成清单条目。下次生成后，**先逐条核对这张清单，再做常规质检**。清单里每一条都要明确写「✅ 已修复」或「❌ 未修复」，未修复即不通过，不能跳过。

> 常见踩坑：用户说"删掉这个箭头"——生成后只看整体觉得差不多就过了，没有专门去找那个箭头有没有消失。**必须专门找它，确认消失再过。**

**典型需要追踪的要求类型：**
- 「删除 X 元素」→ 专门确认 X 不在图中
- 「统一箭头颜色」→ 逐条检查所有箭头颜色一致
- 「不要出现 [Image N] 标签」→ 扫描全图有无方括号文字
- 「X 不能连回图内节点」→ 追踪 X 的终点确认出图
- 「是/否标签在正确分支」→ 逐个菱形节点核对两条分支的标签

#### 第二阶段：常规质检

| 检查项 | 具体内容 |
|--------|---------|
| **完整性** | 所有节点、标签、箭头是否都出现？有无漏掉的元素？ |
| **文字准确性** | 逐字读一遍：有无错字（形近字替换如「则→侧」）、乱码、截断、英文混入？重点查多笔画汉字 |
| **连接正确性** | 每条箭头逐一核对：①起点 ②终点 ③方向 ④标注文字 ⑤「是/否」挂在哪个分支（不能只看箭头存在，必须确认方向和终点） ⑥有无悬空节点 ⑦回环/旁路是否真正连到目标 |
| **多余元素** | 图中每个元素是否都在提示词里有依据？尤其检查：`--ref` 改图时旧元素残留、模型自动加的 `[Image N]` 标签、用户要求删除的元素是否真的消失 |
| **视觉整洁度** | 箭头颜色是否统一（用户要求统一时必查）；有无留白失衡、节点拥挤、线条覆盖 |
| **反模式** | 泳道图、旁路悬空、列间箭头乱、层间扇形展开、节点流水编号、终点节点有多余出发箭头 |

质检结果分三档：通过 → 3.3；有瑕疵但可接受 → 附说明展示给用户；有明显问题 → **直接进入 3.3b 自动修复重试，不停下来问用户，但每次都打印 URL**。

> **质检是强制步骤，与生成路径无关**：无论用 `gen-image.sh`、还是切换到 `--ref` 图改模式（gpt-image-2 gen.sh），每次生成后都必须完整执行 3.2 质检并输出质检结论，不能因为"切了路径"就跳过。

### 3.3 先上传，再展示给用户确认

**先上传拿到 URL，再让用户确认**——不要等用户确认后才上传，上传有时间成本。**质检不通过时也要上传**——用户需要看到实际图片才能判断是否接受瑕疵，不要因为质检失败就只给文字描述。

展示格式：

```
第 N 张 / 共 M 张：[图片标题]

[CDN URL]

质检结论：✅ 通过 / ❌ 未通过 / ⚠️ 有以下瑕疵：[简述]
样式得分：X/10 — [一句话说亮点和扣分项]

请确认：
A) 确认，写入文章，继续下一张
B) 重新生成（可告诉我哪里不对）
C) 跳过这张，暂时留空
```

用户选 A → 写入文章（见第四步），继续下一张
用户选 B → 根据反馈调整提示词，回到 3.1（最多 2 次额外重试）
用户选 C → 代码块保留，不插入图片，继续下一张

### 3.3b 自动修复重试（质检不通过时）

**先判断问题类型，再选修复路径：**

| 问题类型 | 说明 | 修复路径 |
|---------|------|---------|
| **视觉/文字问题** | 文字错误、配色不对、元素缺失、留白失衡 | 用 `--ref` 在原图基础上改（见下方"图改模式"） |
| **结构/箭头问题** | 箭头方向错、连接关系错、布局错 | 重写提示词重新生成（`--ref` 无法可靠修改结构） |

> **`--ref` 的能力边界与两难**：`--ref` 能保留视觉风格（配色、图标、卡片设计），适合改文字、换颜色、调细节。但结构改动（箭头方向、连接关系、布局）往往无效——模型会照抄原图结构。反过来，不用 `--ref` 重新生成，结构可以按提示词改，但视觉风格很难完全复现。
>
> **遇到"想保留风格 + 修结构"时**：这是 GPT Image 2 当前的能力边界，没有完美解法。建议方案：
> 1. 先用 `--ref` 多试几次（偶尔能改成功）
> 2. 若多次无效，评估两个选项：a) 接受原图风格但结构有瑕疵；b) 接受重新生成的结构正确但风格略有差异
> 3. 把选择权交给用户，不要无限循环重试

**图改模式（视觉/文字问题时使用）：**

```bash
# 1. 用 gpt-image-2 的 gen.sh 以原图为参考修改
bash ~/.claude/skills/gpt-image-2/scripts/gen.sh \
  --prompt "在这张图基础上只改：[具体说明改什么]，其他完全保持原样" \
  --ref /tmp/原图.png \
  --out /tmp/新图.png \
  --timeout-sec 300

# 2. 手动上传（gpt-image-2 的 gen.sh 没有 --upload 参数）
picgo upload /tmp/新图.png
# 输出的 URL 即为 CDN 地址
```

**重新生成时的常见修复策略：**

| 检测到的问题 | 修复策略 |
|-------------|---------|
| 泳道图结构 | 改用「N 张卡片从左到右水平排列，卡片之间用粗箭头连接，禁止使用泳道/等高列结构」 |
| 旁路节点悬空 | 对每条旁路末端加「→ 向下连入「[汇合节点名]」」 |
| 列间箭头混乱 | 添加独立「列间连接」段落，逐一声明每列出口连到哪列入口 |
| 多列布局内节点各自出箭头 | 列内的子节点不要各自引出箭头到下一列；改成「列/模块之间用一条整体箭头连接，列内子节点只在列内排列，不单独出箭头」——这样箭头数量少、不交叉、视觉更干净 |
| 层间箭头扇形展开 | 加「每层之间只画一条居中垂直向下箭头，不展开连到各节点」 |
| 文字截断或乱码 | 缩短单节点文字，拆分过长标签 |
| 节点有流水编号 | 加「节点不加流水序号，除非画图提示明确要求」 |
| 金字塔层序画反 | 加「底部最宽为第一层，顶部尖顶为第 N 层，从下到上编号递增」 |
| 留白失衡 | 减少某区域节点数量，或拆分密集节点为简洁描述 |
| 终点节点多出一条出发箭头 | 提示词里若没有明确说「此节点引出箭头」，终点节点不得有出发箭头——在提示词里加「此节点无出发箭头，是流程终点」 |
| 箭头颜色不统一 | 用户说「统一颜色」时，在提示词里指定「所有箭头统一为深灰色，不得出现蓝/绿/紫/红混色」 |

最多自动重试 **3 次**（含第一次）。3 次后仍有问题 → 3.3c。

### 3.3c 三次均未通过：询问用户

```
自动修复 3 次后仍有问题：[问题描述]
最近一次结果：[URL]

A) 接受这张（有瑕疵但够用）
B) 告诉我具体哪里不对，我来重新描述一次
C) 跳过这张，暂时留空
D) 查看 3 次生成结果，我来选一张
```

## 第四步：写入文章

写入时，**必须同时做两件事，缺一不可**：
1. 把最终成功的提示词回写到文章代码块（哪怕只改了一个字也要更新）——提示词是图片的"源码"，内容变更重新生成时用的是最新版
2. 在代码块后面插入图片 URL

常见漏洞：只插了 URL，没更新代码块。写入完成后默念一遍"代码块改了吗？"再继续下一张。

写入格式（代码块保留不删）：

```markdown
​```
# 画图提示：[图片标题]（此处为最终生成成功的版本）
# ...
​```

![图片标题](https://cdn.example.com/img-N.png)
```

写入后告知用户，继续下一张。

---

## 联动规则：文章内容变更时主动提醒

文章内容实质性变化（机制描述改了、节点增删了、流程顺序调整了），且该章节有已生成图片时，**必须主动告知用户**：

> 「[章节名] 的内容有变化，对应图片可能已过时，建议删除旧图重新生成，还是先保留？」

用户选重新生成：走完整流程（质检+确认+上传+写入），旧图 `![...]()` 在新图写入后删除。

## 兼容性一览

| 场景 | 结果 |
|------|------|
| codex 未安装 | 打印安装步骤，跳过，文章不变 |
| gpt-image-2 skill 未安装 | 打印安装指引，跳过，文章不变 |
| picgo 未安装 | 自动安装；失败则存本地，插入本地路径（后续手动替换） |
| picgo 未配置图床 | 存本地，打印配置指引，插入本地路径 |
| 全部就绪 | 生成 → 质检 → 上传 → 用户确认 → CDN URL 写入文章 |
