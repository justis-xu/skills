# 项目问卷生成指南

**必须使用 `AskUserQuestion` 工具交互式提问，不得把所有问题以文本一次性列出。**

---

## 整体流程

```
检查全局画像
    ↓
  存在 → 展示给用户确认 → 有修改则更新 → 进入文章专属问卷
  不存在 → 问前说明 → 建立全局画像 → 进入文章专属问卷
    ↓
文章专属问卷（扫系统后生成）
    ↓
写入 context.md
```

---

## 全局画像

### 路径（优先级从高到低）

| 优先级 | 路径 | 说明 |
|--------|------|------|
| 1 | `.aigogoing/zh-tech-intro/profile.md`（项目目录） | 项目级覆盖 |
| 2 | `$HOME/.aigogoing/zh-tech-intro/profile.md` | 用户全局 |

通常写在用户全局路径，只有特殊项目才用项目级覆盖。

### 检查与加载

**有全局画像时**：把画像内容展示给用户，一句话说明它的作用，然后用 `AskUserQuestion` 确认：

> 「我找到了你之前保存的读者/作者画像，用它来规划这篇文章会更准确。请确认它是否还适用：」
>
> 展示画像内容（读者背景、作者背景、语言偏好）
>
> 然后问：「这个画像还准确吗？」
> - A) 完全准确，直接用
> - B) 大部分准确，需要小改
> - C) 需要重新建立

- **A** → 直接进入文章专属问卷
- **B** → 用 `AskUserQuestion` 只问需要修改的部分，更新画像文件
- **C** → 走完整建立流程

**没有全局画像时**：先说明再建立（见下方"问前说明"）。

### 格式

```markdown
# 读者/作者画像

## 典型读者背景
| 维度 | 程度 |
|------|------|
| 技术背景 | [无 / 初级工程师 / 中级工程师 / 高级工程师] |
| AI/LLM 熟悉度 | [不了解 / 知道概念 / 用过工具 / 深度使用或开发] |
| 分布式/系统设计 | [不了解 / 知道基础 / 有经验] |
| 阅读目的 | [了解系统 / 学习设计 / 技术选型 / 深度研究] |
| 读完后的期望结果 | [知道系统能做什么 / 能判断是否值得研究 / 能上手使用 / 能参与开发] |

## 作者技术背景
| 领域 | 熟悉程度 |
|------|---------|
| [领域1] | [了解概念 / 实际使用 / 深度开发] |
| [领域2] | ... |

## 写作动机
- 典型场景：[对外推广 / 内部沉淀 / 技术分享 / 个人总结]
- 发布渠道：[个人博客 / 公司博客 / 团队内部文档 / newsletter / 演讲稿]

## 默认偏好
- 语言：[中文 / 英文]
- 文章深度：[功能层 / 设计层 / 实现层]
- 文章模式：[A 快速入门 / B 架构深挖 / C 方案对比]
```

## 问前说明

**没有全局画像时，建立画像前必须先说明：**

> 「在开始规划这篇文章之前，我想先了解一下你的典型读者是谁、你自己的技术背景是怎样的。
>
> 这些信息会保存成一个画像文件，以后写其他文章时直接加载，不用重复回答。
>
> 我会分几轮问几个问题，每轮 2~3 题，回答完就开始规划。」

然后用 `AskUserQuestion` 开始提问。

## 文章专属问卷（扫系统后生成）

全局画像确认/建立完后，快速扫该系统（5 分钟），生成 **6~10 个系统专属问题**，分 2~3 轮用 `AskUserQuestion` 问完。

**问题维度：**

### A. 读者对该系统核心概念的熟悉程度（3~5 题）

针对这个系统最重要的 3~5 个概念各出一题，选项统一为：
- 没听说过 / 知道大概 / 用过接触过 / 深入了解

### B. 本篇写作目标（3~4 题）

- **写作动机**：这篇文章为什么要写？和你的全局典型场景有什么不同？
- **读者起点**：读者读这篇文章之前，对这个系统了解多少？（完全不知道 / 听说过名字 / 知道能做什么 / 用过但不了解原理）
- **读完期望**：读者读完这篇文章，你希望他们能做到什么？（可能和全局偏好不同）
- **文章模式**：侧重哪个方向？（A 快速入门 / B 架构深挖 / C 方案对比）
- **发布渠道**（可选）：会发布在哪里？（影响术语解释密度和语气）

### C. 作者对该系统各模块的了解（3~4 题）

针对该系统主要模块各出一题，选项统一为：
- 需要你来研究 / 大致了解 / 比较清楚 / 读过源码

## 写入 context.md

系统专属问卷回答完后，把结果写入**文章所在目录**的 `context.md`（仅包含本篇特有内容，全局画像不重复写入）。

```markdown
# 写作上下文

## 系统
[系统名称和一句话定位]

## 文章语言
[中文 / 英文]

## 文章模式
[A / B / C] — [一句话说明]

## 写作目标
[根据本篇回答整理成一句话：动机 + 读完期望]

## 发布渠道
[个人博客 / 公司博客 / 团队内部 / newsletter / 演讲稿 / 未定]

## 读者画像（来自全局画像 + 本篇补充）
| 维度 | 程度 |
|------|------|
| 技术背景 | ... |
| AI 熟悉度 | ... |
| 读这篇文章前对该系统的了解 | [完全不知道 / 听说过名字 / 知道能做什么 / 用过但不了解原理] |
| 读完后的期望结果 | ... |
| [系统专属概念1] | ... |
| [系统专属概念2] | ... |

## 作者知识储备
| 模块 | 了解程度 |
|------|---------|
| [模块1] | ... |
| [模块2] | ... |

## 特别说明
[用户补充的任何其他要求]
```

## 后续会话加载与补充提问

再次打开同一篇文章时：
1. 静默加载全局画像（无需提示）
2. 检查文章目录的 `context.md` → 有则加载，告知用户「已加载上次对本篇的补充说明」；没有则不提
3. 全局画像不存在 → 走完整问卷流程

**补充提问**：规划中发现某维度信息不足，只用 `AskUserQuestion` 补问 1~2 题：
- 属于通用背景（读者习惯、语言偏好）→ 更新全局画像
- 属于本篇特有（这个系统的模块了解程度）→ 写入文章目录的 `context.md`
