# 骨架写作指南

骨架是动笔前的最后确认。写错了方向比写慢更难收，所以骨架不确认不动笔。

## 骨架的颗粒度

骨架必须规划到所有层级，包括子章节（2.1、2.2、3.1……）乃至更深的嵌套层级（3.1.1、3.1.2……）。只列一级章节标题是不够的——每一个子节都要出现在骨架里，缺的层级意味着没想清楚。

每个章节（不论层级）都必须写清楚三件事：

1. **切入角度**：从哪个问题或场景引入
2. **核心内容**：这节要让读者理解什么、记住什么；有哪些关键数字或机制要覆盖
3. **表现形式**：用图、表格、代码块还是散文；有没有具体示例

用户看骨架时应该能判断：**这节写的东西是不是我想要的？内容会不会太深/太浅？**。骨架的颗粒度要细到"如果按这个骨架写，文章基本定型了"——不是只有标题，而是每节的内容说明。

## 告诉用户

明确告诉用户：这是动笔前的最后确认，如果章节结构、侧重点、或某个机制的取舍有问题，现在改比写完再改省力得多。

## 固定约束

**第一章标题必须是「背景」或「前言」**，不得用其他名称（如"介绍"、"概述"、"引言"等）。

## 骨架示例格式

```
# 一、前言
切入：从问题痛点切入（什么场景下会遇到这个问题）
核心内容：交代系统定位、核心矛盾、解法方向
形式：散文 SCQA 结构；如有亮眼基准数字可加数据表

# 二、整体架构
切入：先给全局视图，再拆解各层职责
核心内容：系统由哪几层/模块组成，各自做什么
形式：架构图（画图提示）+ 简表

## 2.1 核心模块 A
切入：从它解决的问题说起
核心内容：它做什么、为什么这样设计、关键数字 X
形式：散文 + 引用块高亮结论

## 2.2 核心模块 B
切入：和 A 的关系（接收 A 的输出）
核心内容：触发条件、工作机制、数字 Y
形式：流程图（画图提示）+ 表格对比两种情况
```

## 大改时的骨架变更

修改涉及结构调整时（新增大章节、调整顺序、拆分/合并子节），写出变更前后的对比：

```
变更前：2.1 短期记忆 → 2.2 长期记忆 → 2.3 检索
变更后：2.1 记忆层级（含短期/长期对比表）→ 2.2 检索机制（独立成节）
原因：短期/长期概念相互依赖，放在一节里对比更清晰；检索逻辑足够独立，值得单独一节
```

用户确认后再执行。
