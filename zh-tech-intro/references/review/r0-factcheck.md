# R0：真实性核查（第一步跑）

文章基于某个代码库/系统时，**每次审查最先跑这一步**——语言层面的问题（连贯性、翻译腔、AI 痕迹）建立在内容是真的这个前提上，先核实事实再打磨表达，顺序不能反。

## 起因

03-记忆系统.md 这篇文章曾经出现过整节编造的内容（4.4.4「恢复模式」，源码里根本不存在这个机制），以及多处数字写错（如把 REM 置信度公式里 `Math.log1p(6)` 写成了 `ln 7`，深度梦境频率维度的输入字段从 `signalCount` 误写成 `recallCount`）。这些错误不是语言问题，常规的 R5/R1/R2/R3 都查不出来——只有对照源码逐条核对才能发现。

## 执行步骤

1. **提取全文所有可核实的claim**：数字（阈值、权重、上限、超时时间）、公式、字段名、函数名、机制描述（"X 决定 Y""A 先于 B 执行""默认值是 Z"）。逐条列出，标注文章里的大致位置。
2. **定位源码**：找到文章所属系统的源码路径。如果不确定，先 `find`/`grep` 定位实际目录（路径可能因改名、挪动而和上次核查时不同）。
3. **逐条核对**：每条 claim 用 grep/Read 在源码里找到对应的常量、函数、字段，确认数值和逻辑是否一致。不能凭记忆或印象判断——必须看到实际代码。
4. **分类结果**：
   - ✅ 准确：不逐条列，统计数量即可
   - ❌ 错误：源码数值/逻辑与文章不一致，给出源码实际情况 + 文章原文位置 + 修改建议
   - ⚠️ 找不到依据：源码里没有对应实现，可能是编造或过度推测，必须删除或改写成`>`引用块标注"这里是推测"

## 常见陷阱：agent 转述的代码结论也要自己验证一遍数学

委托 agent 做 R0 核查时，agent 给出的"源码实际情况"是对代码的转述，不等于对结论的二次验证——尤其是涉及数学转换的地方。

**实例**：本文里 REM 置信度公式的"读取强度"分量，源码是 `Math.log1p(entry.recallCount) / Math.log1p(6)`。第一次核查时 agent 报告"分母常数是 6"，于是把文章里原本正确的 `ln 7` 改成了 `ln 6`——但 `Math.log1p(6)` 等于 `ln(1+6) = ln(7)`，不是 `ln(6)`。agent 转述代码片段是对的，但把 `log1p(6)` 直接读成了"分母是 6"，漏了 `log1p` 是 `ln(1+x)` 这一层语义，反而把原本正确的内容改错了。

**修复**：收到 agent 的核查报告后，凡是报告里出现"分母是 X""系数是 Y"这类从代码片段推导出的数值结论，自己用一遍代码片段重新算一次（尤其留意 `log1p`/`expm1`/`clamp` 这类语义不是字面意思的函数名），不要直接采信转述的结论就去改文章。代码片段本身是一手证据，转述出的"等价于多少"是二手推导，二手推导才是容易出错的地方。

## 执行方式

claim 数量通常较多（一篇技术文章可能有 30~50 条需要核实的细节），逐条核对工作量大，**优先用 Explore 或 general-purpose agent 做这一步**，自己整理 claim 清单后委托给 agent，要求 agent 给出文件路径+行号级别的核实结果，不接受"应该是对的"这类没有源码依据的回答。

## 报告格式

同 R5：发现问题 → 位置 + 源码实际情况 + 候选改法；未发现问题 → 一句话确认"全文 N 条 claim 核查完毕，源码一致"。
