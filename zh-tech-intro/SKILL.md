# 技术介绍文章 Skill

## 触发识别

| 用户信号 | 流程 |
|---------|------|
| "写一篇"、"新建"、"帮我写" | 从0到1 |
| "修改"、"调整"、"改"、提供已有文章路径 | 修改 |
| "审查"、"检查"、"review"、"看看有没有问题" | 审查 |

---

## 从0到1

**Step 1：快速扫系统 + 生成问卷**

→ 查 `references/planning/questionnaire.md`

先检查文章目录是否有 `context.md`：
- **有** → 加载，告知用户「已加载上次的写作上下文，如需修改说一声」，直接进入 Step 2
- **没有** → 快速扫系统（5 分钟），根据系统特点生成 **10~15 个具体问题**（带选项），覆盖：读者对核心概念的熟悉程度、写作目的、作者对各模块的了解深度、文章语言、文章模式（A/B/C）。用户回答后写入 `context.md`。

**Step 2：获取素材**

读源码、README、论文。记录：
- 核心机制 3~5 个
- 关键数字（容量、阈值、评测结果）——必须源码核实
- 容易误解的地方

**Step 3：写详细骨架，等用户确认**

→ 查 `references/planning/skeleton.md`

骨架确认前不动笔。

**Step 4：逐节写作**

→ 查 `references/writing/style-guide.md`（写作规范全集）
→ 查 `references/writing/anti-patterns.md`（禁止事项）
→ 查 `references/writing/diagram-style.md`（画图规范）

每节完成后自检：
- **读者视角连贯性**：读者没有源码背景，能看懂这节吗？凡是依赖"读者知道 X"才能理解的句子，必须在当前位置补交代或在前面铺垫。
- **先总后分**：每节开头先一句核心结论，再展开细节
- **前后衔接**：和上下节是否流畅衔接

完成所有节后 → 自动触发审查流程

---

## 修改

**小改**（措辞 / 事实 / 单节调整）→ 直接执行 → 触发审查

**大改**（结构 / 新增大章节 / 方向变化）：
1. 写骨架变更方案 → 查 `references/planning/skeleton.md`
2. 用户确认
3. 执行修改
4. 触发审查

**技术准确性的改动**：先查源码核实再执行，不等用户追问"你核实了吗"。

---

## 审查

→ 查 `references/review/index.md`

**核心规则：修改后必须重新全篇审查**，不只看改动处。

## 核心原则

1. **骨架先行**：每子节写清楚切入角度/核心内容/表现形式，用户确认后才动笔
2. **读者视角**：读者没有源码背景——章节内依赖隐含上下文的，必须补交代
3. **修改后重审**：任何修改后，全篇重跑审查，不只看改动处
4. **数字核实**：所有数字对照源码，不引用未核实的 README 数字
5. **推测标注**：源码找不到明确依据的结论，用 `>` 引用块标注"这里是推测"
6. **先总后分**：每节开头一句核心结论，再展开细节
7. **禁止画图 DSL**：禁止 Mermaid / draw.io / Graphviz 等，只用 `画图提示` 代码块
8. **画图不写渲染参数**：不写 px / 色号 / 英文设计术语

## 技能自我进化

遇到以下情况，在当前动作完成后向用户提议更新：

| 情况 | 说明 |
|------|------|
| 规则空白 | 做了判断但没有对应规则，且以后还会遇到 |
| 规则需修正 | 现有规则在实践中有例外或反效果 |
| 新踩坑 | 遇到未记录的问题及修法 |

提议格式：
```
💡 发现可以优化的地方：[说明]

建议：[具体内容]

要更新吗？A) 直接加  B) 我来改  C) 不加
```

图片踩坑同时更新 `references/writing/diagram-style.md` 的「常见踩坑」。
