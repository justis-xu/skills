## 前言（完整示例）

> ⚠️ 注意：文章没有大标题，第一章是 `# 一、简介`。数据表出现在简介后半段，而不是第一句。

mem0（发音 "mem-zero"）是一个为 AI Agent 和对话助手设计的**智能记忆层**。核心价值是：给 LLM 提供跨会话的持久化、个性化记忆，同时让写入和检索尽量智能、低噪声。

它提供两种使用方式：

- **自部署**：`Memory` / `AsyncMemory`，本地运行，自选向量库、LLM、Embedding。
- **云端**：`MemoryClient` / `AsyncMemoryClient`，通过 HTTPS API 访问 Mem0 云服务。

mem0 在 LOCOMO 长对话基准上达到 **91.6% 准确率**，Token 消耗是全量上下文方案的**四分之一**，延迟降低 **12 倍**——但更值得工程师关注的问题是：这套数字背后的系统是怎么设计的？

系统面临两个核心矛盾：

* **写入侧**：原始对话充满噪声、重复、矛盾，直接堆进向量库只会让记忆越积越乱。
* **检索侧**：单路向量搜索对命名实体（品牌名、人名、技术术语）天然不友好——"Goland"在向量空间中可能和"GoLand""IDE""编辑器"混在一起，召回不全。

本文将深入拆解 V3 的**事实提取 Prompt 设计**、**Additive 写入策略**、**实体加权衰减公式**，以及 V2→V3 的根本设计哲学转变。

## 概要设计（Mode B 概要设计层示例）

### 代码整体结构

```
mem0/
├── memory/
│   ├── main.py               # Memory / AsyncMemory 主体（3280 行）
│   ├── storage.py            # SQLite 历史库，短期记忆窗口（10 条）

├── llms/                     # 17 个 LLM 提供商（工厂模式）

├── configs/
```

### 检索记忆（主干流程 overview）

检索整体采用**混合检索 + 可选重排序**的架构，同时运行三路信号然后融合：

- **语义向量检索**捕捉语义相关性（"我不吃辣"能召回"用户对辛辣食物过敏"），但对精确名词不稳定
- **BM25 关键词检索**对精确词匹配更可靠，但无法理解近义词
- **实体加权**专门针对命名实体（品牌名、人名、技术术语），对命中实体的记忆额外加分

三路信号各有所长、互为补充。

## V2 vs V3 对比（版本对比写法示例）

> ⚠️ 注意：版本对比用散文 + 要点，而不是两列功能清单。先说旧方案的缺陷，再说新方案的解法，最后点出核心设计哲学转变。

V2 和 V3 的差距，表面上看是工程优化（减少 LLM 调用次数、引入混合检索），但背后有一个更根本的设计哲学转变：**从"尽量精确地维护记忆"到"尽量保守地积累记忆"**。

```
# 画图提示：横向并排对比图，左侧展示 V2 流程，右侧展示 V3 流程。
#
# 左侧容器（V2）：
#   顶部标签：「高延迟 / 高成本」
#   流程节点（纵向）：
#     「消息输入」→「LLM 调用①：提取事实」→「LLM 调用②：ADD/UPDATE/DELETE 决策」→「写入向量库」
#   两个 LLM 节点用橙色高亮，标注：「串行调用，延迟翻倍」
#
# 右侧容器（V3）：
#   顶部标签：「低延迟 / 低成本」
#   流程节点（纵向）：
#     「消息输入」→「LLM 调用（提取 + 关联一步完成）」→「批量向量化 + 实体关联」→「写入向量库」
#   一个 LLM 节点，标注：「一次调用，linked_memory_ids 关联旧记忆」
#
# 底部横跨左右的说明框：
#   「核心变化：V2 主动 DELETE → V3 纯 Additive，误删风险归零；两次 LLM 调用 → 一次」
#
# 风格：白色背景，极简专业，文字标注全部用中文。
```

* **V2 的逻辑**：让 LLM 全权决定哪些记忆该留、哪些该删、哪些该合并。这套逻辑听起来合理，但实践中有一个致命弱点——LLM 做 DELETE 决策时误删率很高，它容易把"用户最近没提到"误判为"这条记忆已经过时"。一旦删错，只能从历史日志里恢复，成本很高。

* **V3 的策略**：彻底放弃 DELETE，转为纯 Additive 模式：**新事实只能新增或与旧事实合并**，绝不主动删除。记忆库会比 V2 略大，但不会因误删而损失信息。真正需要清除记忆的场景，留给用户通过 `Memory.delete()` 显式处理——这是一个从"系统自动决策"到"人工兜底"的责任边界转移。

* **另一个关键改进**：将"提取"和"决策"合并为一次 LLM 调用。V2 中第一次调用提取事实列表，第二次调用判断每条事实应该 ADD/UPDATE/DELETE——两次串行调用，写入延迟基本翻倍。V3 通过 `linked_memory_ids` 字段让 LLM 在提取时就标注关联关系，一次性完成。提示词也从 V2 的几千字扩展到了近 4 万字，覆盖了大量边界情况的少样本示例，用更好的提示词换取了更少的 LLM 调用次数。

两种模式的核心设计哲学截然相反：

| 维度     | 事实提取                       | 程序记忆提取                    |
| -------- | ------------------------------ | ------------------------------- |
| 记忆对象 | 用户的个人事实、偏好、经历     | Agent 的执行步骤与原始输出      |
| 输出格式 | 严格 JSON（`json.loads` 解析） | 结构化自然语言（Markdown 风格） |
| 核心原则 | 「不漏抽」，宁多勿少           | 「不改原文」，原样保存每步输出  |
| 触发时机 | 每次对话轮次后                 | Agent 执行历史需要压缩时        |

> 记忆系统的误删代价远高于冗余代价——一段 26K token 的对话提炼成 7K token 的精华，不仅节省了成本，反而因为噪声减少让检索更准确。
