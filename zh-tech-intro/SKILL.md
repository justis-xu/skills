# 中文技术介绍文章 Skill

## 开始前必须确认：读者类型与写作目的

**如果用户没有说明目标读者或写作目的，必须在开写前询问以下问题（一次性问，不要分开问）：**

> 开始前想确认几个问题：
> 1. 这篇文章面向什么样的读者？（完全不了解这个领域的普通人 / 有技术背景但不熟悉这个系统的工程师 / 读过类似系统想深入理解设计决策的专家）
> 2. 文章发在哪里？（微信公众号 / 技术博客 / 内部文档 / 其他）
> 3. 写作目的是什么？（让读者了解这个系统是什么 / 帮工程师快速上手 / 做技术选型参考 / 展示核心设计的精妙之处）

根据回答选择对应的读者模式：

### 读者模式 R1：纯小白

**特征**：对这个领域完全没有背景，甚至不了解基础概念。

**写作要求**：
- **不放源代码**：用图示、公式、文字描述替代代码块
- **基础概念必须解释**：涉及认知科学、向量检索等专业概念时，先给直觉类比，再给定义
- **多用 `>` 引用块**：作为阶段性结论的高亮，或关键概念的直觉解释
- **英文术语首次出现必须给中文**：用「中文名（英文）」格式，后续只用中文
- **是什么和为什么必须分段**：两者各成独立段落，不得混写进同一句话
- **多原因必须用列表**：一个设计决策有 2 个以上原因时，用无序列表，不用分号长句
- **编号子标题**：较长章节必须使用 `2.1`、`3.1.2` 格式的数字编号子标题，不出现超过 800 字无子标题的段落

**禁止**：源代码块（包括「简化示意」）、不加解释的行业缩写、一句话里塞多个概念

---

### 读者模式 R2：有技术基础

**特征**：会写代码，了解基本 CS 概念，但不熟悉这个具体系统。

**写作要求**：
- 可以包含简化的伪代码，但**必须标注 `# 简化示意`**
- 专业术语不需要从头解释，但仍需一句话说清它在这个系统里的具体含义
- 是什么和为什么可以在一段内写，但要明显区分（通常是：第一句说定义，后续说原因）
- 多原因用列表
- 配图可以比 R1 更简洁

**禁止**：直接贴大段源码、假设读者完全熟悉系统内部设计

---

### 读者模式 R3：技术专家深度解析

**特征**：深入了解同类系统，来看设计决策、权衡取舍、关键数字。

**写作要求**：
- 可以直接引用源码片段（但需要加中文注释说明「为什么」）
- 重点放在设计取舍、非显而易见的机制、关键参数的来源
- 前言可以更简短，快速进入核心

**禁止**：把显而易见的基础概念解释半页

## Hard Gates（必须满足，开写前逐条确认）

1. **选模式**：开写前必须确认文章属于哪种模式（见下方"三种文章模式"）

2. **读风格**：写作前必须读 `references/style-guide.md`

3. **先研究后写作**：至少读过源代码核心文件或论文摘要，记录实际数字（容量、阈值、评测结果）

4. **先问题后方案**：每个核心机制先说"为什么需要"，再说"怎么实现"

5. **画图提示规范**：
   - 必须放在代码块内，第一行写 `# 画图提示：`
   - 背景统一白色
   - 不出现函数名和英文技术术语作为图中可见标签
   - **禁止使用纵向泳道图**
   - **禁止在 prompt 中写 px、尺寸、颜色值等渲染参数**（会限制模型发挥）
   - **禁止用 ASCII 箭头文字代替真正的画图提示**：不要写 `新消息 → 写入队列 → 超限？→ 驱逐` 这类文字流程图，一律改成正式的 `画图提示` 代码块

   如果用户已经提供了预先画好的图片，处理方式是：在对应的画图提示代码块之后插入 `<img>` 标签，**保留画图提示不删除**，由用户决定最终用哪张。

6. **可复现（R2/R3 专属）**：R2/R3 读者文章至少包含 2 个可复制的代码块，配中文注释。R1 读者文章**禁止代码块**，用图示 + 公式 + 文字描述代替所有实现细节。

7. **代码真实性验证**：文中所有数字（容量、阈值、超参、评测结果）必须对照源码或论文原文核实。发现文档与源码不一致时，以源码为准并在文章中注明。禁止直接引用 README 中未经核实的数字。

   > **反例**：某文档说权重是 `0.20/0.25/0.15/0.15/0.10/0.15`，实际源码是 `0.24/0.30/0.15/0.15/0.10/0.06`。必须核实后再写。

8. **内容详细度**：每个核心机制必须覆盖三层。**三层如何写，取决于读者模式**：

   | 层次 | 定义 | R1（纯小白）写法 | R2/R3 写法 |
   |---|---|---|---|
   | **是什么** | 这个机制/组件的定义 | 先给生活类比，再给系统定义，独立一段 | 一句话定义 |
   | **为什么** | 不这样做会有什么问题（先讲痛点） | 独立一段，多个原因用列表 | 可合并进上段 |
   | **怎么做** | 实现细节、关键数字 | 用图示 + 公式 + 文字，**无代码** | 关键代码片段 + 关键数字 |

   > **R1 反例**（三层全混一段）：  
   > "系统使用 STM 存当前会话消息，这是一个字符容量为 64000 的双端队列，因为 LTM 检索延迟高所以用 STM 提供快速访问。"
   
   > **R1 正例**（三层分段）：  
   > （是什么）"短期记忆相当于对话的工作台——把最近几轮消息放在随手可取的地方，而不是每次都去翻历史档案。系统用一个容量有限的双端队列来实现，最旧的消息在头部，最新的在尾部，默认容量上限是 64000 个字符。"  
   >
   > （为什么）"之所以需要单独的短期记忆，有两个原因：一是长期记忆的检索需要先做向量化计算，再去数据库里做近邻搜索，有一定延迟；二是当前对话里最近说的话，高概率和当前问题相关，单独存起来能做更快的顺序检索。"  
   >
   > （怎么做）"写入时，新消息追加到队列尾部。总字符数超过上限后，最早的消息从头部被移出，放进待压缩缓冲区，由后台异步任务负责把旧消息压缩成摘要，主流程不等待——写入延迟只取决于队列操作本身。"

   > **R2/R3 反例**："系统使用混合搜索提高召回率。" ← 缺少为什么和怎么做  
   > **R2/R3 正例**："单路向量搜索的问题是专有名词在向量空间中容易被稀释，召回不全。混合搜索的解法是：`score = vectorWeight * vectorScore + textWeight * textScore`，两路并行执行后合并，专有名词命中率显著提升。"

9. **前言结构遵循 SCQA**：前言是吸引读者继续阅读的钩子，按 Situation → Complication → Question（隐含）→ Answer 组织，但**绝对不能把这些标签写成章节标题**，框架是内化的，不是贴标签的：
   - **S – Situation**：一两句交代系统是什么、定位是什么（"PowerMem 是一个为 AI Agent 设计的持久化智能记忆层"）
   - **C – Complication**：说清楚核心矛盾——朴素方案在哪里会出问题，为什么不好解决。通常写成"写入侧…检索侧…"的并列句式，不超过 4 句
   - **A – Answer**：简短概括解法，用 2~3 条要点说核心机制，直接写就好，不要给这段起任何英文标签名
   - **数据表（可选）**：如果有令人印象深刻的基准测试数字（准确率对比、延迟对比），放在 S 或 C 附近做锚定；如果项目没有好看的测评数字，**不要强求加表格**，一句话带过或不提
   - **深度预告（可选，且要自然）**：如果需要，用一句"本文将拆解 X、Y、Z" 收尾，不要写成编号列表，不要单独起一个标题叫"深度预告"

   > **反例**（机械套框架标签）：
   >
   > ```
   > ### SCQA 框架：数据表现
   > ### 核心矛盾
   > ### 解法
   > ### 深度预告
   > ```
   >
   > 这些标签是写作框架的内部术语，写进文章里读者会一头雾水，也会让整篇文章看起来生硬。
   >
   > **正例**（框架内化，自然流动）：
   > "PowerMem 在 LOCOMO 长对话基准上的准确率达 78.7%，延迟仅 1.44s——但这套数字背后的设计是怎么来的？
   >
   > | 维度     | PowerMem | 全量上下文 | 提升       |
   > | -------- | -------- | ---------- | ---------- |
   > | 准确率   | 78.7%    | 52.9%      | +48.8%     |
   > | 检索延迟 | 1.44s    | 17.12s     | 11.9× 更快 |
   >
   > PowerMem 围绕两条主干流程构建：**写入侧**的核心问题是原始对话充满噪声，直接存进向量库只会越堆越乱；**检索侧**的核心问题是单路向量搜索对命名实体天然不友好。本文将深入拆解事实提取、三路混合检索、Ebbinghaus 遗忘曲线三个核心机制。"

10. **不写文章大标题**：文章第一个标题就是 `# 一、前言`（或 `# 一、简介`），不在最开头写 `# PowerMem：AI 智能记忆系统深度解析` 这类大标题。标题过长、定语过多，读者一眼就知道是 AI 写的。

11. **不虚构伪代码**：只给出源码中真实存在的代码片段。如果源码中没有某段逻辑的清晰实现，用文字描述即可；不要自己发明 `# 简化示意` 代码块冒充实现细节。

12. **动机说明整合进段落，不用加粗疑问句**：行文中遇到需要解释设计动机的地方，把「是什么」和「为什么」写成两个独立段落，动机直接作为第二段的第一句引出，不要用 `**为什么不直接存原始消息？**` 这类加粗疑问句——它会让文章读起来像 FAQ，破坏叙事流畅感。也不要开独立 `###` 小节，除非「为什么」需要 4 段以上才能回答清楚。

    > **正例**（分段自然引出动机）：
    >
    > "STM 底层是一个容量有限的双端队列，最旧的消息在头部，最新的在尾部。容量单位是字符数，默认上限 64000 个字符。
    >
    > 使用 STM 而不是直接查 LTM，有两个原因：
    > - 长期记忆的检索需要向量化查询，有一定延迟
    > - 当前会话的最近几轮对话高概率相关，顺序检索更快、更准"

13. **多原因必须用列表，不用分号长句**：当一个设计决策有 2 个或以上原因/好处/代价时，必须用无序列表，不要用分号把它们堆成一个长句。

    > **反例**：「这个设计有三个好处：写入零延迟；Ingestion 失败可以重跑；可以对历史消息重新提炼。」  
    > **正例**：
    > 「这个设计有三个好处：
    > - 写入零延迟，接口不阻塞
    > - Ingestion 失败可以重跑，对历史消息没有破坏性
    > - 更换更好的 LLM 后，可以对历史消息重新提炼」

14. **所有英文必须有中文，裸露英文强制禁止**：正文中任何英文单词或短语，无论是组件名、人名、操作命令、学术术语，都必须满足以下规则之一：

    - **首次出现**：给出「中文名（英文）」格式，后续只使用中文名
    - **通用术语可直接替换**：能找到约定俗成中文译名的，直接用中文，不需要保留英文

    需要处理的常见类型：

    | 类型 | 反例（不允许） | 正例 |
    |---|---|---|
    | 组件名 | Derivative、IngestionService | 派生节点（Derivative）、摄取服务（IngestionService） |
    | 人名 | Tulving、Atkinson & Shiffrin | 图尔文（Endel Tulving）、阿特金森和谢夫林 |
    | 操作命令 | ADD、DELETE | 新增（ADD）、删除（DELETE） |
    | 技术术语 | Cross-Encoder、citation | 交叉编码器（Cross-Encoder）、溯源记录 |
    | 集合/对象名 | Episode | 原始消息，或首次出现时写「原始消息（Episode）」 |

    图示中的节点标签必须**全用中文**，不允许任何英文出现在图中可见位置。

19. **源代码注释不得直接引用英文原文**：行文中提到"源码注释说明了某个设计意图"时，必须将注释翻译成中文后再写进正文，不要直接贴英文原句（如 `` `Forward recall is better than backward recall` ``）。可以说明"源码注释专门说明了这个设计意图"，然后给出中文解释。

    > **反例**：「源代码注释也直接写道：`Forward recall is better than backward recall`。」  
    > **正例**：「源代码里也有注释专门说明了这个设计意图：向前方向（时间更晚的回应）的相关性高于向后方向（时间更早的背景）。」

20. **插入真实图片后，文字必须描述图的实际内容**：当文章中插入了 `<img>` 真实图片（而非画图提示占位）时，紧跟图片的文字段落必须根据图片**实际展示的内容**来写——描述图中可以看到什么、关键的观察结论是什么。不能只写模糊的背景介绍（如"整个系统有两条主干流程"），而是要说"图中左侧是 X 路径，右侧是 Y，底部是 Z"。

    > **反例**：「MemMachine 的核心由三大子系统组成，整个系统有两条完全独立的主干流程：[图片]」  
    > **正例**：「[图片] 图中可以看到系统的三大模块：左侧是情节记忆路径，负责……；右侧是语义记忆路径，负责……；底部是两条路径共用的资源层……」

15. **`>` 引用块的两种用途**：
    - **前置高亮**：在一个机制或设计原则的段落开头，用 `>` 先给出一句核心结论，让读者带着结论读细节
    - **后置总结**：一段讨论结束后，提炼性的核心结论或有哲理感的句子用 `>` 呈现

    引用块至少每 2～3 个章节出现一次，不要让全文连续 5 个段落都是纯文字。

16. **删除与主题无关的细节**：写作时遇到某个实现细节，先问"这对读者理解核心设计有帮助吗？"——如果答案是"这只是边角的工程实现，不是这个系统区别于其他系统的地方"，直接删除，不要为了"完整"而填充。

    > **判断标准**：删掉这段，读者对系统的核心理解会减少吗？不会减少 → 删。

17. **结构化元素选择指南**：文章中散文、列表、表格、引用块要交替出现，选择规则如下：

    | 内容特征 | 推荐形式 |
    |---|---|
    | 一个事物/机制的描述 | 散文段落 |
    | 同一设计的 2+ 个原因/好处/代价 | 无序列表 |
    | 有明确顺序的步骤 | 有序列表 |
    | 多个方案/维度的横向对比 | 表格 |
    | 多个同类别选项的特征汇总 | 表格 |
    | 核心结论/设计原则/权衡提炼 | `>` 引用块 |
    | 连续 3 段以上纯散文 | 插入代码块/图/表格/引用块打断节奏 |

18. **数字编号子标题**（R1 必须，R2 推荐，R3 可选）：内容较长的章节（正文超过 600 字或包含 3 个以上子主题）必须拆成 `2.1`、`3.1.2` 格式的编号子标题，不允许大段文字无结构堆叠。

    > 子标题是给读者的"路牌"——他们可以先扫一遍标题，决定哪里跳读，哪里精读。没有子标题的长章节等于强迫读者连续阅读。

## 工作流程（四阶段）

### 阶段 1：定位与选模

**执行**：

1. 确认文章主题和目标读者（想快速上手？想深度理解？想做技术选型？）
2. 选择文章模式：
   - **Mode A**：工具/框架快速入门 → 侧重"是什么 + 怎么用 + 和同类比有什么不同"
   - **Mode B**：架构源码深挖 → 侧重"为什么这样设计 + 关键机制 + 源码实现"
   - **Mode C**：版本/方案对比 → 侧重"设计哲学差异 + 权衡取舍 + 适用场景"
3. 确认有哪些可读的素材（源码目录、论文、README、分析笔记）
4. 生成文章骨架草稿，用户确认后进入下一阶段

### 阶段 2：研究与骨架

**执行**：

1. 阅读源代码/文档，记录：
   - 关键数字（容量上限、阈值、超参、评测结果）
   - 核心数据结构的字段
   - 主要函数的调用关系
2. **与文档/README 交叉核实**：发现不一致，以源码为准，在文章中注明
3. 确定 2~3 个最值得深讲的核心设计（最独特或最容易被误解的部分）
4. 细化大纲（每章节 3~5 个要点），标注哪些地方需要画图、哪些需要代码块
5. 如需配图，先列出"图清单"（每张图展示什么、用哪种图类型）

### 阶段 3：逐节写作

**原则**：

- 一次只写一个章节，写完后暂停等待用户反馈
- 每节完成后自检：是否覆盖了「是什么/为什么/怎么做」三层？是否有具体数字和代码？

**前言写法（开头重点关注）**：

- 前言按 SCQA 逻辑流动（Situation → Complication → Answer），但**不要把这些词写成章节标题**，框架是内化的
- 数据表只在有吸引人的基准数字时才加；无好数据时跳过，不要强加表格
- 深度预告如果需要，只用一句话，不要单独设标题、不要写成编号列表
- 禁用套话开头（"今天我们来聊聊 X"、"随着 AI 的发展……"）

### 阶段 4：定稿

自检清单：

**结构与内容**
- [ ] 前言自然流动，没有 SCQA/CQRD 框架标签出现在文章里（无"核心矛盾"标题、无"解法 Bullet"标题、无"深度预告"标题）
- [ ] 如果有吸引人的基准数字则有数据表，没有就不加
- [ ] 每个重要机制都覆盖了「是什么/为什么/怎么做」三层，且按读者模式使用对应写法
- [ ] 每个关键参数（容量、阈值）都已与源码核实，数字正确
- [ ] 删除了与核心设计无关的边角实现细节

**格式与节奏**
- [ ] 表格、代码块（R2/R3）/图示（R1）、散文、引用块四者交替，不连续超过 3 个同类型块
- [ ] 2 个以上原因/好处/代价的地方已改成无序列表，没有分号长句
- [ ] 较长章节已用数字编号子标题（R1 必须，R2 推荐）
- [ ] `>` 引用块每 2～3 节至少出现一次

**语言与图示**
- [ ] 没有 `**为什么 XXX？**` 加粗疑问句，动机说明已整合进段落或子标题
- [ ] 正文中无裸露英文——所有英文首次出现已给出「中文（英文）」格式，后续只用中文；能直接替换为中文的已替换
- [ ] 没有直接引用英文源码注释原文，已翻译成中文并说明"源码注释说明了……"
- [ ] 插入了真实图片的位置，紧跟图片的文字描述了图的实际内容（而非模糊背景介绍）
- [ ] 没有 ASCII 箭头文字流程图，全部改为画图提示代码块
- [ ] 画图提示在代码块内，无 px/尺寸/颜色值，无泳道图，无函数名，图中标签全中文
- [ ] 如有预先画好的图，已插入对应画图提示之后，原画图提示保留未删除
- [ ] （R1）正文中无源代码块

## 三种文章模式

### Mode A：工具/框架快速入门

**骨架**：

```
# 一、前言（SCQA 内化结构，有吸引人数字则加数据表）
# 二、它是什么（定义 + 核心价值 + 类比）
# 三、快速上手（安装 + 最小可跑示例 + 关键 API）
# 四、核心能力（分节讲 3~5 个主要功能，每个配「是什么/为什么/怎么做」三层）
# 五、和同类工具对比（对比表格 + 适用场景结论）
# 六、总结
```

### Mode B：架构源码深挖

**骨架**：

```
# 一、前言（SCQA 内化结构，有吸引人数字则加数据表）
# 二、概要设计（代码结构目录树 + 模块依赖图 + 主干流程鸟瞰，不深入细节）
# 三、[主干流程 A] 详细设计（是什么 → 为什么这样设计 → 怎么实现，逐层拆解）
# 四、[主干流程 B] 详细设计（同上）
# 五、关键设计：[最独特机制]（公式/数据结构 + 直觉解释 + 代码 + 权衡取舍）
# 六、[可选] 特殊子系统深讲
# 七、评测体系（如有论文或 benchmark）
# 八、总结
```

**说明**：

- 主干流程 A/B 根据系统特点命名。数据库系统：写入/查询；编译器：解析/代码生成；Agent 系统：规划/执行。
- **概要设计**是 Mode B 的关键承接层：前言说"是什么和为什么"，概要设计给读者一张地图（代码在哪里、模块怎么连、主流程长什么样），再进入每条流程的详细拆解。没有概要设计，读者会在细节章节中迷失。

**读者模式对 Mode B 的影响**：

| 要素           | R1（纯小白）                              | R2（有技术背景）              | R3（技术专家）           |
| -------------- | ----------------------------------------- | ----------------------------- | ------------------------ |
| 代码块         | 禁止，改用图示 + 公式 + 文字              | 可用简化伪代码（标 # 简化示意） | 可引用真实源码片段        |
| 专业术语       | 首次出现必须类比解释                      | 一句话说清在本系统中的含义    | 直接使用，不解释          |
| 认知科学背景   | 必须介绍（Tulving、Atkinson 等引用模型）  | 一段简述即可                  | 可跳过或一句话带过        |
| 子标题编号     | 必须（2.1, 3.1.2 等）                     | 推荐                          | 可选                     |
| 是什么 vs 为什么 | 必须分段                                | 分段或第一段内明显区分        | 可合并                   |
| `>` 引用块频率 | 每 2 节至少 1 个                          | 每 3 节至少 1 个              | 按需                     |

### Mode C：版本/方案对比

**骨架**：

```
# 一、前言（SCQA 内化结构，对比背景）
# 二、问题根源（旧方案为什么不够好，是什么/为什么/怎么暴露问题）
# 三、[方案/版本 A]（核心设计 + 优势 + 局限）
# 四、[方案/版本 B]（核心设计 + 优势 + 局限）
# 五、关键差异深挖（选 2~3 个维度做横向对比，用对比图）
# 六、适用场景与选型建议
# 七、总结
```

## 画图提示规范（摘要）

> 完整规范和 6 个实战示例见 `references/diagram-style.md`

**三条强制规则**：

1. 节点标签全部用中文（`ShortTermMemory` → 「短期记忆」）
2. 不出现函数名、类名、字段名、英文缩写
3. **prompt 中不写 px、尺寸、颜色值、字号**——这些渲染细节写了反而限制模型，越少越好

**图类型快速对照**（完整版含参考图见 `diagram-style.md`）：

| 场景                   | 图类型                                     |
| ---------------------- | ------------------------------------------ |
| 多层系统架构总览       | 横向多层架构图                             |
| 有阶段划分的处理流程   | 竖向三阶段流程图                           |
| 多维度权重/评分        | 横向条形图 + 说明卡片                      |
| 公式 + 分级条件        | 左右分栏（公式+卡片）                      |
| 两方案对比             | 并排对比图                                 |
| 多阶段内各有子流程     | 横向阶段列流程图                           |
| 树状结构搜索/遍历      | 树形层次检索路径图                         |
| 跨模块流程（原泳道图） | **竖向三阶段 or 横向阶段列（禁用泳道图）** |

## 禁止事项

- 禁止：开写前没有询问/确认目标读者类型（R1/R2/R3）
- 禁止：跳过大纲阶段直接写正文
- 禁止：一口气写完全文再给用户看
- 禁止：用"今天我们来聊聊 X"等套话开头
- 禁止：前言用"SCQA"/"核心矛盾"/"解法 Bullet"/"深度预告"等框架标签作为章节标题
- 禁止：项目没有好看的基准数字却强行加数据表
- 禁止：核心机制只说"是什么"，不解释"为什么"和"怎么做"
- 禁止：文章数字未与源码核实
- 禁止：画图 prompt 中写 px、尺寸、颜色值、字号等渲染细节
- 禁止：使用纵向泳道图
- 禁止：图中出现函数名或英文字段名
- 禁止：文章最开头加 `# [系统名]：[定位描述]` 大标题，第一章就是 `# 一、前言`
- 禁止：写源码中不存在的伪代码（`# 简化示意`）冒充实现细节
- 禁止：表格维度名中英混写（写 `关联度` 而不是 `关联度 relatedness`）
- 禁止（R1）：在正文中出现源代码块，用图示 + 公式 + 文字描述代替
- 禁止（R1）：在同一句话或同一段内同时说「是什么」和「为什么」，必须分段
- 禁止（任何模式）：用分号长句列举 2 个以上的原因/好处/代价，改用无序列表
- 禁止（任何模式）：正文中直接使用英文组件名而不给中文译名，尤其是首次出现时
- 禁止（任何模式）：用 `**为什么 XXX？**` 加粗疑问句作为行内标题引导动机说明，改为分段叙述或编号子标题
