# 完整禁止事项清单

> 从 SKILL.md 拆分出来，内容完整保留。写作过程中遇到拿不准的地方，查阅此文件。

---

- 禁止：开写前没有询问/确认目标读者类型（R1/R2/R3）
- 禁止：跳过大纲阶段直接写正文
- 禁止：一口气写完全文再给用户看
- 禁止：用"今天我们来聊聊 X"等套话开头
- 禁止：前言用"SCQA"/"核心矛盾"/"解法 Bullet"/"深度预告"等框架标签作为章节标题
- 禁止：项目没有好看的基准数字却强行加数据表
- 禁止：核心机制只说"是什么"，不解释"为什么"和"怎么做"
- 禁止：文章数字未与源码核实
- 禁止：画图 prompt 中写 px、尺寸、颜色值（色号）、字号等渲染细节
- 禁止：画图 prompt 中使用英文设计术语（`pill`、`border-radius`、`icon badge`、`chip` 等），改用中文描述（"小圆角标签"、"圆角边框"、"圆形徽标"）
- 禁止：使用纵向泳道图
- 禁止：图中出现函数名或英文字段名
- **【超级重要】禁止：使用 Mermaid、draw.io、Chart.js、Graphviz、PlantUML 等任何代码化画图工具**——所有图示必须且只能用 `画图提示` 代码块表达；用户说"画个图"也只写 `画图提示`，绝不输出画图 DSL 代码
- 禁止：文章最开头加 `# [系统名]：[定位描述]` 大标题，第一章就是 `# 一、前言`
- 禁止：写源码中不存在的伪代码（`# 简化示意`）冒充实现细节
- 禁止：表格维度名中英混写（写 `关联度` 而不是 `关联度 relatedness`）
- 禁止（R1）：在正文中出现源代码块，用图示 + 公式 + 文字描述代替
- 禁止（R1）：在同一句话或同一段内同时说「是什么」和「为什么」，必须分段
- 禁止（任何模式）：用分号长句列举同一事物的 2 个以上原因/好处/代价，改用无序列表（描述多个并列实体各自职责的分号散文不受此限）
- 禁止（任何模式）：正文中直接使用英文组件名而不给中文译名，尤其是首次出现时
- 禁止（任何模式）：用 `**为什么 XXX？**` 加粗疑问句作为行内标题引导动机说明，改为分段叙述或编号子标题
- 禁止：用代码块解释概念性内容（热度公式、算法权重等），改用 LaTeX 公式或画图提示
- 禁止：把关键常量/阈值/超参集中到单独的"关键常量"节，应分散到各自对应机制的上下文中
- 禁止：一个子标题下覆盖两个可独立理解的机制而不拆分
- 禁止：在某节重复讲解已在其他章节详细讲过的内容
- 禁止：章节或子标题使用"关键设计："、"核心机制："、"重要组件："等元描述前缀
- 禁止：解释"看不见"的机制（后台进程、钩子、异步任务）时只用抽象描述，不给具体例子（触发时用户在做什么、产出了什么）
- 禁止：使用技术腔中文（如"活在 X 里"、"高效清桌"、"清桌后"），改用日常中文表达
- 禁止：写完一节就继续下一节，不回头看整章乃至整篇文章的衔接与均衡性
- 禁止：多个并列实体的描述每项超过 20 字时还用分号散文，改用有序或无序列表
- 禁止：解释含有多个子流程或子路径的复杂机制时，直接跳入细节而不先给主线总览（主线总览应包含一张流程图或对比图，展示各路径的触发条件与执行顺序）
- 禁止：在文章中引用源代码片段时，将代码注释保留为英文原文——必须将注释翻译为中文，确保读者不需要翻译就能理解代码意图；翻译时意思到位即可，不必逐字对应
- 禁止：在文章中引用源代码片段时，将代码注释保留为英文原文——必须将注释翻译为中文，确保读者不需要翻译就能理解代码意图；翻译时意思到位即可，不必逐字对应
- 禁止：用行内代码（`` `text` ``）包裹中文关键词、章节名、概念名等非代码内容；行内代码只用于文件路径、命令、字段名、代码取值
- 禁止：能用表格或图讲清楚的内容用大段文字描述替代；凡信息可结构化，必须结构化
- 禁止：源码中找不到明确注释或代码依据的推测性结论，直接写成确定事实；推测必须用 `>` 引用块标注"这里是推测"
- 禁止：多阶段管道流程图（4 个以上阶段）使用等宽列 + 贯通顶部深色标题栏——这是泳道感的根本原因，无论怎么描述都会渲染成泳道图；改用卡片+箭头布局（见 diagram-style.md）
