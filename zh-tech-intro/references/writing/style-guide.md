# 写作风格指南（中文技术深度介绍文章）

> 本指南基于 MemoryOS、PowerMem、mem0 等 AI 系统介绍文章的写作实践归纳而来，适用于需要阅读源代码或论文后撰写的技术深度文章。

---

## 作者人设（一句话）

工程师视角的技术分析者：读过源码、踩过坑、关注设计决策背后的取舍；写法像"我把代码拆给你看，顺便解释为什么要这么设计"，不端着，但有深度。

---

## 风格指纹（读者一眼能认出的点）

### 叙事逻辑

- **先问题，后方案**：每个核心机制，先说"这里有什么麻烦"，再说"它怎么解决"。不要上来就介绍实现。
- **数据锚定**：用具体数字替代形容词。不写"容量很大"，写"单用户可存 ~1500 轮对话（约 3000 个 QA 对）"。
- **直觉解释 + 技术细节**：先用类比或生活场景让读者建立直觉，再给源码/公式。
- **承上启下**：每节结尾用一句话点出"下一节要解决什么"，帮读者维持连续阅读节奏。
- **常量分散到上下文，不单独成节**：不要把系统的关键参数（阈值、容量、超参）统一汇进"关键常量"一节。每个参数在第一次有实际意义的地方引入（紧跟讲述它的机制），并解释这个值从哪里来、为什么是这个数。读者在需要用到时看到，比在孤立的常量表里看到更容易理解和记忆。

```
❌ 反例（集中汇总，脱离上下文）：

### 关键常量
- GLOBAL_SEARCH_TOPK = 10
- DIRECTORY_DOMINANCE_RATIO = 1.2
- MAX_CONVERGENCE_ROUNDS = 3
```

```
✅ 正例（分散到各自对应的机制里）：

全局搜索取前 10 个结果，按类型分流：L2 文件进候选集，
L0/L1 目录进递归起始点优先队列。

……（递归终止小节）……

递归终止有两个条件：
- 目录分数 ≤ 最佳候选文件 × 1.2，认为好文件已找到，停止展开
- 连续 3 轮 Top-K 结果无变化，整体收敛，退出循环
```

- **一标题一主题**：如果一个 `###` 子标题下实际讲了两个独立机制，拆成两个子标题。判断标准：这两个机制能各自独立被理解、互不依赖吗？能 → 必须拆开。

- **微型子节不单独成节**：一节只有 1~2 句话或一张小表，不要给它独立的子标题——直接折进上级段落，或和相邻的短节合并。频繁的空壳标题会让读者感觉"目录膨胀但没有内容"。判断标准：删掉标题直接把内容写进上一节，读者理解会减少吗？不会 → 合并。

```
❌ 反例（四个只有 1-2 句话的微型子节）：

#### 3.2.2.1 触发时机
每轮对话结束后触发，即查询循环全部走完的那一刻。

#### 3.2.2.2 异步启动
启动后主流程不等待，直接继续。

#### 3.2.2.3 节流与互斥
（一张 2 行的表格）

#### 3.2.2.4 尾随提取
（一张 3 行的表格）

✅ 正例（合并成一个有实质内容的子节）：

#### 3.2.2.1 触发、节流与并发控制
每轮查询循环全部走完时触发……（触发定义）……后台异步启动……（节流与互斥合并成表）……（尾随提取合并进来）
```

- **端到端示例前置，不在末尾补"格式节"**：如果一个操作涉及多个步骤（例如"先写文件，再更新索引"），在进入各类子类型讲解之前，先给一个完整的端到端示例，让读者先看到整体格式。不要在介绍完所有子类型后再补一节"格式说明"——那时读者对格式的疑惑早已积累，且此时再解释会显得像补丁。

- **流程描述不用"阶段/步骤"作为组织结构**：描述一个系统做了哪几件事时，直觉上的写法是按执行顺序枚举（Phase 1 → Phase 2 → Phase 3……），但这会变成流水账——读者知道了"先做什么再做什么"，却不知道"为什么这么设计"。更好的组织方式是：识别出真正独立的 2~3 件核心工作，用"第一件事……第二件事……"自然连接，每件事重点说清楚**做什么**和**为什么**，执行顺序如果重要可以在最后用一句话说明（"三件事的顺序有理由：先扫描才知道要合并什么，合并完才知道索引怎么更新"）。

```
❌ 反例（按阶段枚举，流水账）：

#### 四阶段工作流
| Phase 1 | 定向 | 扫描记忆目录 |
| Phase 2 | 收集 | 从三类来源找新信息 |
| Phase 3 | 整合 | 合并碎片 |
| Phase 4 | 精简 | 更新索引 |

**Phase 2：…**（展开表格）
**Phase 3：…**（展开表格）
**Phase 4：…**（展开表格）

✅ 正例（以"做什么+为什么"组织）：

每次运行前先扫描现有目录，搞清楚"已有什么"再动手。然后做三件事：

第一件事是**扫描三类来源，找需要新增或修正的内容**……（表格+why）

第二件事是**合并碎片**……（表格+why）

第三件事是**精简索引**……（表格+why）

> 顺序有理由：先扫描才知道要合并什么，合并完才知道索引怎么更新。
```

```
❌ 反例（两个独立机制挤在一个标题下）：

### 3.4 Rerank 与热度加权
```

```
✅ 正例（各自独立成节）：

### 3.4 Rerank
### 3.5 热度加权
```

- **禁止跨节重复**：某个机制已在其他章节详细讲解时，当前节删除重复内容，必要时一句"详见第 X 节"带过。判断标准：删掉这段，读者对当前位置的理解会减少吗？不会 → 删。

```
❌ 反例（第四章重复讲第三章已有的检索效率内容）：

## 四、三层语义索引
...（介绍三层结构）...

### 4.4 三层索引的检索效率
层次检索时，L0 低分子树直接跳过，L1 再筛，L2 精确命中...
（← 这段实质内容已在第三章"层次检索"节讲过）
```

```
✅ 正例（第四章只讲结构本身，检索逻辑不重复）：

## 四、三层语义索引
...（介绍三层结构，如何被检索利用详见第三节）...
```

### 叙事结构（非常稳定）

Mode B 架构深挖类的典型骨架节奏：

```
前言：数据对标 + 核心矛盾（读者为什么应该关心？）
↓
概要设计：代码结构目录树 + 模块依赖图 + 主干流程鸟瞰（告诉读者"地图长什么样"）
↓
主干流程A 详细设计（数据怎么进去）→ 主干流程B 详细设计（数据怎么出来）
↓
关键设计：最独特/最容易误解的一两个机制（深挖）
↓
总结：关键结论（3 条以内要点）+ 下一步可以做什么
```

**为什么需要概要设计层**：前言给了"是什么和为什么"，但读者还不知道代码在哪里、模块怎么连。概要设计用一棵目录树 + 一张依赖图 + 主流程的 overview（不展开细节），让读者在进入深挖之前先有全局地图，避免在细节章节中迷失。

前言**按 SCQA 逻辑**（Situation → Complication → Answer）组织，但**不要把 S/C/A 写成章节标题**，也不要出现"核心矛盾"、"解法 Bullet"、"深度预告"这类框架标签作为标题——这些是写作框架的内部术语，写进文章会让读者困惑。

数据表**可选**：项目有令人印象深刻的基准测试数字（与基线的显著对比）时才加；如果数字不够吸引人，一句话带过或直接跳过，不要为了"符合格式"强行加表格。

## 语言风格

### 中英文混写原则

- **保留英文**：工具名（FAISS、Redis）、协议（gRPC）、算法名（LRU）、代码中的变量/函数名
- **翻译为中文**：概念、模块名、图中的节点标签
- **禁止中英夹杂作为说明性文字**：不写"我们用了一个 special 的 algorithm"

**约定俗成保留英文的术语**（不翻译，直接使用英文）：

| 术语 | 不要写成 | 原因 |
|---|---|---|
| Token | 词元、令牌 | 行业通用，翻译后反而陌生 |
| MCP | 模型上下文协议 | 已作为专有名词流通 |
| API | 接口协议 | 直接用英文缩写即可 |
| SDK | 软件开发工具包 | 同上 |
| Agent | 智能体（可用，但 Agent 更常见） | 视语境选择，不强制翻译 |

> 判断标准：如果这个词在中文技术社区里被读者直接用英文说出来（"这个 Token 超了"、"用 MCP 接进来"），就不要翻译。

### 句式节奏

- 段落短：正文段落通常 2~4 行；超过 5 行考虑拆分或变成列表/代码块
- 短句优先：长句用分号/破折号拆开，不堆砌从句
- 避免模糊词：不用"很多"、"相当于"、"差不多"，换成"~N 条"、"等价于"、"约 X%"

### "为什么"的写法：分段叙述，不用加粗疑问句

需要解释某个设计决策的动机时，**把「是什么」和「为什么」分成两个独立段落**，动机作为第二段的开头自然引出。

不要用加粗疑问句（`**为什么不直接存原始消息？**`）——它让文章读起来像 FAQ，破坏叙事节奏。也不要开独立的 `###` 小节，除非「为什么」需要 4 段以上才能说清楚。

```
✅ 正例（分段自然引出动机）：

STM 底层是一个容量有限的双端队列，最旧的消息在头部，最新的在尾部。容量单位是字符数，默认上限 64000 个字符。

使用 STM 而不是直接查 LTM，有两个原因：
- 长期记忆的检索需要向量化查询，有一定延迟
- 当前会话的最近几轮对话高概率相关，顺序检索更快、更准
```

```
❌ 反例（加粗疑问句，读起来像 FAQ）：

STM 底层是一个容量有限的双端队列。

**为什么不直接查 LTM？**

原因是……
```

当一个设计决策有 2 个以上的**原因/好处/代价**时，必须用**无序列表**，不用分号长句：

```
✅ 正例（列表）：
这个设计有三个好处：
- 写入零延迟，接口不阻塞
- Ingestion 失败可以重跑，对历史消息没有破坏性
- 更换更好的 LLM 后，可以对历史消息重新提炼

❌ 反例（分号长句堆原因）：
这个设计有三个好处：写入零延迟；Ingestion 失败可以重跑；可以重新提炼。
```

注意：这条规则针对的是"列举同一事物的多个原因/好处/代价"。**用分号描述多个并列实体各自的职责，是正常的散文节奏，不受此限制**：

```
✅ 正例（并列实体描述，分号合理）：
写入引擎把原始对话提炼成结构化记忆；检索引擎在问答时把最相关的记忆取回来；
遗忘引擎定期淘汰低热度的陈旧记忆；巩固引擎负责去重、合并和层间迁移。

（这是在描述 4 个引擎各自做什么，不是在列举"某一个设计的多个原因"）
```

判断方法：把分号换成"其次"或"另外"读一读——如果语义通顺，说明是原因列举，改用列表；如果语义断裂，说明是并列实体描述，保留分号散文。

### `>` 引用块的两种用途

**前置高亮**：在一个机制或设计原则的段落之前，用 `>` 先抛出核心结论，让读者带着结论去读后面的细节。适合"结论反直觉、需要先建立预期"的地方。

```
> 记忆系统的误删代价远高于冗余代价——这条不对称性，是几乎所有生产系统都选择保守策略的根本原因。

面对过期、重复、冲突三类问题，直觉上的解法是……（后续展开分析）
```

**后置总结**：一段讨论结束后，如果核心结论具有提炼性或哲理感，用 `>` 单独呈现，起到视觉停顿 + 记忆锚点的作用。

```
> 记忆不是被粗暴删除，而是像人类大脑一样，随着时间自然遗忘，并在需要时被重新强化。
```

两种用途都有效；不要把这类句子嵌在正文段落末尾——引用块的独立成行才能发挥停顿感。每 2～3 节至少出现一次，不要让全文连续 5 段都是纯散文。

### 常用句式模板

```
在 [场景] 下，[旧方案/朴素做法] 会带来 [具体问题]。
```
```
[模块/机制名] 负责 [职责]，核心逻辑是 [一句话核心]。
```
```
这里有一个 [问题/权衡]：[陈述矛盾]。[方案] 的做法是 [描述]，代价是 [取舍]。
```
```
为什么是 [具体数字]？[数字] 来自 [实验/论文/代码中的注释]，[直觉解释]。
```
```
图 [N] 展示了 [流程名]，可以看到 [关键现象/结论]。
```

## 禁用词与替换建议

| 禁用 | 替换 |
| --- | --- |
| 今天我们来聊聊 X | 直接从数据/矛盾/场景开头 |
| 非常强大 / 效果很好 | 给具体指标（准确率 +N%，延迟 -Xms） |
| 如图所示 | 图 N 展示了… / 可以看到… |
| 大家知道… | 删掉，直接说结论 |
| 值得一提的是 | 删掉，直接说 |
| 无论如何 | 删掉 |
| 相信读者已经了解 X | 给一句解释，或脚注 |
| 门槛（描述触发条件时） | 条件、前提、启动条件 |
| 门控（如"时间门控"） | 时间间隔限制、时间窗口、时间约束 |
| 信号（描述系统扫描/收集的内容时） | 直接描述实际内容：新增内容、需要修正的内容、需要更新的记忆 |

> **从英文直译的技术词汇**：中文技术写作中最常见的问题之一是把英文技术术语直接音译或意译后嵌入中文句子。"信号"（signal）、"门控"（gate control）、"门槛"（threshold）在英文原文里是精确的技术词，但直译进中文后往往语义发生偏移或读起来生硬。
>
> 判断方法：把这个词读给一个没有英文背景的中文读者，他能不能立刻理解？能 → 保留；不能 → 换成描述实际行为的中文短语。
>
> 常见替换模式：不说"找信号"，说"扫描三类来源找需要新增或修正的内容"；不说"时间门控"，直接说"距上次运行 ≥ 24h"；不说"满足四道门槛"，说"四项条件全部满足"。

## Markdown 格式规范

### 标题层级

- `#` — 文章标题（只出现一次）
- `##` — 章节标题（对应大纲的一、二、三……）
- `###` — 小节标题（可在必要时使用，但不超过 3 层）
- **不用数字前缀**（不写 `## 2.1 核心组件`），用文字标题
- **不加元描述前缀**：不写 `## 关键设计：三层语义索引`，直接写 `## 三层语义索引`。"关键设计"、"核心机制"、"重要组件"是作者的评判，不是内容本身；删掉后标题更干净，读者也不会觉得是套模板

```
❌ 反例：
## 关键设计：三层语义索引
## 核心机制：Rerank 与热度加权
## 重要组件：意图分析器

✅ 正例：
## 三层语义索引
## Rerank
## 意图分析
```

### 强调与格式

- `**加粗**` — 关键术语第一次出现、重要结论、参数名
- `` `行内代码` `` — 变量名、类名、函数名、文件路径
- `> 引用块` — 补充说明、重要警告、来自论文/README 的直接引用（需标来源）

### 表格

- 前言"数据表"放关键数字（容量、阈值、评测指标）
- 架构章节用"类比映射表"（日常概念 → 系统组件）帮助读者建立心智模型
- 对比章节用横向对比表（维度 × 方案）

**禁止"表格总览 + 加粗段落展开"双重结构**：不要先用表格列出 N 个部分，再用加粗行内标题（`**部分A**　内容……`）逐一展开同样的内容——这是在用两种形式重复同一份信息，读者会感到冗余。选一种：要么只用表格并在表格里写完，要么用正式的 `###` 子标题分节展开，不要两种混用。

```
❌ 反例（表格总览后又用加粗行内标题展开）：

这份提示词分五个部分：

| 部分 | 作用 |
|------|------|
| 任务说明 | 明确角色与权限 |
| 预注入清单 | 告知已有什么 |
| …… | …… |

**任务说明**　提取 Agent 收到的任务开头是……

**预注入清单**　提取 Agent 启动前，系统会先扫描……

✅ 正例 A（只用表格，表内写完）：

| 部分 | 作用 | 关键约束 |
|------|------|---------|
| 任务说明 | 明确角色与权限 | 写入限制在记忆目录，禁止调查代码 |
| 预注入清单 | 告知已有什么 | 文件名 + 类型 + 时间 + 描述全部附上 |

✅ 正例 B（只用子标题展开，不用表格总览）：

#### 任务说明与工具沙箱
提取 Agent 收到的任务开头……

#### 预注入记忆清单
提取 Agent 启动前……
```

**示例表格中的"跳过"行**：如果一个操作示例的表格里大多数行都是"跳过/无变化"，不要保留那些空行——只列出有实质动作的行，然后加一句话说明其余各章节跳过。

```
❌ 反例（10 行里 7 行是"跳过"）：

| 章节 | 本轮操作 | 改后内容 |
|------|---------|---------|
| 会话标题 | 跳过 | - |
| 当前状态 | 改写 | …… |
| 任务规格 | 跳过 | - |
| 文件与函数 | 跳过 | - |
| …… | …… | …… |

✅ 正例（只保留有动作的行，其余一句话带过）：

本轮只更新了三个章节，其余七章无实质变化，直接跳过：

| 章节 | 本轮操作 | 改后内容 |
|------|---------|---------|
| 当前状态 | 改写 | …… |
| 错误与纠正 | 追加 | …… |
| 工作日志 | 追加 | …… |
```

### 代码块

- Shell 命令：用 \`\`\`bash
- 源代码片段：用对应语言标注（\`\`\`python、\`\`\`go 等）
- 伪代码/简化示意：允许，但必须在代码块内第一行注释 `# 简化示意` 或 `# 伪代码`
- 画图提示：必须用代码块，第一行写 `# 画图提示：[描述]`
- **公式/图优先于代码**：概念性内容（热度权重公式、算法评分、数学推导）优先用 **LaTeX 公式**（行内 `$...$`，块级 `$$...$$`）或**画图提示**表达，不用代码块来解释工作原理；代码块只用于展示真实源码片段
- **不在正文散文中嵌入英文源码原文字符串**：说明"源码中如何描述某件事"时，用中文意译，不直接把英文字符串嵌进正文（例：不写 `（源码原文："The file has already been read for you"）`，改成"提示词里明确说明文件内容已预先读入"）；如确需展示源码原文，放进代码块

```
❌ 反例（用代码块解释热度计算逻辑）：

```python
import math
def hotness(n, t):
    freq = 1 / (1 + math.exp(-math.log(n + 1)))  # 频率分量
    decay = math.exp(-math.log(2) / 7 * t)        # 时效衰减
    return freq * decay
```

✅ 正例（用 LaTeX 公式直接表达）：

$$\text{hotness} = \frac{1}{1+e^{-\ln(n+1)}} \times e^{-\frac{\ln 2}{7} \cdot t}$$

- **频率分量**：对访问次数 $n$ 做对数 Sigmoid 压缩，值域 (0, 1)
- **时效分量**：指数衰减，$t$ 为距上次更新的天数，半衰期 7 天
```

### 代码块内注释

- 注释用中文，简短（不超过半行）
- 解释"为什么"，不是"是什么"（例如 `# 热度低于阈值则淘汰` 而非 `# 比较热度值`）

### 图片插入规范

- 每张图出现前一句话说明"它展示什么"，出现后（或同段）说明"从图中可以得出什么结论"
- 图片不当装饰用：如果一张图只是"好看"，删掉
- 画图提示代码块后，正文继续分析图的含义

## 开头钩子写法

前三句决定读者是否继续阅读。三种有效开头：

**1. 数据引导**
```
[系统名] 在 [评测基准] 上达到 [具体数字]，但更让工程师感兴趣的问题是：这个结果背后的设计是什么样的？
```

**2. 矛盾/反直觉**
```
大多数 [某类系统] 选择 [常见方案]。[系统名] 反过来：[反直觉的设计]。这个看似奇怪的选择，背后有[原因]。
```

**3. 痛点场景**
```
[具体场景]：[问题描述]。这是 [系统类型] 最难处理的一类情况。[系统名] 给出了一个[形容词]的答案。
```

**4. 专家受众：亮点先行**

当目标读者是技术专家（R3），对这类系统的问题背景已经熟悉时，跳过痛点铺垫，直接抛出"这个系统做了什么反直觉的设计"：

```
[系统名] 在 [同类问题] 上有几个反直觉的选择：[亮点 A]、[亮点 B]、[亮点 C]。本文拆解这些设计背后的取舍。
```

或者用亮点无序列表替代文字开头，让读者扫一眼就能判断是否值得精读。

```
❌ 反例（对专家读者仍铺垫问题背景）：

AI Agent 的记忆问题一直是工程难题——对话变长时，把所有历史塞进上下文
会导致 token 暴涨；而朴素的向量检索又容易召回不精准……
（← 专家读者已经知道这些，不需要再解释）

✅ 正例（直接抛出设计亮点）：

OpenViking 有几个反直觉的设计：
- 不用向量库做扁平检索，而是维护有层次的目录摘要（L0/L1/L2）
- 写入前让 LLM 先读现有记忆再决策，而不是一次性覆盖
- 热度加权默认关闭（alpha=0.0），纯语义分也能跑，开启后才混入频率信号
本文拆解这些选择背后的取舍。
```

**避免**：
- "今天来聊聊 X，它是一个…"
- "随着 AI 的发展，[领域] 越来越重要"
- "本文将介绍…"

## 收尾写法

**总结节**（每篇必须有）结构：

1. 用 2~4 条要点列核心结论（可以是设计要点、权衡取舍、适用场景）
2. 一句话说"这个系统在什么场景下值得关注/使用/借鉴"
3. （可选）指向下一个话题或延伸阅读

**不要**：
- 重复前面说过的所有内容
- 写"通过本文的介绍，相信读者已经…"
- 写"欢迎点赞关注"等发布类收尾（此 Skill 面向技术文档，非公众号运营）

---

## 节奏控制（防疲劳）

- 每隔 3~5 个段落必须出现"结构变化"：代码块 / 表格 / 画图提示 / 引用块
- 同类型块（纯文字、纯代码）不连续超过 3 个
- 每章 500~900 字为宜；超过 1000 字考虑拆节或增加图表消化密度
