## Hard Gates（必须满足，开写前逐条确认）

1. **选模式**：开写前必须确认文章属于哪种模式（见下方"三种文章模式"）

2. **读风格**：写作前必须读 `references/style-guide.md`

3. **先研究后写作**：至少读过源代码核心文件或论文摘要，记录实际数字（容量、阈值、评测结果）

4. **先问题后方案**：每个核心机制先说"为什么需要"，再说"怎么实现"

5. **画图提示规范**：
   - 必须放在代码块内，第一行写 `# 画图提示：`
   - 背景统一白色
   - 不出现函数名和英文技术术语作为图中可见标签
   - **禁止使用纵向泳道图**
   - **禁止在 prompt 中写 px、尺寸、颜色值等渲染参数**（会限制模型发挥）

6. **可复现**：至少 2 个可复制的代码块，配解释说明

7. **代码真实性验证**：文中所有数字（容量、阈值、超参、评测结果）必须对照源码或论文原文核实。发现文档与源码不一致时，以源码为准并在文章中注明。禁止直接引用 README 中未经核实的数字。

   > **反例**：某文档说权重是 `0.20/0.25/0.15/0.15/0.10/0.15`，实际源码是 `0.24/0.30/0.15/0.15/0.10/0.06`。必须核实后再写。

8. **内容详细度**：每个核心机制必须覆盖三层：
   - **是什么**：用一句话或类比说清楚这个机制/组件的定义
   - **为什么**：说清楚不这样做会有什么问题（先讲痛点）
   - **怎么做**：给出实现细节，包括关键数字、数据结构、代码片段

   > **反例**："系统使用混合搜索提高召回率。" ← 缺少为什么和怎么做  
   > **正例**："单路向量搜索的问题是专有名词（如项目代号、人名）在向量空间中容易被稀释，召回不全。混合搜索的解法是向量得分和 BM25 得分加权求和：`score = vectorWeight * vectorScore + textWeight * textScore`，两路并行执行后合并，专有名词命中率显著提升。"

9. **前言结构遵循金字塔原理（CQRD）**：前言不是流水账，必须按以下顺序组织：
   - **C – Conclusion（结论先行）**：第一段就说清楚这个系统的核心价值，用数据表格锚定（准确率 X%、延迟 Xms、容量 X 条）
   - **Q – Question（提出矛盾）**：说清楚核心技术挑战是什么——为什么这个问题难解决？
   - **R – Resolution（解法概述）**：用 2~3 条 bullet 说清楚系统的解题思路，各对应一个核心机制
   - **D – Details（深度预告）**：一句话点出本文要深入拆解哪些部分，帮读者决定是否继续阅读

   > **反例**（流水账）："本系统是一个 AI 记忆层，支持向量搜索和全文搜索，还有梦境功能……"  
   > **正例**（CQRD）：
   > "MemoryCore 在混合搜索评测上准确率达 78.7%，检索 p95 延迟 1.44s。
   >
   > | 维度 | MemoryCore | 全量上下文 | 提升 |
   > |---|---|---|---|
   > | 准确率 | 78.7% | 52.9% | +48.8% |
   > | 检索延迟 | 1.44s | 17.12s | 11.9× 更快 |
   >
   > 两个核心难题是：写入侧，原始对话不能直接存——充满噪声和重复；检索侧，单路向量搜索对专有名词召回不全。MemoryCore 的解法是用 LLM 提炼结构化事实管理记忆生命周期，加三路混合搜索叠加多样性重排。本文将拆解同步管道的原子重建机制、混合搜索的权重设计，以及梦境系统如何把短期记忆晋升为长期记忆。"

## 工作流程（四阶段）

### 阶段 1：定位与选模

**执行**：
1. 确认文章主题和目标读者（想快速上手？想深度理解？想做技术选型？）
2. 选择文章模式：
   - **Mode A**：工具/框架快速入门 → 侧重"是什么 + 怎么用 + 和同类比有什么不同"
   - **Mode B**：架构源码深挖 → 侧重"为什么这样设计 + 关键机制 + 源码实现"
   - **Mode C**：版本/方案对比 → 侧重"设计哲学差异 + 权衡取舍 + 适用场景"
3. 确认有哪些可读的素材（源码目录、论文、README、分析笔记）
4. 生成文章骨架草稿，用户确认后进入下一阶段

### 阶段 2：研究与骨架

**执行**：
1. 阅读源代码/文档，记录：
   - 关键数字（容量上限、阈值、超参、评测结果）
   - 核心数据结构的字段
   - 主要函数的调用关系
2. **与文档/README 交叉核实**：发现不一致，以源码为准，在文章中注明
3. 确定 2~3 个最值得深讲的核心设计（最独特或最容易被误解的部分）
4. 细化大纲（每章节 3~5 个要点），标注哪些地方需要画图、哪些需要代码块
5. 如需配图，先列出"图清单"（每张图展示什么、用哪种图类型）

### 阶段 3：逐节写作

**原则**：
- 一次只写一个章节，写完后暂停等待用户反馈
- 每节完成后自检：是否覆盖了「是什么/为什么/怎么做」三层？是否有具体数字和代码？

**Hook 优化（开头特别关注）**：
- 前言必须按 CQRD 结构：数据表 → 核心矛盾 → 解法 bullet → 深度预告
- 禁用套话开头（"今天我们来聊聊 X"、"随着 AI 的发展……"）

### 阶段 4：定稿

自检清单：
- [ ] 前言有数据表、核心矛盾、解法 bullet，符合 CQRD 结构
- [ ] 每个重要机制都覆盖了「是什么/为什么/怎么做」三层
- [ ] 每个关键参数（容量、阈值）都已与源码核实，数字正确
- [ ] 画图提示在代码块内，无 px/尺寸/颜色值，无泳道图，无函数名
- [ ] 代码块内注释是中文，简短
- [ ] 表格、代码块、散文三者交替，不连续超过 3 个同类型块

## 三种文章模式

### Mode A：工具/框架快速入门

**骨架**：
```
# 一、前言（数据表 + CQRD 结构）
# 二、它是什么（定义 + 核心价值 + 类比）
# 三、快速上手（安装 + 最小可跑示例 + 关键 API）
# 四、核心能力（分节讲 3~5 个主要功能，每个配「是什么/为什么/怎么做」三层）
# 五、和同类工具对比（对比表格 + 适用场景结论）
# 六、总结
```

### Mode B：架构源码深挖

**骨架**：
```
# 一、前言（数据表 + CQRD 结构：结论先行 → 核心矛盾 → 解法概述 → 深度预告）
# 二、整体架构（类比映射表 + 架构图 + 核心组件职责 + 代码结构）
# 三、[主干流程 A]（是什么 → 为什么这样设计 → 怎么实现，逐层拆解）
# 四、[主干流程 B]（同上）
# 五、关键设计：[最独特机制]（公式/数据结构 + 直觉解释 + 代码 + 权衡取舍）
# 六、[可选] 特殊子系统深讲
# 七、评测体系（如有论文或 benchmark）
# 八、总结
```

**说明**：主干流程 A/B 根据系统特点命名。数据库系统：写入/查询；编译器：解析/代码生成；Agent 系统：规划/执行。

### Mode C：版本/方案对比

**骨架**：
```
# 一、前言（对比背景 + CQRD 结构）
# 二、问题根源（旧方案为什么不够好，是什么/为什么/怎么暴露问题）
# 三、[方案/版本 A]（核心设计 + 优势 + 局限）
# 四、[方案/版本 B]（核心设计 + 优势 + 局限）
# 五、关键差异深挖（选 2~3 个维度做横向对比，用对比图）
# 六、适用场景与选型建议
# 七、总结
```

## 画图提示规范（摘要）

> 完整规范和 6 个实战示例见 `references/diagram-style.md`

**三条强制规则**：
1. 节点标签全部用中文（`ShortTermMemory` → 「短期记忆」）
2. 不出现函数名、类名、字段名、英文缩写
3. **prompt 中不写 px、尺寸、颜色值、字号**——这些渲染细节写了反而限制模型，越少越好

**图类型快速对照**（完整版含参考图见 `diagram-style.md`）：

| 场景 | 图类型 |
| --- | --- |
| 多层系统架构总览 | 横向多层架构图 |
| 有阶段划分的处理流程 | 竖向三阶段流程图 |
| 多维度权重/评分 | 横向条形图 + 说明卡片 |
| 公式 + 分级条件 | 左右分栏（公式+卡片） |
| 两方案对比 | 并排对比图 |
| 多阶段内各有子流程 | 横向阶段列流程图 |
| 树状结构搜索/遍历 | 树形层次检索路径图 |
| 跨模块流程（原泳道图） | **竖向三阶段 or 横向阶段列（禁用泳道图）** |

## 禁止事项

- 禁止：跳过大纲阶段直接写正文
- 禁止：一口气写完全文再给用户看
- 禁止：用"今天我们来聊聊 X"等套话开头
- 禁止：前言不符合 CQRD 结构（没有数据表、没有核心矛盾）
- 禁止：核心机制只说"是什么"，不解释"为什么"和"怎么做"
- 禁止：文章数字未与源码核实
- 禁止：画图 prompt 中写 px、尺寸、颜色值、字号等渲染细节
- 禁止：使用纵向泳道图
- 禁止：图中出现函数名或英文字段名
