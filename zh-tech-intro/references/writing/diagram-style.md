# 画图风格参考指南

> 生图工具：**gpt-image-2**（经实践验证）

---

## 一、核心原则

画图提示是给 AI 生图工具的 **prompt**，核心是描述「展示什么信息、用什么结构呈现」。

### 绝对禁止写入 prompt 的内容

| 禁止 | 原因 |
| --- | --- |
| CSS 颜色值（`#3B82F6`、`rgba(59,130,246,0.5)`） | 过度限制模型发挥，生成结果反而变差 |
| 像素/尺寸（`800px`、`1200×600`、`14pt`、`2px`） | 生图模型不接受绝对尺寸，写了等于没写 |
| 字号/字重（`bold`、`font-size: 16`） | 同上，渲染细节由模型决定 |
| SVG/HTML/CSS 代码片段 | 不是代码生成任务 |
| 箭头粗细、边框样式等渲染细节 | 描述内容和结构，不描述样式参数 |

### 应该写的内容

- **图类型**：横向多层架构图、竖向三阶段流程图、条形图+卡片……
- **节点标签**：全部用中文，描述语义（「同步决策」而非 `SyncDecision`）
- **箭头语义**：说明关系（「降级路由」「触发迁移」），不说方向
- **关键数据**：阈值、比例、数字直接写进来（「得分 0.62」「权重 30%」）
- **视觉层次**：「重点突出」「虚线框表示可选」「深色背景强调」

---

## 二、【禁用】泳道图

**不要使用纵向泳道图（Vertical Swimlane）。**

gpt-image-1 实践验证：纵向泳道图 prompt 生成结果极差——泳道线混乱、节点溢出边界、文字截断、整体像线框图而非正式图表。

**替代方案**：

| 原泳道图场景 | 替换为 |
| --- | --- |
| 跨模块的完整流程（跨角色） | 竖向三阶段流程图（§3.2） |
| 数据在多层/多组件间流转 | 横向多层架构图（§3.1）或横向阶段列流程图（§3.7） |
| 强调「谁做什么」的分工 | 横向阶段列流程图（§3.7），顶部加阶段标签 |

## 三、图类型参考模板

### 3.1 横向多层架构图

适用：展示系统分层结构、各层组件和整体数据流。  
参考图：
- `assets/01-architecture.png`（MemoryCore 四层架构）
- `assets/14-2-layered-architecture.jpg`（LLM-Wiki 四层架构，**重点参考**：左侧彩色层名徽章、节点图标+副标题、层间单箭头标语义）

**三区布局规范**（每行一层，左中右三区）：

| 区域 | 内容 | 视觉要求 |
| --- | --- | --- |
| 左区（层名徽章） | 层名文字 | 带颜色背景的圆角矩形，颜色比该行浅色背景深一档 |
| 中区（节点组） | 横向并排节点，每节点：小图标 + 加粗名称 + 灰色副标（1~2行） | 圆角矩形，浅色背景行 |
| 右区（角色描述） | 一句话说这层"是什么角色" | 灰色小字，垂直居中 |

**属性卡片 vs 流程节点**（最容易混淆，必须看清楚再写）：

| 情形 | 节点含义 | 写法 | 箭头 |
| --- | --- | --- | --- |
| 展示该层的**特征/属性** | 并排的属性描述（如：原始消息 / 顺序读取 / 会话级） | 「横向并排三张属性卡片，**卡片之间不用箭头连接**」 | ❌ 无 |
| 展示该层内部的**处理步骤** | 有先后顺序的操作（如：解析→提炼→落库） | 「三个步骤节点，箭头相连」 | ✅ 有 |

错误写法「内容三个节点串联」容易让生图工具在属性列表之间画出箭头，整层变成流程图而非存储架构图。

**层外数据流规范**：

架构图中层与层之间的数据流箭头（如「实时写入」「异步提炼」「混合检索」）必须明确声明在色带**外侧**，不能穿插在层内卡片之间。格式：

```
# 色带外部的数据流（均在色带轮廓外侧，不穿插在节点之间）：
#   顶部：「[来源]」→ 上方色带，标注「[操作]」
#   两层之间：居中垂直箭头，标注「[操作]」
#   右侧：[侧边线说明]
```

**层间箭头规范**：
- 每两层之间只画**一条**居中向下箭头，**不要扇形展开**连到每个节点
- 箭头标注具体操作语义（「调用」「读取 & 写入」「策略查询」），不写"数据流"这类泛称
- 箭头标签下方加白色矩形衬底，防止与背景混叠

```
# 画图提示：横向多层架构总览图，展示[系统名]的[N]层结构与数据流。
#
# 整体布局：[N] 行（每行一层）从上到下排列，每行分三区：
#   左区：层名徽章，带颜色背景的圆角矩形，颜色比该行背景深一档，层名居中
#   中区：节点横向并排，每节点由小图标 + 加粗名称 + 灰色副标组成
#   右区：一句角色定位描述，灰色小字
#
# 行 1【[层名]】（[浅色]背景行，左徽章[深色]）：
#   [N] 个节点，各带图标：
#     「[节点A]」  [副标题]
#     「[节点B]」  [副标题]
#   右区：[一句角色描述]
#
# 行 2【[层名]】（[浅色]背景行，左徽章[深色]）：
#   [N] 个节点，副标题为串联步骤：
#     「[节点C]」  [步骤1] → [步骤2] → [步骤3]
#   右区：[一句角色描述]
#
# 层间箭头（每层之间一条，垂直向下居中，不展开连到各节点，标签有白色衬底）：
#   [层名A] → [层名B]：「[操作语义]」
#   [层名B] → [层名C]：「[操作语义]」
#
# 风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。
```

### 3.2 竖向三阶段串行流程图

适用：有清晰阶段划分的处理流程，替代泳道图。  
参考图：
- `assets/02-dreaming-flow.png`（梦境三阶段）
- `assets/14-3-ingest-flow.jpg`（Ingest 摄入流程，参考：左侧主流程 + 右侧信息卡、步骤编号徽章、决策菱形）
- `assets/14-4-query-flow.jpg`（Query 查询流程，参考：多步骤带子分支、右侧补充说明卡）

**标题栏位置：顶部横条（不是左侧竖条）**

每个阶段框的标题栏必须是**顶部横贯全宽的横条**，不要用左侧竖条。原因：左侧竖条挤占水平空间，步骤节点区域变窄，图整体显得拥挤。

```
# ✅ 正确：顶部横条
#   顶部深蓝横条标题栏（横贯矩形全宽），白色文字「[名称] · [副标题]」，左侧有图标；
#   标题条下方主体区域：…

# ❌ 错误：左侧竖条
#   左侧深蓝色竖条标题栏，白色文字「[名称] · [副标题]」；
```

**异步阶段标准表示**：

当某个阶段是**异步执行**（不阻塞主流程），需同时给出三个视觉提示：
1. 阶段框外轮廓改为**虚线**（橙色虚线）
2. 背景色选**浅黄**（与同步阶段的蓝/绿色系形成区分）
3. 标题栏**右端**加「**异步**」白色小胶囊标签

```
#   阶段二【[名称]】（浅黄背景，整行宽圆角矩形，整体外框为橙色虚线）：
#     顶部深橙横条标题栏（横贯矩形全宽）「[名称] · [副标题]」，标题栏右端有「异步」白色小胶囊标签；
#     标题条下方主体区域：…
#     主体最右侧有一行灰色小字「不阻塞主流程」
```

**步骤节点布局**：同一阶段内的多个步骤节点必须等宽、均匀分布，不要挤在一侧留大片空白。

```
# 画图提示：竖向三阶段流程图，展示[流程名]
#
# 顶部触发器：「[触发条件]」→ 判断菱形「[已完成检查]？」→ 是则结束，否则继续
#
# 三个串联阶段框（圆角矩形，从上到下）：
#   阶段一【[名称]】（浅蓝背景，整行宽）：
#     顶部深蓝横条标题栏（横贯全宽），左侧图标，白色文字「[名称] · [副标题]」
#     标题条下方：横向排列 N 个白色圆角步骤节点，等宽均匀分布，箭头相连：
#       「[步骤1]」→「[步骤2]」→「[步骤3]」
#   阶段二【[名称]】（浅黄背景，整行宽，整体外框橙色虚线）：
#     顶部深橙横条标题栏（横贯全宽）「[名称] · [副标题]」，右端「异步」白色小胶囊标签
#     标题条下方：横向排列 N 个步骤节点，等宽均匀分布，箭头相连
#     主体最右侧灰色小字：「不阻塞主流程」
#   阶段三【[名称]】（浅绿背景，整行宽）：
#     顶部深绿横条标题栏（横贯全宽），左侧图标，白色文字「[名称] · [副标题]」
#     标题条下方：左侧菱形「[判断节点]」，右侧引出 N 条等间距纵向分布的水平箭头，
#       各指向 N 张等高圆角卡片（纵向均匀排列）
# 阶段间：每两个阶段之间一条居中实线向下箭头
#
# 风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。
```

### 3.6 竖向串行流程图（含判断分支）

适用：单一路径的处理步骤，含条件判断分支。

**⚠️ 多旁路汇合时的四条强制规则**（经实践验证，缺失任一条都会出现连线错误）：

1. **旁路终点必须显式连入汇合节点**：每条旁路的最后一个节点，必须在 prompt 中明确写「→ 向下连入「[汇合节点名]」」，不能依赖模型自行推断。
2. **所有箭头居中进出**：在 prompt 中加一句「所有箭头从节点中央水平或垂直引出，进入目标节点中央」。
3. **多旁路纵向空间均匀分配**：有 N 条旁路时，在 prompt 中说明「N 条路径在图中纵向空间均匀分配，不要相互紧贴」。
4. **颜色一致性**：在 prompt 中加「所有节点配色保持同一色调体系，由模型自行决定，不要有的节点深色调有的浅色调」。

**⚠️ 禁止自动编号**：流程节点不加流水序号（①②③），除非明确要求。

```
# 画图提示：竖向流程图，展示[流程名]的处理步骤与分支逻辑。
#
# 从上到下：
#   「[入口节点]」（椭圆）
#     ↓
#   「[判断条件A]」（菱形）
#     是 →→ 右侧旁路：「[旁路A节点1]」→「[旁路A节点2]」→ 向下连入「[汇合节点]」
#     否 ↓
#   「[判断条件B]」（菱形）
#     是 →→ 右侧旁路：「[旁路B节点1]」→「[旁路B节点2]」→ 向下连入「[汇合节点]」
#     否 ↓
#   「[主路径节点组]」（主路径向下）→ 向下连入「[汇合节点]」
#     ↓
#   「[汇合节点]」（三条路径在此汇合）
#
# 布局要求：
#   · 三条路径（旁路A / 旁路B / 主路径）纵向空间均匀分配，不要相互紧贴
#   · 所有箭头从节点中央引出、进入目标节点中央
#   · 所有节点配色保持同一色调体系，由模型自行决定
#
# 风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。
```

### 3.7 横向阶段列流程图

适用：多个处理阶段，每个阶段内有纵向子流程，阶段间横向推进。  
参考图：`assets/05-write-chain.png`（OpenViking 写入完整链路）

**⚠️ 列间连接是最容易混乱的地方**，必须在 prompt 中逐一声明列间连线，不能只描述列内节点。  
核心规则：
- **每列的出口节点必须明确点名**连接到下一列的哪个入口节点。
- **列内分支终止时**（如某条件判断为"跳过"的终止路径），必须标注「此路径止于本列，不跨列」。
- **箭头颜色不要多于两种**，颜色多了模型会把线画乱；如需区分异步/同步，只用实线/虚线区分即可，不加颜色。

```
# 画图提示：横向阶段列流程图，展示[系统名]的完整[流程名]。
#
# 顶部：[N]个阶段标题列，编号 ①②③④，每列一个深色背景标题：
#   ① 「[阶段1名]」
#   ② 「[阶段2名]」
#   … 依此类推
#
# 各列纵向节点（圆角矩形，从上到下）：
#   第①列（从上到下）：
#     「[节点A]」
#     菱形「[判断条件]？」
#       是 → 「[终止节点]」（此路径止于第①列，不跨列）
#       否 ↓ 继续
#   第②列（从上到下）：
#     「[节点B]」
#     「[节点C]」
#   第③列（从上到下）：
#     「[节点D]」
#     「[节点E]」
#   第④列（从上到下）：
#     「[节点F]」
#     「[节点G]」
#
# 列间连接（必须逐一声明）：
#   第①列底部「[节点名]」 → 第②列顶部「[节点名]」（水平箭头）
#   第②列底部「[节点名]」 → 第③列顶部「[节点名]」（水平箭头）
#   第③列底部「[节点名]」 → 第④列顶部「[节点名]」（水平箭头）
#
# 箭头：统一使用实线箭头，异步流程用虚线，不超过两种线型
# 底部说明框（可选）：「[整体策略总结一句话]」（重点标注）
#
# 风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。
```

### 3.8 树形层次检索路径图

适用：展示树状结构的遍历/搜索路径，标注每个节点的分数或状态。  
参考图：`assets/04-tree-search.png`（层次检索路径图）

```
# 画图提示：树形层次检索路径图，展示[查询/搜索]在[树状结构]中的遍历路径与收敛过程。
#
# 外层大框：「[空间名称]（[结构描述]）」
# 树结构（从上到下）：
#   根节点（L0，蓝色高亮）：「[根节点名]（L0，分数 [X]）」
#   L1 子节点（横向排列，4个）：
#     「[目录A]（L1，分数 [低]）」→ 虚线框标「低于阈值，整个子树跳过」
#     「[目录B]（L1，分数 [高]）」→ 继续向下
```

### 3.10 金字塔分层图

适用：展示层次递进关系，底层最基础/最宽，顶层最高级/最窄（如优先级金字塔、处理链路分层）。

**⚠️ 金字塔的四条强制规则**（经实践验证，违反任一条都会导致图形扭曲或层次混乱）：

1. **层序方向**：底层 = 第一层（最基础）= 最宽；顶层 = 最高层（最高级）= 最窄的塔尖。**编号从下到上递增**。模型默认容易画反，必须在 prompt 中明确标注「底部最宽为第一层，顶部尖顶为最高层」。
2. **角度必须一致**：三条规则—左侧斜边从底层左角到顶点是一条连续直线；右侧斜边从底层右角到顶点是一条连续直线；所有层的左右两侧均共用同一条斜边，不单独画。违反此规则会导致各层角度不一致，视觉极丑。
3. **必须有塔尖**：顶层要收成三角形的真实尖顶（apex），不是梯形。
4. **不加标题**：金字塔本身不加图表标题，除非用户明确要求。


### 3.13 两层流程图：主流程 + 展开细节

适用：一个步骤本身有几个子步骤，单独画一张图太单薄，但全塞进主流程一行又太挤——典型场景是"5步主流程，其中第4步还能再拆成4个子步骤"。比单独画两张稀疏的图density更高，比硬塞成一行8+卡片更不挤。

**核心规范**：
- 上层是主流程，横向卡片，其中一张"被展开"的卡片用更深的颜色/边框突出，小字写「展开见下方」
- 该卡片正下方引一条竖直箭头连到下层
- 下层是子步骤展开，用浅灰虚线边框的矩形框整体框住，左上角标小字说明这是哪一步的细节，框内卡片可以比上层卡片略小、颜色用同色系的浅一档变体
- 整张图依然要显式声明「横向宽屏构图，宽高比约16:9」（见踩坑14），否则上下两层会被压成竖版导致两层都拥挤

```
# 画图提示：横向宽屏构图，宽高比约16:9，禁止竖版，不要任何大标题文字，背景纯白。纵向两层流程图，风格为高级技术PPT配图，留白宽松不拥挤。
#
# 上层主流程：[N]个带图标的[颜色]圆角卡片从左到右水平排列，卡片间距宽松，用实线箭头连接。

#   卡片[N]（[图标]）「[末步]」，小字「[支撑数据]」
```

## 七、常见踩坑教训（经实践验证汇总）

以下教训均来自实际生图迭代，按问题类型归纳，写 prompt 前必读。

### 踩坑 1：旁路节点没连到汇合节点（竖向流程图）

**现象**：旁路终止节点悬空在右侧，没有连线到底部的汇合节点（如「向量库」），三条路径没有真正汇合。  
**根因**：prompt 只描述了旁路节点的内容，没有明确说它最终连向哪里。  
**修复**：旁路的每个终止节点，必须写「→ 向下连入「[汇合节点名]」」，不能假设模型会自动推断所有路径汇合到同一终点。

### 踩坑 2：横向阶段列流程图的列间连线完全混乱

**现象**：多列阶段图中，列与列之间的箭头乱连，有的从中间节点连到错误的列，整体线条无法理解。  
**根因**：prompt 只描述了每列的内部节点，没有声明列间如何连接，模型凭推断随机连线。  
**修复**：必须在 prompt 中用「**列间连接**」独立段落，逐一写出「第①列底部「X」→ 第②列顶部「Y」」。列内有分支且终止的，要明确标「此路径止于本列，不跨列」。箭头颜色种类不超过两种（多颜色会加剧混乱）。

### 踩坑 3：金字塔层序画反

**现象**：prompt 写「第一层：最高级操作」，模型把第一层画在最底部（最宽处），内容的高低语义与视觉位置倒置。  
**根因**：层序编号与层级含义不一致，模型按编号把「第一层」画在最下面。  
**修复**：金字塔层序必须「底层 = 第一层 = 最基础 = 最宽」，从下到上编号递增。在 prompt 中明确写「底部最宽为第一层，顶部尖顶为第 N 层」。

### 踩坑 4：金字塔各层斜边角度不一致

**现象**：三层各自独立画了斜边，角度各不相同，看起来像三个梯形堆叠而非一个完整三角形，视觉极丑。  
**根因**：prompt 没有约束三层共用同一对斜边，模型给每层独立画了斜边。  
**修复**：在 prompt 中加「左侧斜边和右侧斜边是贯通全图的两条连续直线，从底层两角延伸至顶部尖顶，各层均共用此斜边，不单独画」。

### 踩坑 5：金字塔带了不必要的大标题

**现象**：生成图上方出现醒目的大标题文字，占用大量空间，图显得臃肿。  
**根因**：prompt 里写了「标题：XXX」，模型把它渲染成了图表大标题。  
**修复**：金字塔图默认不加标题，prompt 中不写「标题：」字段。如需图表说明，放在文章正文中。

### 踩坑 6：流程节点被自动加上流水编号

**现象**：竖向流程图中每个节点前出现「1.」「2.」「3.」等序号，视觉噪声大。  
**根因**：prompt 用了有序列表格式描述节点，模型把序号当成节点标签的一部分渲染出来。  
**修复**：节点描述不加编号，除非明确需要。用「→」连接表示顺序，而不是「1. 2. 3.」。

### 踩坑 7：单个节点内容过多导致一侧留白严重

**现象**：某一节点因描述内容多宽度撑大，左侧大面积留白，整体比例严重失衡。  
**根因**：把节点名 + 子分类枚举混在同一个节点里，节点宽度撑大后左侧无内容区域显得空旷。  
**修复**：把节点内的子分类内容提炼为行内简短格式（如「专有名词 / 复合名词 / 引用文本 / 基础名词」一行平铺），或者拆成单独的说明行放在节点内，不要和节点名混排导致宽度失控。

### 踩坑 8：循环终止条件混入输出类型菱形

**现象**：描述一个有最大轮次限制的循环时，把「超过最大轮次」作为第三条分支塞进了「本轮输出属于哪一类？」的决策菱形，导致流程图逻辑错误——读者以为 LLM 会输出三种东西，但实际上 LLM 只能输出「工具调用」或「最终结果」两种，「超过最大轮次」是循环计数器的状态，根本不是 LLM 的输出类型。

**根因**：菱形节点的语义是「对同一事件的不同分支」。循环计数器超限是**循环守门条件**，与 LLM 输出类型属于两件完全不同的事，不该放进同一个菱形。

**修复**：循环终止条件单独提前为循环入口的第一个菱形，在调用 LLM 之前先判断「是否超过最大轮次」；通过后再调用 LLM，后面的分支菱形只负责判断输出类型。两个菱形的职责分开，各自只有两条出路。

```
❌ 反例（循环终止条件混入输出类型菱形，菱形三出口）：

进入循环
  ↓
LLM 调用
  ↓
「本轮输出属于哪一类？」（菱形）
  ├── 工具调用 → 执行工具 → 循环
  ├── 最终结果 → 返回结果
  └── 超过最大轮次 → 返回空列表   ← ❌ 不是 LLM 的输出类型

✅ 正例（两个串联菱形，各司其职）：

进入本轮迭代
  ↓
「是否超过最大轮次？」（菱形）← 循环守门，在调用 LLM 之前
  ├── 是 → 返回空列表（终止）
  └── 否 → 调用 LLM
        ↓
       「本轮输出属于哪一类？」（菱形）← 只判断 LLM 输出，只有两条出路
         ├── 工具调用 → 执行工具 → 循环回入口
         └── 最终结果 → 返回结果（终止）
```

### 踩坑 9：层间箭头扇形展开，连到每个节点（横向多层架构图）

**现象**：从上层底部画出 N 条箭头分别连到下层的每个节点，形成扇形，图面混乱，视觉噪声极大。  
**根因**：prompt 写了"层间连接"但没有明确"只画一条"，模型默认逐节点连线。  
**修复**：在 prompt 中明确写「每层之间只画一条箭头，垂直向下居中，**不要展开连到各个节点**」。

### 踩坑 10：层名标签用纯灰色文字，视觉上与行内容分不清

**现象**：层名标签和正文内容融为一体，读者难以快速定位当前看的是哪一层。  
**根因**：prompt 写"灰色文字"但没有要求徽章背景，模型没有给层名加视觉区分。  
**修复**：在 prompt 中写「层名徽章带颜色背景的圆角矩形，颜色比该行浅色背景深一档」，让层名和内容形成明显的视觉层级差。

### 踩坑 11：存储层内节点用箭头串联，整层变成流程图

**现象**：存储架构图里某一层的多个属性节点（如"原始消息 / 顺序读取 / 会话级"）被 `→` 箭头串联，视觉上读起来像处理步骤序列，而非该层的并列特征描述。  
**根因**：prompt 写了「内容三个节点串联」，模型把"串联"字面理解为有先后关系的流程，自动加上了箭头。  
**修复**：区分两类写法——
- 展示**属性/特征**（无顺序关系）：「横向并排三张属性卡片，卡片之间**不用箭头连接**」
- 展示**处理步骤**（有先后关系）：「三个步骤节点，箭头相连」

两类绝对不能混用。存储架构图的层内节点几乎全是属性，不要写"串联"。

### 踩坑 13：只画极简骨架节点，图显得单薄单调

**现象**：流程图节点只写一个动作短语（如「写入基础数据库」），不带任何支撑数据，多张图放在一起几乎一个样子——读者光看图形结构记不住任何具体信息，图沦为"过场插画"而非"信息载体"。

**根因**：写 prompt 时把"画图提示"当成了"章节标题大纲"，只搬了小节标题或步骤名，没有把正文里已经核实过的关键数字、对比、设计取舍一并放进节点里。

**修复**：画图前检查对应正文段落，把以下三类内容尽量塞进节点的副标题/小字里（节点主标题仍保持简短）：
- **关键数字**：阈值、权重、上限（「截断到 limit 条」不如「截断到 limit 条（默认 10）」）
- **对比/取舍**：两个分支或两个版本的差异点（「会话碎片初始分 0.58，日记碎片 0.62」）
- **一句话设计动机**：为什么这步存在（「截断去重，避免同语义碎片重复占位」）

同一篇文章里多张流程图，如果正文对每一步都有具体说明，图也应该体现出"信息密度递增"——内容越具体的小节，配图的节点副标题也应该越具体，不能因为追求"简洁"而把图削减成和正文脱节的骨架图。

**反向陷阱：堆细节导致关系模糊**。加密度不是把某几个节点（比如文件）做成视觉主角、把其余环节降级成它们头上的浮动注释文字——这样图会显得"信息有了，但关系没画出来"。补充密度时优先保证：①每个环节（不管是文件还是处理步骤）视觉权重相同，都是独立节点；②节点间用清晰箭头依次连接，一条主线不分叉；③单个节点内文字控制在一行核心信息，不堆砌多行细节。先有清楚的链条骨架，再考虑要不要在节点副标题里加数字——顺序不能反。

### 踩坑 14：横向多卡片图默认生成竖版画布，导致挤压拥挤

**现象**：prompt 描述的是"5个卡片横向排列"，但 gpt-image-2 默认按竖版（如 1024×1536）出图，5 张卡片被压缩进一个窄画布的横向带里，卡片间距被挤没，文字拥挤——肉眼看是"图画对了，但很挤"，容易被误判成"卡片数量太多需要精简"，实际是画布方向不对。

**根因**：prompt 只描述了内容排列方式（"横向排列"），没有显式声明整张画布的宽高比，模型按自己的默认偏好（竖版）出图，横向内容被迫塞进竖版空间。

**修复**：横向多卡片/多列布局，prompt 开头必须显式加一句画布方向声明：「横向宽屏构图，宽高比约 16:9，禁止竖版」。这一句和"卡片之间留白宽松不拥挤"要同时写——前者保证画布形状对，后者保证有这个形状之后模型不会把多余空间留成大片空白而不是分给卡片间距。

### 踩坑 15：未声明背景色，模型给整图刷了大片强调色背景

**现象**：prompt 里给某个步骤起了"浅紫色卡片"之类的强调色，结果模型把强调色直接铺成了整张图的背景色，原本该是白底的图变成了大片紫色/蓝色背景，与同系列其他图风格不统一。

**根因**：prompt 描述了"该用什么颜色"，但没有同时声明"这个颜色只能用在哪里、不能用在哪里"，模型在颜色取舍上选择了影响范围最大的"刷背景"。

**修复**：使用强调色时，在 prompt 里同时写两句——画布层「整体背景必须是纯白色，不要任何大面积紫色/彩色背景色块」，节点层「[颜色]只用在图标线条和强调色上」。两句缺一不可，只写后者模型仍可能在卡片外的空白处刷色。

### 踩坑 12：数据流箭头穿插在层内节点之间（横向多层/两层架构图）

**现象**：架构图中「实时写入」「异步提炼」「混合检索」等数据流箭头被画在色带内部，穿过属性卡片之间的间隙，视觉上和属性节点混成一团，读者分不清哪些是层的内容、哪些是数据流向。  
**根因**：prompt 没有明确说数据流箭头画在色带外侧，模型在最近的空间里把箭头就近插入了层内。  
**修复**：在 prompt 中加独立段落，明确写：

```
# 色带外部的数据流（均在色带轮廓外侧，不穿插在节点之间）：
#   顶部：「[来源]」→ 上方色带顶部，标注「[操作]」
#   两层之间：居中垂直箭头，标注「[操作]」
#   右侧：侧边线同时触碰两层，末端标注「[操作]」
```
