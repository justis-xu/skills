# 画图风格参考指南

> 生图工具：**gpt-image-2**（经实践验证）

## 零、风格关键词触发

**写画图提示之前，先检查用户是否提到了风格关键词。** 匹配到关键词时，在 prompt 开头自动插入对应的风格模板片段，**同时**按下方规则决定是否传 `--ref`。

| 关键词 | 触发风格 | 模板位置 |
| --- | --- | --- |
| 手绘、插画、手绘风格、插画风格、手绘插画、像手画的、手绘感 | 手绘插画风格 | → §八，用「风格关键词」段的模板片段开头，**必须**加 `--ref` |
| PPT、演讲风、幻灯片、技术PPT、slide、机器人风格 | 技术PPT风格 | → §九，用「prompt 风格描述」段的模板片段开头，**必须**加 `--ref` |
| （无关键词 / 默认） | 极简专业 PPT 风格 | → 按 §三 各图类型模板，结尾加「风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。」 |

### 手绘风格的生成规则（两者缺一不可）

**① `--ref` 优先**：传入参考图锚定视觉风格（有机线条、水彩填色感），仅靠 prompt 描述无法稳定复现。推荐参考图：

```
--ref ~/.claude/skills/zh-tech-intro/assets/handdrawn-style/handdrawn-01-skill-creation-loop.png
```

（也可换 `handdrawn-02` 至 `handdrawn-05`，选和目标图类型最接近的）

**② prompt 风格描述不可省略**：`--ref` 锁视觉质感，prompt 里的风格词控制模型对内容的理解方向，两者共同作用。prompt 开头必须加：

```
手绘插画风格，有机线条感，水彩填色质感，背景纯白。
```

**完整调用示例：**

```bash
RAND=$(openssl rand -hex 4)
bash ~/.claude/skills/gpt-image-2/scripts/gen.sh \
  --prompt "手绘插画风格，有机线条感，水彩填色质感，背景纯白。[图内容描述...]" \
  --ref ~/.claude/skills/zh-tech-intro/assets/handdrawn-style/handdrawn-01-skill-creation-loop.png \
  --out /tmp/zh-tech-img-${RAND}.png \
  --timeout-sec 300
```

参考图示例见 `../../assets/handdrawn-style/`，详细模板见 §八。

## 一、核心原则

### ⚡ 第一原则：描述视觉物体，不指定显示文字

模型在「画什么」和「写什么」之间有隐含的选择权——prompt 的写法决定它把预算花在哪里。

| 低效（指定文字） | 高效（触发元素） |
|---|---|
| 「三张卡片，分别写：快速响应、精准定位、低延迟」 | 「三张卡片，每张左侧带彩色图标，主题依次是闪电、靶心、时钟」 |
| 「右侧补充说明：该步骤负责过滤低质量输入，阈值 0.6」 | 「右侧灰色小注卡片，带盾牌图标，一句话质量门描述」 |
| 「底部备注区，文字说明 A、B、C 三种情况」 | 「底部图例区，三个色块 + 三个短标签，横向排开」 |

描述**视觉物体**（图标、色块、卡片、箭头）→ 模型画元素；描述**文字内容**（说明、注释、备注）→ 模型显示文字。

---

### 视觉效果描述优先于文字内容

| 多写 | 少写 |
|------|------|
| 整体氛围（「明亮通透」「色彩饱和但不杂乱」） | 节点副标签的详细文字 |
| 色彩对比（「图标是全图最鲜艳的元素，和浅色底色形成反差」） | 每张卡片的具体说明文字 |
| 视觉层次（「卡片是主角，箭头细而克制」「标题条比底色深一档」） | 过度精确的文字标注 |
| 空间感（「留白充裕」「两侧色块等高对齐」） | 渲染参数（颜色值、尺寸） |

---

### 绝对禁止

| 禁止 | 原因 |
| --- | --- |
| CSS 颜色值（`#3B82F6`） | 过度限制模型发挥，结果反而变差 |
| 像素/尺寸（`800px`、`14pt`） | 生图模型不接受绝对尺寸 |
| 字号/字重（`bold`、`font-size: 16`） | 渲染细节由模型决定 |
| SVG/HTML/CSS 代码片段 | 不是代码生成任务 |
| 箭头粗细、边框样式等渲染细节 | 描述内容和结构，不描述样式参数 |

### 应该写的内容

- **图类型**：横向多层架构图、竖向三阶段流程图、条形图+卡片……
- **节点标签**：全部用中文，描述语义（「同步决策」而非 `SyncDecision`）
- **箭头语义**：说明关系（「降级路由」「触发迁移」），不说方向
- **关键数据**：阈值、比例、数字直接写进来（「得分 0.62」「权重 30%」）
- **视觉层次**：「重点突出」「虚线框表示可选」「深色背景强调」

---

## 二、【禁用】泳道图

**不要使用纵向泳道图（Vertical Swimlane）。**

gpt-image-1 实践验证：纵向泳道图 prompt 生成结果极差——泳道线混乱、节点溢出边界、文字截断、整体像线框图而非正式图表。

**替代方案**：

| 原泳道图场景 | 替换为 |
| --- | --- |
| 跨模块的完整流程（跨角色） | 竖向三阶段流程图（§3.2） |
| 数据在多层/多组件间流转 | 横向多层架构图（§3.1）或横向阶段列流程图（§3.7） |
| 强调「谁做什么」的分工 | 横向阶段列流程图（§3.7），顶部加阶段标签 |

## 三、图类型参考模板

### 3.1 横向多层架构图

适用：展示系统分层结构、各层组件和整体数据流。  
参考图：
- `../../assets/01-architecture.png`（MemoryCore 四层架构）
- `../../assets/14-2-layered-architecture.jpg`（LLM-Wiki 四层架构，**重点参考**：左侧彩色层名徽章、节点图标+副标题、层间单箭头标语义）

**三区布局规范**（每行一层，左中右三区）：

| 区域 | 内容 | 视觉要求 |
| --- | --- | --- |
| 左区（层名徽章） | 层名文字 | 带颜色背景的圆角矩形，颜色比该行浅色背景深一档 |
| 中区（节点组） | 横向并排节点，每节点：小图标 + 加粗名称 + 灰色副标（1~2行） | 圆角矩形，浅色背景行 |
| 右区（角色描述） | 一句话说这层"是什么角色" | 灰色小字，垂直居中 |

**属性卡片 vs 流程节点**（最容易混淆，必须看清楚再写）：

| 情形 | 节点含义 | 写法 | 箭头 |
| --- | --- | --- | --- |
| 展示该层的**特征/属性** | 并排的属性描述（如：原始消息 / 顺序读取 / 会话级） | 「横向并排三张属性卡片，**卡片之间不用箭头连接**」 | ❌ 无 |
| 展示该层内部的**处理步骤** | 有先后顺序的操作（如：解析→提炼→落库） | 「三个步骤节点，箭头相连」 | ✅ 有 |

错误写法「内容三个节点串联」容易让生图工具在属性列表之间画出箭头，整层变成流程图而非存储架构图。

**层外数据流规范**：

架构图中层与层之间的数据流箭头（如「实时写入」「异步提炼」「混合检索」）必须明确声明在色带**外侧**，不能穿插在层内卡片之间。格式：

```
# 色带外部的数据流（均在色带轮廓外侧，不穿插在节点之间）：
#   顶部：「[来源]」→ 上方色带，标注「[操作]」
#   两层之间：居中垂直箭头，标注「[操作]」
#   右侧：[侧边线说明]
```

**层间箭头规范**：
- 每两层之间只画**一条**居中向下箭头，**不要扇形展开**连到每个节点
- 箭头标注具体操作语义（「调用」「读取 & 写入」「策略查询」），不写"数据流"这类泛称
- 箭头标签下方加白色矩形衬底，防止与背景混叠

```
# 画图提示：横向多层架构总览图，展示[系统名]的[N]层结构与数据流。
#
# 整体布局：[N] 行（每行一层）从上到下排列，每行分三区：
#   左区：层名徽章，带颜色背景的圆角矩形，颜色比该行背景深一档，层名居中
#   中区：节点横向并排，每节点由小图标 + 加粗名称 + 灰色副标组成
#   右区：一句角色定位描述，灰色小字
#
# 行 1【[层名]】（[浅色]背景行，左徽章[深色]）：
#   [N] 个节点，各带图标：
#     「[节点A]」  [副标题]
#     「[节点B]」  [副标题]
#   右区：[一句角色描述]
#
# 行 2【[层名]】（[浅色]背景行，左徽章[深色]）：
#   [N] 个节点，副标题为串联步骤：
#     「[节点C]」  [步骤1] → [步骤2] → [步骤3]
#   右区：[一句角色描述]
#
# 层间箭头（每层之间一条，垂直向下居中，不展开连到各节点，标签有白色衬底）：
#   [层名A] → [层名B]：「[操作语义]」
#   [层名B] → [层名C]：「[操作语义]」
#
# 风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。
```

### 3.2 竖向三阶段串行流程图

适用：有清晰阶段划分的处理流程，替代泳道图。  
参考图：
- `../../assets/02-dreaming-flow.png`（梦境三阶段）
- `../../assets/14-3-ingest-flow.jpg`（Ingest 摄入流程，参考：左侧主流程 + 右侧信息卡、步骤编号徽章、决策菱形）
- `../../assets/14-4-query-flow.jpg`（Query 查询流程，参考：多步骤带子分支、右侧补充说明卡）

**标题栏位置：顶部横条（不是左侧竖条）**

每个阶段框的标题栏必须是**顶部横贯全宽的横条**，不要用左侧竖条。原因：左侧竖条挤占水平空间，步骤节点区域变窄，图整体显得拥挤。

```
# ✅ 正确：顶部横条
#   顶部深蓝横条标题栏（横贯矩形全宽），白色文字「[名称] · [副标题]」，左侧有图标；
#   标题条下方主体区域：…

# ❌ 错误：左侧竖条
#   左侧深蓝色竖条标题栏，白色文字「[名称] · [副标题]」；
```

**异步阶段标准表示**：

当某个阶段是**异步执行**（不阻塞主流程），需同时给出三个视觉提示：
1. 阶段框外轮廓改为**虚线**（橙色虚线）
2. 背景色选**浅黄**（与同步阶段的蓝/绿色系形成区分）
3. 标题栏**右端**加「**异步**」白色小胶囊标签

```
#   阶段二【[名称]】（浅黄背景，整行宽圆角矩形，整体外框为橙色虚线）：
#     顶部深橙横条标题栏（横贯矩形全宽）「[名称] · [副标题]」，标题栏右端有「异步」白色小胶囊标签；
#     标题条下方主体区域：…
#     主体最右侧有一行灰色小字「不阻塞主流程」
```

**步骤节点布局**：同一阶段内的多个步骤节点必须等宽、均匀分布，不要挤在一侧留大片空白。

```
# 画图提示：竖向三阶段流程图，展示[流程名]
#
# 顶部触发器：「[触发条件]」→ 判断菱形「[已完成检查]？」→ 是则结束，否则继续
#
# 三个串联阶段框（圆角矩形，从上到下）：
#   阶段一【[名称]】（浅蓝背景，整行宽）：
#     顶部深蓝横条标题栏（横贯全宽），左侧图标，白色文字「[名称] · [副标题]」
#     标题条下方：横向排列 N 个白色圆角步骤节点，等宽均匀分布，箭头相连：
#       「[步骤1]」→「[步骤2]」→「[步骤3]」
#   阶段二【[名称]】（浅黄背景，整行宽，整体外框橙色虚线）：
#     顶部深橙横条标题栏（横贯全宽）「[名称] · [副标题]」，右端「异步」白色小胶囊标签
#     标题条下方：横向排列 N 个步骤节点，等宽均匀分布，箭头相连
#     主体最右侧灰色小字：「不阻塞主流程」
#   阶段三【[名称]】（浅绿背景，整行宽）：
#     顶部深绿横条标题栏（横贯全宽），左侧图标，白色文字「[名称] · [副标题]」
#     标题条下方：左侧菱形「[判断节点]」，右侧引出 N 条等间距纵向分布的水平箭头，
#       各指向 N 张等高圆角卡片（纵向均匀排列）
# 阶段间：每两个阶段之间一条居中实线向下箭头
#
# 风格：白色背景，高级技术 PPT 配图，极简专业，文字标注全部用中文。
```

### 3.6 竖向串行流程图（含判断分支）

适用：单一路径的处理步骤，含条件判断分支。

**⚠️ 多旁路汇合时的四条强制规则**（经实践验证，缺失任一条都会出现连线错误）：

1. **旁路终点必须显式连入汇合节点**：每条旁路的最后一个节点，必须在 prompt 中明确写「→ 向下连入「[汇合节点名]」」，不能依赖模型自行推断。
2. **所有箭头居中进出**：在 prompt 中加一句「所有箭头从节点中央水平或垂直引出，进入目标节点中央」。
3. **多旁路纵向空间均匀分配**：有 N 条旁路时，在 prompt 中说明「N 条路径在图中纵向空间均匀分配，不要相互紧贴」。
4. **颜色一致性**：在 prompt 中加「所有节点配色保持同一色调体系，由模型自行决定，不要有的节点深色调有的浅色调」。

**⚠️ 禁止自动编号**：流程节点不加流水序号（①②③），除非明确要求。

```
# 画图提示：竖向流程图，展示[流程名]的处理步骤与分支逻辑。
#
# 从上到下：
#   「[入口节点]」（椭圆）
#     ↓
#   「[判断条件A]」（菱形）
#     是 →→ 右侧旁路：「[旁路A节点1]」→「[旁路A节点2]」→ 向下连入「[汇合节点]」
#     否 ↓
#   「[判断条件B]」（菱形）
#     是 →→ 右侧旁路：「[旁路B节点1]」→「[旁路B节点2]」→ 向下连入「[汇合节点]」
#     否 ↓
#   「[主路径节点组]」（主路径向下）→ 向下连入「[汇合节点]」
#     ↓
#   「[汇合节点]」（三条路径在此汇合）
#
# 布局要求：
```

### 3.7 横向阶段列流程图

适用：多个处理阶段，每个阶段内有纵向子流程，阶段间横向推进。  
参考图：`../../assets/05-write-chain.png`（OpenViking 写入完整链路）

**⚠️ 列间连接是最容易混乱的地方**，必须在 prompt 中逐一声明列间连线，不能只描述列内节点。  
核心规则：
- **每列的出口节点必须明确点名**连接到下一列的哪个入口节点。
- **列内分支终止时**（如某条件判断为"跳过"的终止路径），必须标注「此路径止于本列，不跨列」。

### 3.10 金字塔分层图

适用：展示层次递进关系，底层最基础/最宽，顶层最高级/最窄（如优先级金字塔、处理链路分层）。

**⚠️ 金字塔的四条强制规则**（经实践验证，违反任一条都会导致图形扭曲或层次混乱）：

1. **层序方向**：底层 = 第一层（最基础）= 最宽；顶层 = 最高层（最高级）= 最窄的塔尖。**编号从下到上递增**。模型默认容易画反，必须在 prompt 中明确标注「底部最宽为第一层，顶部尖顶为最高层」。
2. **角度必须一致**：三条规则—左侧斜边从底层左角到顶点是一条连续直线；右侧斜边从底层右角到顶点是一条连续直线；所有层的左右两侧均共用同一条斜边，不单独画。违反此规则会导致各层角度不一致，视觉极丑。
3. **必须有塔尖**：顶层要收成三角形的真实尖顶（apex），不是梯形。
4. **不加标题**：金字塔本身不加图表标题，除非用户明确要求。

```
# 画图提示：金字塔分层图，展示[N]层[主题]结构。
#
# 整体形状：等腰三角形，顶部收成尖顶（apex），无图表标题。
# 三角形两侧斜边从底层两角延伸至顶点，是贯通全图的连续直线，各层共用同一斜边。
#
# 从下到上共 [N] 层（底部最宽为第一层，顶部尖顶处为第[N]层）：
#
#   第一层（底部，最宽）：
#     层名标题（加粗）：「[名称]」
#     属性行（带小图标）：
#       📋 处理：[处理描述]
#       ⚙️ 操作：[操作描述]
#       ⏱️ 时间成本：低（[说明]）
#
#   第二层（中部）：
#     层名标题（加粗）：「[名称]」
#     属性行（带小图标）：
#       📋 处理：[处理描述]
#       ⚙️ 操作：[操作描述]
#       ⏱️ 时间成本：中（[说明]）
#
#   第三层（顶部，最窄，含塔尖）：
#     层名标题（加粗）：「[名称]」
#     属性行（带小图标）：
#       📋 处理：[处理描述]

#       ⏱️ 时间成本：高（[说明]）
#
# 各层配色：三层分别用三种协调色（如蓝/橙/红渐变色系），颜色由模型自行决定。
```

### 3.13 两层流程图：主流程 + 展开细节

适用：一个步骤本身有几个子步骤，单独画一张图太单薄，但全塞进主流程一行又太挤——典型场景是"5步主流程，其中第4步还能再拆成4个子步骤"。比单独画两张稀疏的图density更高，比硬塞成一行8+卡片更不挤。

**核心规范**：
- 上层是主流程，横向卡片，其中一张"被展开"的卡片用更深的颜色/边框突出，小字写「展开见下方」
- 该卡片正下方引一条竖直箭头连到下层
- 下层是子步骤展开，用浅灰虚线边框的矩形框整体框住，左上角标小字说明这是哪一步的细节，框内卡片可以比上层卡片略小、颜色用同色系的浅一档变体
- 整张图依然要显式声明「横向宽屏构图，宽高比约16:9」（见踩坑14），否则上下两层会被压成竖版导致两层都拥挤

```

# 上层主流程：[N]个带图标的[颜色]圆角卡片从左到右水平排列，卡片间距宽松，用实线箭头连接。
```

## 七、常见踩坑教训

以下均为通用问题，图类型专属的规则已整合进 §三 各模板的 ⚠️ 段落。

### 踩坑 1：流程节点被自动加上流水编号

**现象**：节点前出现「1.」「2.」「3.」序号，视觉噪声大。  
**根因**：prompt 用有序列表格式描述节点，模型把序号当作标签的一部分渲染出来。  
**修复**：节点描述不加编号。用「→」表示顺序，不用「1. 2. 3.」。

### 踩坑 2：单个节点内容过多导致一侧留白严重

**现象**：某节点宽度撑大，左侧大面积留白，整体比例严重失衡。  
**根因**：把节点名 + 子分类枚举混在同一个节点里，宽度失控后左侧空旷。  
**修复**：子分类内容提炼为行内简短格式（「专有名词 / 复合名词 / 引用文本」一行平铺），不要和节点名混排。

### 踩坑 3：循环终止条件混入输出类型菱形

**现象**：把「超过最大轮次」作为第三条分支塞进「本轮输出属于哪一类？」的决策菱形，逻辑错误——LLM 只能输出「工具调用」或「最终结果」两种，「超过最大轮次」是循环计数器的状态，不是 LLM 的输出类型。  
**根因**：菱形节点的语义是「对同一事件的不同分支」，循环守门条件是另一件事。  
**修复**：循环终止条件单独提前为第一个菱形，在调用 LLM 之前先判断「是否超过最大轮次」；后面的分支菱形只负责判断输出类型，各自只有两条出路。

```
❌ 反例（菱形三出口）：       ✅ 正例（两个串联菱形，各司其职）：

进入循环                      进入本轮迭代
  ↓                             ↓
LLM 调用                      「是否超过最大轮次？」
  ↓                             ├── 是 → 返回空列表
「本轮输出哪一类？」              └── 否 → 调用 LLM
  ├── 工具调用 → 执行 → 循环           ↓
  ├── 最终结果 → 返回              「本轮输出哪一类？」
  └── 超过轮次 → 返回空 ❌          ├── 工具调用 → 执行 → 循环回入口
                                  └── 最终结果 → 返回
```

### 踩坑 4：只画极简骨架节点，图显得单薄单调

**现象**：节点只写一个动作短语，不带任何支撑数据，图沦为"过场插画"而非"信息载体"。  
**根因**：把画图提示当成章节标题大纲，只搬了步骤名，没有把关键数字、对比、设计取舍放进节点。  
**修复**：画图前检查对应正文段落，把以下三类内容塞进节点副标题/小字里：
- **关键数字**：阈值、权重、上限（「截断到 limit 条（默认 10）」）
- **对比/取舍**：两个版本的差异点（「初始分 0.58 vs 0.62」）
- **一句话设计动机**：为什么这步存在

**反向陷阱**：加密度不是把某些节点做成视觉主角、把其余降级成浮动注释——这样"信息有了，但关系没画出来"。优先保证每个节点视觉权重相同、箭头主线清晰，再考虑在副标题里补数字。

### 踩坑 5：横向多卡片图默认生成竖版画布，导致挤压拥挤（见 §3.13）

**现象**：描述"5个卡片横向排列"，但模型按竖版出图，卡片间距被挤没。  
**根因**：prompt 只描述排列方式，没有声明画布宽高比，模型用默认竖版。  
**修复**：横向多卡片/多列布局，prompt 开头必须加：「横向宽屏构图，宽高比约 16:9，禁止竖版」，同时写「卡片之间留白宽松不拥挤」。

### 踩坑 6：未声明背景色，模型给整图刷了大片强调色

**现象**：给某步骤起了"浅紫色卡片"，模型把强调色铺成整张图背景色。  
**根因**：描述了"用什么颜色"，但没声明"只能用在哪里"，模型选了影响范围最大的刷背景。  
**修复**：使用强调色时同时写两句——「整体背景必须是纯白色」＋「[颜色]只用在图标线条和强调色上」，两句缺一不可。

### 踩坑 7：色块颜色过深，整体压抑

**现象**：大面积深色色块，整张图沉重、阅读费力，和白色主背景割裂。  
**根因**：prompt 写了「深色背景」，模型忠实执行。  
**修复**：背景色块用浅色系，标题条比色块底色深一档即可。关键词：「明亮通透」「浅[颜色]背景，不要深色」。

## 八、手绘插画风格参考库

这套风格适合技术文章配图、社交媒体传播图、流程说明图。核心特征：

| 特征 | 描述 |
| --- | --- |
| 线条感 | 圆角矩形有轻微手绘线条感，非完全机械规整 |
| 字体 | 略带手写质感，中文圆润、标题加粗 |

| 步骤编号 | 填色圆圈（①②③…），左上角或正上方，颜色比卡片底色深一档 |

## 九、技术PPT风格参考库（techppt-style）

> 参考图存放于 `../../assets/techppt-style/`

| 特征 | 描述 |
| --- | --- |
| 标题 | 粗体大字，无横幅边框，直接写在顶部居中 |

| 吉祥物 | 卡通机器人，固定出现在角落起点缀作用 |
| 内容密度 | 每个框内可放多行结构化内容（指标·工具·失败模式） |
